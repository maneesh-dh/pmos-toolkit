Skill development via /feature-sdlc — Requirements

Date: 2026-05-11 · Last updated: 2026-05-11 · Status: Draft · Tier: 3 — Feature

Companion: 00_seed_brief.md (the brainstorming output that locked the 14 decisions, the target phase map, and the file-move plan). This requirements doc frames the problem, users, journeys, success criteria, and the FR/NFR set the /spec stage needs. The locked decisions in the seed brief are settled inputs, not open questions.

Problem

The pmos-toolkit has three skills that all drive essentially the same SDLC pipeline (requirements → spec → plan → execute → verify → complete-dev): /feature-sdlc (feature development), /create-skill (authoring a new skill), and /update-skills (applying feedback to existing skills). Only /feature-sdlc has git-worktree isolation and a resumable state.yaml; /create-skill and /update-skills re-implement the pipeline without those, so skill-authoring work cannot be paused/resumed and is not isolated from the working tree. The three implementations have drifted and must be kept in sync by hand.

Separately, skill quality is currently governed only by a prose "Conventions" checklist inside /create-skill with no scoring and no self-refinement loop — and that checklist is pmos-toolkit-specific, so it does not help users author skills for their own repos or plugins under the broader SKILLS standard.

Who experiences this?

  • Skill author (new skill) — a developer/PM who wants a new /slash-command skill, in pmos-toolkit or in their own repo/plugin. Today they run /create-skill; they get no worktree, no resume, and a quality bar that is just a checklist.
  • Skill author (applying feedback) — someone who has feedback about an existing skill (free-form notes, or a /retro paste-back) and wants it triaged and implemented end-to-end. Today they run /update-skills; same gaps, plus a triage-doc-based resume that diverges from /feature-sdlc's.
  • Feature developer — runs /feature-sdlc as today. Their experience must not regress; they should also benefit from the phase-numbering cleanup.
  • Toolkit maintainer — currently keeps three pipeline implementations in sync. After this change, one.

Why now?

/feature-sdlc recently gained harness-native worktree creation, drift-checked resume, and a v3 state schema (v2.35.0). That makes it a credible single home for every pipeline-driven workflow. The skill-authoring ecosystem has also matured — Anthropic, OpenAI Codex, and the agentskills.io open standard now publish concrete authoring guidance — so a research-grounded patterns/eval pair is feasible and worth building once.

Goals & Non-Goals

Goals

  • A skill author can take a new skill from idea to merged-and-released through one orchestrator (/feature-sdlc skill <desc>), with git-worktree isolation and --resume — measured by: the run produces a worktree + state.yaml and survives a session restart at any phase boundary.
  • A skill author can take feedback about an existing skill to a merged change through the same orchestrator (/feature-sdlc skill --from-feedback <input>) — measured by: feedback (raw text or /retro paste-back) is triaged with explicit keep/drop approval, then implemented via the standard pipeline.
  • Every skill produced or modified through skill-dev mode is scored against a binary eval rubric and refined until it passes (or the user knowingly accepts residual failures) — measured by: a state.yaml record of the eval loop with pass/fail per check and the residual set.
  • The patterns guide and eval rubric are portable across the SKILLS standard (Claude Code primary; Codex and other compliant agents in scope) — measured by: no check or instruction is Claude-Code-only where a portable phrasing exists; per-platform specifics are labelled as such.
  • /feature-sdlc's phases are linearly numbered with optional steps as alphabetic sub-phases — measured by: the phase list reads 0, 0a, 0b, 0c, 0d, 1, 2, 2a, 3, 3a, 3b, 3c, 4, 5, 6, 6a, 7, 8, 8a, 9, 10 with no gaps and no out-of-order entries.
  • The feature developer's experience is unchanged except for the renumbering and the new (suppressed-in-feature-mode-by-default) plumbing — measured by: a bare /feature-sdlc <idea> run executes the same logical steps as 2.36.0.
  • /create-skill and /update-skills are removed from the loadable plugin and preserved for reference — measured by: both directories live under archive/skills/ with an archive/README.md; the plugin loader no longer exposes them.

Non-Goals (explicit scope cuts)

  • NOT building a multi-skill batch mode for --from-feedback — because the single-skill flow keeps the state.yaml/resume model simple; a /retro touching N skills is run N times.
  • NOT keeping /create-skill's ~/.pmos/skill-specs/ spec location — because every other pipeline artifact lives in the feature folder and one location is simpler.
  • NOT adding a canonical-path-enforcement phase — because "where skills live / how releases work" is host-repo policy that /execute already reads from the repo's CLAUDE.md; baking pmos-toolkit specifics into a generic skill would break it for other repos.
  • NOT shipping the "focused / change-scoped eval subset" for --from-feedback in this iteration — because the full-rubric run is correct and simpler; the focused variant is a follow-up.
  • NOT renaming /feature-sdlc — because the cost of churning README/manifests/muscle-memory outweighs the naming nicety; a thin /skill-sdlc alias covers discoverability.
  • NOT changing how /wireframes or /prototype work — they are simply suppressed in skill-dev modes (skills have no UI).
  • NOT auto-applying feedback findings without explicit user approval — the triage keep/drop gate is mandatory.

User Experience Analysis

Motivation

  • Job to be done (new skill): "I keep typing the same prompt / doing the same workflow — turn it into a reliable, well-formed skill and ship it." Importance: high for power users; a badly-formed skill silently fails to trigger or bloats context. Alternatives: hand-write SKILL.md and hope; use /create-skill (no isolation/resume, checklist-only quality).
  • Job to be done (feedback): "I ran /retro (or jotted notes) and want the real problems fixed in the skill, end-to-end, without losing the audit trail." Importance: high — this is how skills improve. Alternatives: /update-skills (diverging resume); manual edits (no triage, no verify).

Satisfaction Signals

  • The author sees a single, obvious command and a familiar pipeline they already trust from feature work.
  • A killed session resumes exactly where it stopped, in an isolated worktree, with no lost work.
  • The eval loop reports concrete, checkable failures ("description embeds a step list", "body is 540 lines > 500") and the final skill passes them — the author trusts it will trigger and stay maintainable.
  • A skill authored for a non-pmos repo works the same way and the patterns guidance still applies.

Solution Direction

Fold both skill-authoring skills into /feature-sdlc as a skill subcommand, archive the originals, add a research-grounded patterns guide + binary eval rubric + a TDD-style refinement sub-phase, ship a thin /skill-sdlc alias, and linearly renumber /feature-sdlc. The pipeline spine (requirements → spec → plan → execute → verify → complete-dev) and the worktree/resume machinery are reused unchanged; a mode switch turns mode-specific phases on/off.

Modes (explicit subcommand, per locked decision 2):

  • /feature-sdlc <idea> → feature mode — today's behavior.
  • /feature-sdlc skill <description> → skill-new mode.
  • /feature-sdlc skill --from-feedback <text | path | --from-retro> → skill-feedback mode (single skill per invocation).
  • /feature-sdlc list → unchanged.

User Journeys

Primary Journey — author a new skill

  1. Author runs /feature-sdlc skill "a skill that summarizes a PR diff into release notes" from their repo.
  2. Orchestrator derives a slug, creates a worktree + feat/<slug> branch, writes state.yaml with mode: skill-new, enters the worktree.
  3. Phase 0d resolves a tier using the create-skill matrix (phase count / reference/ files / eval rubric / pipeline integration), surfaces it for confirmation, passes --tier N down.
  4. Phase 2 /requirements runs with the description + skill-patterns.md as seed; folded /msf-req runs (skills are user-facing).
  5. Phase 2a /grill (Tier 2+) adversarially reviews the requirements.
  6. Phase 3a creativity gate offered (Recommended = Skip); 3b/3c wireframes/prototype are not presented (suppressed).
  7. Phase 4 /spec → Phase 5 /plan → Phase 6 /execute writes the SKILL.md (+ any reference/ files), with skill-patterns.md as the implementation reference, honoring the host repo's CLAUDE.md for save location and release policy.
  8. Phase 6a runs skill-eval-check.sh (deterministic) + a reviewer subagent (LLM-judge) against the written files; the reviewer returns pass/fail per check and, for each fail, a concrete fix note — it makes no edits. Failures are bundled and handed back to /execute; re-score; cap 2 loops; then a structured ask: Accept-as-risk / iterate manually / abort. The loop count + residual checks are recorded in state.yaml.
  9. Phase 7 /verify (non-skippable) re-runs skill-eval.md as a final idempotent gate and grades the host-repo release prereqs.
  10. Phase 8 /complete-dev merges, bumps version, regenerates changelog, deploys per repo norms, pushes.
  11. Phase 8a /retro gate (Recommended = Skip); Phase 9 summary; Phase 10 capture learnings.

Primary Journey — apply feedback to an existing skill

  1. Author runs /feature-sdlc skill --from-feedback --from-retro (or pastes notes / a file path).
  2. Orchestrator creates the worktree/branch, writes state.yaml with mode: skill-feedback, enters the worktree.
  3. Phase 0c parses the input (using retro-parser.md if it's a /retro paste-back), critiques each finding against the current SKILL.md ("already handled?", classify, recommend), and surfaces a keep/drop approval; the result is the approved-change list (an artifact in the feature folder + a state.yaml entry).
  4. Phase 0d resolves a tier from the size of the approved change set; passes --tier N down.
  5. Phase 2 /requirements runs with a seed built from the approved findings + current-SKILL.md excerpts + skill-patterns.md (the moved seed-requirements-template.md).
  6. Phases 2a → 4 → 5 → 6 → 6a → 7 → 8 → 8a → 9 → 10 proceed as in the new-skill journey, targeting the skill being modified.

Primary Journey — feature development (unchanged)

A bare /feature-sdlc <idea> run executes the same logical steps as 2.36.0; the only visible change is the linear phase numbering and the renumbered state.yaml phase ids (with a back-compat migration on resume).

Alternate Journeys

  • /skill-sdlc alias: /skill-sdlc <desc> or /skill-sdlc --from-feedback … forwards verbatim to /feature-sdlc skill …; identical behavior thereafter.
  • Resume: the author re-launches Claude in the worktree and runs /feature-sdlc --resume; the drift check + schema check pass, the status table is shown, and the cursor jumps to the first non-complete phase — regardless of mode.
  • --no-worktree: skill-dev modes honor it the same way feature mode does (state lives in the launch cwd, worktree_path: null, drift check bypassed).
  • Tier 1 skill: the create-skill matrix can still resolve Tier 1 (one-shot utility); /grill is then skipped (Tier 2+ only), and the pipeline is shorter — but /verify and the Phase 6a eval gate still run.

Error Journeys

  • Eval loop exhausts 2 iterations with residual failures: orchestrator presents Accept-as-risk (residuals logged + handed to /verify as known findings) / keep iterating manually / abort. No silent pass.
  • Feedback input has zero actionable findings: Phase 0c reports "no actionable findings in <input>" and exits cleanly without creating downstream artifacts.
  • Resuming a pre-v4 state.yaml: auto-migrate (add mode defaulting to feature, renumber phase ids, elide removed ids) with a chat log line; never block, never prompt — same contract as the existing v1→v2→v3 migrations.
  • A skill-dev run targets a repo whose CLAUDE.md says nothing about where skills go: /execute falls back to the SKILLS-standard default location (.claude/skills/<name>/ or the platform equivalent) and notes the assumption; no hard failure.
  • Old command invoked: a user typing /create-skill or /update-skills after archival gets "unknown skill"; the README + archive/README.md point them at /feature-sdlc skill (acceptable — no shim required).

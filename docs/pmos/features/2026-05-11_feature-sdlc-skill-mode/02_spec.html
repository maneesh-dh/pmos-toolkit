1. Problem Statement

The pmos-toolkit ships three skills that each re-implement the same SDLC spine (requirements → spec → plan → execute → verify → complete-dev): /feature-sdlc, /create-skill, /update-skills. Only /feature-sdlc has git-worktree isolation and a resumable state.yaml; the two skill-authoring skills lack both, have drifted from each other, and must be hand-synced. Skill quality is governed only by a prose "Conventions" checklist inside /create-skill — no scoring, no refinement loop, and pmos-toolkit-specific so it doesn't help authors targeting their own repos under the open SKILLS standard. Primary success metric: skill-authoring entry points drop from 3 → 2 (/feature-sdlc + thin /skill-sdlc alias) and pipeline implementations to keep in sync drop from 3 → 1, with every skill-dev run worktree-isolated, resumable, and scored against a binary eval rubric before merge.

3. Non-Goals

  • NOT keeping /create-skill's ~/.pmos/skill-specs/ location — every artifact lives in the feature folder.
  • NOT allowing multiple new skills in one run — skill-new is one skill per invocation; only skill-feedback is multi-skill (G11).
  • NOT adding a canonical-path-enforcement phase — that's host-repo policy (/execute reads the repo's CLAUDE.md); baking pmos specifics into a generic skill breaks it elsewhere.
  • NOT shipping the focused/change-scoped eval subset for --from-feedback this iteration — full-rubric run is correct and simpler; focused variant is a follow-up.
  • NOT renaming /feature-sdlc — a thin /skill-sdlc alias covers discoverability.
  • NOT changing how /wireframes//prototype work — simply suppressed in skill-dev modes (no UI).
  • NOT auto-applying feedback findings without explicit approval — the triage keep/drop gate is mandatory.
  • NOT hardening /complete-dev for arbitrary repos — Phase 8 in skill-dev delegates wholly to it and accepts its graceful degradation (G12).
  • NOT a runtime feature flag — this is a structural change; /feature-sdlc skill simply does not exist until the release lands.

5. User Personas & Journeys

Personas and journeys are fully developed in 01_requirements.html §User Experience Analysis and §User Journeys. Recap of the three primary actors and the control-flow each follows:

5.1 Skill author — new skill (primary)

/feature-sdlc skill "<description>" → slug + worktree + feat/<slug> + state.yaml(pipeline_mode: skill-new) → Phase 0d (one repo-shape pass → tier / skill-location / target_platform, confirmed once) → Phase 2 /requirements (+ folded /msf-req; skill-patterns.md cited as acceptance criteria) → 2a /grill (Tier 2+) → 3a creativity gate (Recommended Skip); 3b/3c not presented → 4 /spec → 5 /plan → [compact checkpoint] → 6 /execute (writes SKILL.md + any reference/ files; skill-patterns.md as implementation reference; honours host CLAUDE.md) → 6a skill-eval TDD loop → [compact checkpoint] → 7 /verify (non-skippable; re-runs skill-eval.md) → 8 /complete-dev → 8a /retro gate (Recommended Skip) → 9 summary → 10 capture learnings.

5.2 Skill author — feedback to one or more existing skills (primary)

/feature-sdlc skill --from-feedback <text | path | --from-retro> → worktree + state.yaml(pipeline_mode: skill-feedback) → Phase 0c (parse input [retro-parser.md if a /retro paste-back], extract + confirm the target-skill set, per-finding critique against each skill's current SKILL.md, keep/drop approval, write 0c_feedback_triage.html) → 0d (per-skill tier from each skill's approved-change-set size; run tier = max, shown with the breakdown; target_platform per skill) → 2 /requirements (seed = approved findings + current-SKILL.md excerpts for every targeted skill + skill-patterns.md; via the moved seed-requirements-template.md) → produces one combined requirements doc with a per-skill section → 2a → 3a → 4 → 5 → 6 (/execute edits each targeted SKILL.md) → 6a (scores each skill's files; failures across any skill feed back via the plan addendum) → 7 → 8 (/complete-dev → one version bump + one changelog entry covering all the skills) → 8a → 9 → 10.

5.3 Feature developer — unchanged

A bare /feature-sdlc <idea> executes the same logical steps as 2.36.0; the only visible change is the linear phase numbering. state.yaml phase ids stay name-based (G5); on resume of a pre-v4 file, the v3→v4 migration adds pipeline_mode: feature and bumps the version (FR-13).

5.4 Alternate & error journeys

  • /skill-sdlc alias: forwards verbatim to /feature-sdlc skill …; identical thereafter.
  • Resume (any mode): re-launch Claude in the worktree, /feature-sdlc --resume; drift + schema checks pass, status table shown, cursor jumps to the first non-complete phase.
  • --no-worktree: skill modes honour it (state in launch cwd, worktree_path: null, drift bypassed).
  • Tier-1 skill (G8): short-form /requirements → skip /grill + /creativity → short-form /spec → /plan → /execute → 6a (still runs) → /verify (non-skippable) → /complete-dev; folded /msf-req runs but brief.
  • Eval loop exhausts 2 iters with residuals: Accept-as-risk / iterate manually / (restore iteration 1 if iter 2 net-worse) / abort. No silent pass.
  • Feedback input has zero actionable findings: Phase 0c reports "no actionable findings in <input>" and exits cleanly — no downstream artifacts.
  • Resuming a pre-v4 state.yaml: auto-migrate (add pipeline_mode: feature, bump version) with a chat log line; never block, never prompt — same contract as v1→v2→v3.
  • Host repo's CLAUDE.md says nothing about skill location: /execute falls back to .claude/skills/<name>/ (or the platform equivalent) and notes the assumption; no hard failure.
  • Old command invoked: /create-skill / /update-skills → "unknown skill"; README + archive/README.md redirect to /feature-sdlc skill … (acceptable — no shim).

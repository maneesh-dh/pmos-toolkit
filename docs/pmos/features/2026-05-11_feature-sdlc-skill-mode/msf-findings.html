Executive Summary

#PriorityFindingEffort
R1MustTest the v3→v4 state.yaml resume migration against an actual in-flight feature-mode worktree fixture (mid-pipeline, not just a fresh one) — the renumber + mode-add is the single biggest regression risk for the Feature-developer persona.M
R2ShouldMake the Tier-1 skill path genuinely lightweight (short-form /requirements + /spec, no /grill, minimal gates). If authoring a 30-line utility skill still costs requirements→spec→plan→execute→verify→eval ceremony, authors will hand-write SKILL.md and bypass the orchestrator — defeating the consolidation. Partially OQ in the doc.M
R3ShouldGive skill-eval-check.sh / skill-eval.md an explicit target-platform input (Claude Code vs Codex vs generic SKILLS). Without it, Codex-target skills get false failures on Claude-Code-only frontmatter fields, and the "portable across SKILLS standard" goal isn't actually testable. The edge-cases table names the scenario but not the mechanism.S
R4ShouldRoute model-invocation: add skill-authoring trigger phrases ("create a skill", "new slash command", "write a skill", "update this skill") to /feature-sdlc's description so a user who'd have said "/create-skill" lands on /feature-sdlc skill without knowing the subcommand. The README pointer is a fallback, not a discovery path.S
R5ShouldKeep [llm-judge] checks few and ruthlessly well-specified, and require the reviewer's per-fail note to be a concrete edit instruction (not a vibe). The 2-iteration cap only works if one iteration can realistically clear a fail; subjective checks with vague feedback will routinely exhaust the loop and dump residuals on the author.M
R6ShouldResolve OQ5 now: commit 00_seed_brief.md (it's already in the feature-folder commit). The requirements doc treats it as the authoritative phase-map source, so a resumed session in a fresh checkout must have it; "scratch" is not a safe option once a downstream artifact depends on it.XS
R7NiceDocument the "run N times" workaround for multi-skill /retro output prominently in /skill-sdlc help text + the feedback-mode intro, so the Toolkit-maintainer persona isn't surprised by N worktrees / N /complete-devs / N version bumps. The single-skill cut is sound; the surprise is the cost.XS
R8NiceIn feedback mode, when Phase 0c critique says a finding is "already handled", show where (the SKILL.md line/section) in the keep/drop dialog — otherwise the author can't tell a real dedupe from a missed nuance and will tend to "keep" everything, inflating the change set and the resolved tier.S

Motivation

  • Skill author (new skill). Job: "turn a repeated prompt/workflow into a reliable skill and ship it." Strongly motivated when they already trust /feature-sdlc from feature work — the consolidation raises motivation (familiar pipeline, isolation, resume they didn't have). Risk to motivation: discovery — if they don't learn the skill subcommand exists, they either keep reaching for /create-skill (now "unknown skill") or hand-write. → R4.
  • Skill author (applying feedback). Job: "get the real problems from /retro fixed end-to-end with the audit trail intact." Motivation is high and the triage keep/drop gate + feature-folder artifact directly serve "audit trail". Mild demotivator: the mandatory tier confirmation + full pipeline for what may be a one-line wording fix to a SKILL.md. → R2 (shared with new-skill).
  • Feature developer. Not a new job — the requirement is "don't make my thing worse." Motivation is neutral-to-positive (cleaner phase numbers). The only thing that erodes it is a botched resume of an in-flight run after upgrade. → R1.
  • Toolkit maintainer. Job: "stop hand-syncing three pipeline implementations." Cleanly served (3→1). No motivation gap; the only friction is the multi-skill-feedback ergonomics. → R7.

Satisfaction

  • "One obvious command, a pipeline I already trust." Fulfilled iff R4 lands — otherwise the command is obvious only to people who read the changelog.
  • "A killed session resumes exactly where it stopped." Fulfilled for skill-dev modes by reusing the existing machinery (D12) — the satisfaction depends on R1 (the migration that makes existing in-flight runs survive the upgrade) more than on the new modes, which start clean on v4.
  • "Concrete, checkable failures — I trust the skill will trigger and stay maintainable." Fulfilled only if skill-eval.md's pass-conditions are unambiguous and the deterministic subset is large (Success Metrics already targets "every [deterministic] check covered by skill-eval-check.sh"). R5 is the risk: a rubric heavy on soft judgments produces "looks fine" by another name. The /spec stage owns nailing each pass-condition.
  • "A skill authored for a non-pmos repo works the same way." Fulfilled only with R3 — "portable" must be testable, not just asserted. Reassurance signal the author wants: the eval output saying which platform it scored against.
  • Toolkit maintainer: high satisfaction — "only /feature-sdlc contains pipeline-orchestration prose" is a concrete, gratifying end state. No shortfall.

Assumptions Surfaced (for user verification)

  • The skill-dev pipeline runs phases in the 00_seed_brief.md order (0 → 0a → 0b → 0c → 0d → 1 → 2 → 2a → 3a → 4 → 5 → 6 → 6a → 7 → 8 → 8a → 9 → 10); 3b/3c are not presented in skill-dev modes. R2/R8 assume this ordering.
  • Tier-1 in the create-skill matrix still means a meaningfully shorter pipeline (not just "skip /grill"). R2 is moot if the matrix already collapses upstream phases for Tier 1.
  • The reviewer subagent in Phase 6a receives the written SKILL.md (+ reference files) inline and returns structured per-check pass/fail + fix notes — i.e., it can produce concrete edit instructions, not just scores. R5 assumes this is the design intent (it is, per D10) and flags the quality bar on the rubric, not the mechanism.
  • "already handled?" in Phase 0c is a per-finding classification the orchestrator computes against the current SKILL.md — R8 assumes the location is available to cite (it must be, to make the determination).

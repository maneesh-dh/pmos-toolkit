Problem

Product managers and builders frequently need to run surveys — to understand user problems, validate hypotheses, or measure satisfaction — but most are not trained survey methodologists. The result is surveys riddled with leading questions, double-barreled items, unbalanced scales, missing opt-outs, and bloated length, which produce biased or unusable data. Today there is no assistant in the toolkit that takes a rough survey intent and turns it into a methodologically sound, ready-to-field instrument.

Who experiences this?

The primary persona is a product manager or founder/builder designing a survey for their own users — comfortable writing prose, familiar with their product domain, but not a research-methods expert. They typically reach for Google Forms, Typeform, or SurveyMonkey and start typing questions directly into the platform. A secondary persona is someone with an existing survey they want a second opinion on before sending it out.

Why now?

The pmos-toolkit already ships PM-facing skills for the requirements→spec→plan pipeline, plus design/critique utilities. Survey design is a high-frequency PM task with a well-established best-practice literature (Pew, Dillman, Krosnick, NN/g) that is straightforward to encode as a generation + critique loop — the same shape as the toolkit's other "generate, critique with a reviewer subagent, refine" skills. There is also now a stable HTML-authoring substrate (_shared/html-authoring/) the skill can reuse for the survey artifact and its viewer.

Goals & Non-Goals

Goals

  • A PM can go from a paragraph of intent (or an existing survey) to a fielding-ready, methodologically reviewed survey in one skill invocation — measured by the run producing a committed survey.html plus critique artifacts plus at least one platform-import file.
  • The generated survey respects the audience and time budget the user specified — measured by the simulated-respondent pass reporting an estimated completion time within the user's stated budget (or the skill flagging the overage and proposing cuts).
  • Common framing anti-patterns are caught and fixed before the user fields the survey — measured by the question-eval critique enumerating every question and the apply-step resolving each finding (fix / modify / skip-with-reason / defer).
  • The survey is adapted to its research mode (generative / evaluative / hybrid) — measured by the design's question-type mix matching the mode (open-ended-heavy for generative, closed/comparable for evaluative).
  • The user can view and walk the survey before fielding it — measured by an index.html viewer + local serve script that renders the survey as an interactive, fillable preview.
  • The survey can be moved into a real platform without re-typing it — measured by the skill emitting a valid import artifact for the chosen platform (Typeform / SurveyMonkey / Google Forms).

Non-Goals

  • NOT actually creating the survey in the third-party platform via live API calls — because that needs the user's API tokens and per-platform auth setup; the skill emits an import-ready file and instructions, and the user runs the import. (May be revisited later.)
  • NOT handling survey distribution, sampling panels, recruitment, or quota management — because that is a fielding/operations concern, not survey design; the skill stops at "ready to field."
  • NOT analyzing survey responses after fielding — because that is a separate downstream activity; this skill ends at survey creation.
  • NOT supporting conjoint / MaxDiff / advanced experimental designs in v1 — because they need specialist tooling and statistical setup beyond a question-list generator; the skill may suggest them but will not author them.
  • NOT a general form builder — because the scope is research surveys specifically; payment forms, registration forms, and quizzes are out of scope.

User Experience Analysis

Motivation

  • Job to be done: "I need to learn something from my users, and I want the survey to give me trustworthy answers — without me having to become a survey-methods expert first."
  • Importance / urgency: Moderate-to-high and recurring. A badly designed survey wastes the respondents' goodwill (you usually get one shot at a given audience) and produces decisions based on noise. PMs feel the cost acutely after the fact ("the data was useless").
  • Alternatives: Type questions straight into Google Forms/Typeform (no review); copy a template from a blog post (generic, not fitted to the question); ask a research colleague to review (slow, not always available); read a best-practices article and self-edit (high effort, easy to miss things). The skill competes on "expert review, instantly, fitted to your exact intent."

Satisfaction Signals

  • The user reads the critique and thinks "oh — yeah, that question was double-barreled" — the skill taught them something.
  • The estimated completion time comes back under their budget, or the skill clearly shows what to cut to get there.
  • The import file opens in their platform and produces a recognisable survey with minimal cleanup.
  • The user fields the survey with confidence rather than vague unease.

Solution Direction

A new pmos-toolkit skill, survey-design, living at plugins/pmos-toolkit/skills/survey-design/SKILL.md, that reuses the toolkit's shared substrate (.pmos/settings.yaml for docs_path, _shared/html-authoring/ for the artifact + viewer + serve script, the platform-adaptation and non-interactive conventions, the interactive-prompts helper). It ships a reference/ directory containing the baked-in research: a survey best-practices guide, a question-framing anti-pattern catalog (with BAD→FIXED examples and detection heuristics), and a per-platform export/import guide.

Runtime shape (the skill's phases, at a behavior level — exact phase boundaries are the spec's job):

  1. Intake. The user provides initial context — free text describing the survey they want, and/or a path to an existing survey to refine.
  2. Variable interpretation. The skill extracts what it can from the context: intended audience, time-to-complete budget, survey mode (generative = understand problems / evaluative = validate a hypothesis / hybrid), and an optional max-question cap. For each variable it can't confidently infer, it asks via a single batched AskUserQuestion call (one question per variable).
  3. Initial design. The skill generates a survey: a short intro, logical sections, and questions within each section — applying the best-practices reference (funnel order, balanced scales, opt-outs, mode-appropriate question-type mix, length budget). It writes survey.html (readable + interactive preview) under {docs_path}/survey-design/{YYYY-MM-DD}_<survey-slug>/ and commits it.
  4. Reviewer critique. A single reviewer subagent is dispatched with both rubrics. It returns two files: survey-eval.md (overall-survey critique — structure, ordering, length/burden, mode fit, accessibility, ethics/PII) and question-eval.md (every question evaluated against the anti-pattern catalog, with the specific defect, severity, and a proposed rewrite). The skill surfaces findings to the user in batched AskUserQuestion calls (Fix as proposed / Modify / Skip / Defer per finding).
  5. Apply critique. The skill applies the user's dispositions to survey.html and commits.
  6. Simulated-respondent pass. The skill walks the survey in the voice of the target persona, capturing friction, confusion, language-comprehension gaps, time-to-complete estimate, and drop-off risk. It produces a fix list and asks the user — via batched AskUserQuestion — which fixes to apply. On confirmation it applies them and commits.
  7. Viewer. The skill generates an index.html viewer for the survey folder and a light local serve script (reusing the _shared/html-authoring/ serve.js / viewer.js / index-generator.md pattern), so the user can open the survey in a browser and walk it.
  8. Export. The skill asks the user which platform to target (Typeform / SurveyMonkey / Google Forms; Qualtrics offered as a documented stretch). It generates the import-conducive artifact for that platform and a short "how to import" note, and commits.

Why a generate→critique→simulate→export loop (rather than a one-shot generator): survey quality is dominated by defects that are obvious in review but easy to miss while drafting; a dedicated critique pass with a named rubric is how the toolkit's other skills get reliability, and the simulated-respondent pass adds the "did anyone actually try it" check that catches length and comprehension problems a static rubric won't.

User Journeys

Primary Journey (happy path) — design a new survey

  1. PM runs /survey-design with a paragraph: "I want to understand why trial users don't convert. Audience: people who started a trial in the last 30 days and didn't upgrade. Should take ~3 minutes."
  2. The skill confirms it read: audience = recent non-converting trial users; time budget = ~3 min; mode = it infers generative (understanding "why") but asks to confirm; question cap = none stated. The user confirms mode = generative.
  3. The skill generates a ~7-question survey: a one-line intro, a screening confirmation, an open-ended "what were you hoping to accomplish", a closed "which of these got in your way" (forced-choice yes/no per item per the catalog), an open "what would have changed your mind", a single satisfaction-with-trial scale, and an optional closing free-text. Writes survey.html, commits "survey-design: initial draft for trial-conversion".
  4. The reviewer subagent returns: survey-eval.md says the screening item should come first and the satisfaction scale is unbalanced; question-eval.md flags Q4 as double-barreled ("price and complexity") and Q3 as slightly leading. The skill surfaces 4 findings; the user picks Fix-as-proposed on three and Modify on one (rewords it themselves).
  5. The skill applies the dispositions, commits "survey-design: apply review for trial-conversion".
  6. The simulated-respondent pass walks it as "Maya, a trial user who churned": estimates 2m40s, flags that "onboarding friction" is jargon, suggests Q2 and Q5 could merge. The skill asks which of the 3 fixes to apply; the user takes 2. Applies, commits "survey-design: apply simulation fixes for trial-conversion".
  7. The skill generates index.html + serve script; tells the user node assets/serve.js (or equivalent) to view it.
  8. The skill asks which platform; user picks Google Forms. The skill writes export/build-google-form.gs + export/README.md with the paste-and-run instructions, commits "survey-design: add Google Forms export for trial-conversion".
  9. The skill prints a summary: folder path, commit list, the import file, and the viewer command.

Alternate Journeys

  • Refine an existing survey. The user passes a path to an existing survey (an .html/.md/pasted text). The skill parses it into its internal model, runs variable interpretation (asking only for what the existing survey doesn't make clear), then enters at the reviewer-critique step rather than generating from scratch. All later steps proceed normally.
  • Evaluative survey. Intake mode = evaluative. The generated design leans on closed, comparable items (rating scales, NPS/CSAT where appropriate, A/B-able wording), fewer open-ended items, and the critique rubric weights comparability and scale balance more heavily.
  • Hybrid survey. Mode = hybrid: a generative section (open exploration) followed by an evaluative section (closed measurement), with the funnel-order rule applied across the boundary.
  • Question cap set. The user caps at, say, 8 questions. If the generated design would exceed it, the skill prioritises and trims to the cap, and the simulated pass reports against the cap.
  • Skip the export. The user declines to export (just wants the design + viewer). The skill ends after the viewer step.
  • Non-interactive run. Invoked with --non-interactive (e.g. from /feature-sdlc or a script): the skill picks the recommended option at every gate, logs deferred questions to an open-questions sidecar, and still produces all artifacts.

Error Journeys

  • Intake too thin to proceed. The context is one vague sentence with no audience and no purpose. The skill asks the variable-interpretation questions; if the user still can't say what the survey is for, the skill states it cannot design a survey without a research goal and stops (rather than guessing).
  • Existing survey can't be parsed. The provided file isn't recognisably a survey. The skill says so, shows what it could extract, and offers to treat the content as free-text intake instead.
  • Reviewer subagent fails or returns malformed output. The skill surfaces the failure, keeps the committed draft, and offers to retry the critique or proceed without it (with a warning that the survey is unreviewed).
  • Not in a git repo / dirty tree. The skill's commits are best-effort: if the working tree isn't a git repo, it writes the artifacts and warns that it couldn't commit; it does not refuse to produce the survey.
  • Export platform not chosen / unsupported. If the user names a platform the skill doesn't support, it lists the supported set and offers the closest match (or a generic structured survey.json + the readable HTML).
  • Length can't fit the budget. If even the trimmed design exceeds the time budget for the questions the user insists on keeping, the skill fields the survey anyway but prominently flags the expected drop-off and the overage.

Open Questions

#Question
1Should the skill persist a canonical survey.json alongside survey.html (D5), or keep HTML as the single source and have exporters re-parse it? (Recommendation: persist JSON — to be confirmed in spec.)
2Auto-tier of the skill artifact in /create-skill terms — almost certainly Tier 3 (multi-phase, reviewer subagent, AskUserQuestion gates, reference dir); confirm in spec so the right conventions (non-interactive block, findings protocol) are applied.
3How interactive must survey.html actually be — full fillable form with state, or "visually shows controls but not wired"? (Affects whether the simulated-respondent pass reads the HTML or the JSON model.)
4Does the simulated-respondent pass run as a subagent (cleaner context isolation) or inline? (Leaning inline, since it needs the same survey context the main agent has — but the spec should decide.)
5For the export step — should the skill offer to generate multiple platform files in one run if the user wants, or strictly one per invocation? (User's brief says "where would they want to export" — singular — but multi is cheap.)

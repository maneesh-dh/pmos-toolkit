1. Problem Statement

Product managers and builders run surveys often but aren't survey methodologists, so they ship instruments full of leading questions, double-barreled items, unbalanced scales, missing opt-outs, and excess length — producing biased or unusable data. We are building a new pmos-toolkit skill, /survey-design, that takes a rough survey intent (or an existing survey) and drives it through a generate → reviewer-critique → simulated-respondent → export loop to a methodologically sound, fielding-ready instrument. Primary success metric: a successful run always produces a committed survey.json + survey.html + preview.html + survey-eval.md + question-eval.md + index.html, and (unless the user opts out) at least one platform import file — with the question-critique referencing every question and the simulated time estimate reported against the user's stated budget.

2. Goals

#GoalSuccess metric
G1One invocation turns a paragraph of intent (or an existing survey) into a reviewed, fielding-ready survey.Run produces all canonical artifacts (G-list above) + ≥1 export file unless opted out.
G2The survey respects the audience + time budget the user gave.Simulated-respondent estimate ≤ stated budget, or the overage is explicitly flagged with proposed cuts.
G3Common framing anti-patterns are caught and resolved before fielding.question-eval.md references every question by id; the apply-step records a disposition (fix/modify/skip-with-reason/defer) for every finding.
G4The survey is adapted to its research mode (generative / evaluative / hybrid).Generated question-type mix matches the mode (open-ended-heavy for generative; closed/comparable for evaluative); recorded in survey.json.mode.
G5The user can view and walk the survey before fielding.preview.html renders a fillable form reachable to the end; index.html links it.
G6The survey moves into a real platform without re-typing.For each chosen platform the skill emits a structurally valid import artifact + an import README.
G7Usable interactively and unattended.--non-interactive run completes, auto-picking recommended options and writing an open-questions sidecar.
G8Ships clean against toolkit release prereqs.SKILL.md at canonical path; manifest versions synced in both .claude-plugin + .codex-plugin; README row added; description ≥5 trigger phrases; non-interactive block + findings protocol present.

3. Non-Goals

  • NOT calling platform APIs to create the survey live — because that needs the user's tokens + per-platform auth; the skill emits import files + instructions.
  • NOT handling distribution, sampling panels, recruitment, or quota management — fielding/ops, not design.
  • NOT analyzing survey responses — separate downstream activity.
  • NOT conjoint / MaxDiff / advanced experimental designs in v1 — needs specialist statistical tooling; the skill may suggest them.
  • NOT a general form builder (payment/registration/quiz forms) — scope is research surveys.
  • NOT replacing real pretesting — the simulated-respondent pass is a heuristic stand-in, not cognitive interviews / a soft launch; the skill says so.

D20The 8-step runtime workflow maps to SKILL.md phases 0–9 (see §6.1). Steps 4–5 (critique + apply) and step 6 (simulation + apply) are each a dispatch → structured-findings → batched-AskUserQuestion apply-loop → re-render → commit micro-cycle. The user can always re-invoke critique.

5. User Personas & Journeys

5.1 PM / builder (primary)

Comfortable writing prose, knows their product domain, not a research-methods expert. Runs /survey-design in their product repo (which may or may not have pmos set up). Wants trustworthy survey data without becoming a methodologist. Secondary: someone with an existing survey wanting a methods second-opinion before sending.

5.2 Journey: design a new survey

(Detailed in 01_requirements.html#primary-journey.) Intake paragraph → confirm interpreted variables (audience / time budget / mode / optional cap; AskUserQuestion for gaps) → generate survey.json + render survey.html/preview.html + commit → reviewer subagent → batched-AskUserQuestion dispositions → apply + re-render + commit → simulated-respondent subagent → batched-AskUserQuestion which fixes → apply + re-render + commit → generate index.html + serve.js → ask export platform(s) → generate export/* + README + commit → print summary.

5.3 Journey: refine an existing survey

User passes a path to an existing survey (HTML / markdown / pasted text). The skill parses it into a best-effort survey.json, runs variable interpretation (asking only what the source doesn't make clear), then enters at the reviewer-critique step. All later steps proceed normally. If the source can't be parsed as a survey, the skill says so, shows what it extracted, and offers to treat the content as free-text intake instead.

7.2 Intake & variable interpretation

IDRequirement
FR-10The skill accepts as its argument: free-text survey intent, OR a path to an existing survey (any of .html/.md/.txt), OR nothing. With no argument it MUST ask (AskUserQuestion) for the survey's purpose + audience before doing anything else.
FR-11Given an existing-survey path, the skill MUST attempt to parse it into a survey.json (best-effort: titles → sections, recognizable questions/options → questions). If parsing yields nothing recognizable, it MUST report so, show what it extracted, and offer to treat the content as free-text intake.
FR-12The skill MUST attempt to infer four variables from the context: audience (free text), time_budget_min (integer minutes; if the user gave a range, use the upper bound), mode (one of generative / evaluative / hybrid), max_questions (integer or null). It records what it inferred and presents it back to the user.
FR-13For each variable it could NOT confidently infer, the skill MUST ask via a single batched AskUserQuestion call — one question per missing variable (up to 4). Each question's options MUST include a (Recommended) default (mode: recommend hybrid when genuinely ambiguous; time budget: recommend "~5 minutes" for a general audience; cap: recommend "no cap"; audience: no recommended default — that one is defer-only: free-form if it must be asked, since there's no sensible auto-pick — but in practice the audience is almost always in the context).
FR-14If the user cannot articulate a research goal/purpose for the survey, the skill MUST state it cannot design a survey without one and stop (it MUST NOT guess a purpose).

7.3 Initial design generation (Phase 3)

IDRequirement
FR-20The skill MUST build a survey.json conforming to the schema in §9.1, applying reference/survey-best-practices.md: an intro/consent block; logical sections; funnel order (general → specific); screening/qualifying questions first; demographics/sensitive items last (unless used for routing); balanced rating scales with labeled poles + opt-out options; mode-appropriate question-type mix (open-ended-heavy for generative; closed/comparable for evaluative; a generative section then an evaluative section for hybrid); a closing optional open-ended catch-all.
FR-21The generated survey MUST aim to fit the time budget. The skill estimates time as ≈ (Σ per-question cost) where open-text ≈ 30s, closed single/multi ≈ 8s, rating/scale ≈ 6s, matrix row ≈ 5s, intro/consent ≈ 15s (these constants live in the skill and may be tuned). If the estimate exceeds time_budget_min, the skill MUST trim/prioritize down to the budget (and to max_questions if set) before writing — keeping screening + the highest-value items.
FR-22The skill MUST NOT generate any of the anti-patterns catalogued in reference/question-antipatterns.md (no leading/loaded/double-barreled/double-negative/jargon/absolute stems; no unbalanced scales; no overlapping ranges; no non-exhaustive options; no question/answer-scale mismatch; no agree/disagree batteries — use construct-specific scales; no "select all that apply" where per-item prevalence matters — use forced-choice yes/no grids; no required sensitive items without an opt-out; no PII not needed by the stated purpose; no "anonymous" claim if any identifier is collected).
FR-23After building survey.json, the skill MUST render survey.html (substrate-compliant: uses template.html shape; every <h2>/<h3> has a kebab-case id; no inline <script>/<style> in the body; assets referenced as assets/…?v=<plugin-version>; companion survey.sections.json written) AND preview.html + survey-preview.js (§11), AND seed/refresh index.html.
FR-24The skill MUST git add + git commit the new artifacts with message survey-design: initial draft for <slug>. If the cwd is not a git repo or the commit fails, it MUST warn and continue (best-effort, D9). Folder-slug collisions MUST be resolved by appending -2/-3/… — never overwrite an existing survey folder.

7.4 Reviewer critique (Phases 4–5)

IDRequirement
FR-30The skill MUST dispatch ONE reviewer subagent (Task tool, or sequential inline if subagents unavailable). The prompt MUST contain: the survey context ({audience, time_budget_min, mode, max_questions, estimated_minutes}), the full survey.json as text, and the contents (or paths) of reference/survey-best-practices.md and reference/question-antipatterns.md. In non-interactive mode the prompt's first line MUST be [mode: non-interactive].
FR-31The reviewer subagent's return contract: a survey-level critique body and a per-question critique body, where (a) the per-question body has exactly one section per question keyed by question.id (verdict "no issues" allowed); (b) every finding is the structured shape of D17 (target, severity ∈ {blocker, should-fix, nit}, defect, message, proposed_fix); (c) survey-level concerns covered: section structure + order (funnel), length vs. budget, mode fit, scale-balance summary, accessibility, ethics/PII, intro/consent quality.
FR-32The parent MUST write the two bodies to survey-eval.md and question-eval.md (markdown, not HTML — these are subagent critique sidecars, matching the <skill>-eval convention). The parent MUST verify count(per-question sections) == count(questions in survey.json); on mismatch it MUST re-dispatch once with the missing ids called out, and if still mismatched, surface the gap to the user and proceed without auto-applying.
FR-33The skill MUST present the findings to the user via batched AskUserQuestion calls: group by category, ≤4 per call, one question per finding, severity-tagged ([Blocker]/[Should-fix]/[Nit]), options Fix as proposed / Modify / Skip / Defer. In non-interactive mode it MUST auto-pick Fix as proposed for blocker/should-fix and Defer for nit, logging each to the OQ buffer.
FR-34The skill MUST apply the dispositions by mutating survey.json (never editing the rendered files directly), then re-render survey.html/preview.html/index.html, append a "Dispositions" section to question-eval.md/survey-eval.md, and git commit with message survey-design: apply review for <slug>.
FR-35If the reviewer subagent fails or returns malformed output, the skill MUST surface the failure, keep the committed draft, and offer (AskUserQuestion) to retry the critique or proceed without it (warning that the survey is unreviewed). In non-interactive mode it proceeds without it and logs the failure.

7.5 Simulated-respondent pass (Phase 6)

IDRequirement
FR-40The skill MUST dispatch the simulated-respondent subagent once per persona. Default = one persona = the stated audience; the skill MAY derive a second persona only if the audience is clearly heterogeneous (e.g., "trial users AND admins"). The prompt MUST contain: the survey content (the rendered question text + options, or survey.json as text), the persona description, and a friction-walk rubric (walk question-by-question; for each, note any friction / confusion / comprehension gap / time pressure; estimate cumulative minutes; flag drop-off-risk points).
FR-41The subagent's return contract: { persona, estimated_minutes, per_question: [{id, friction[], confusion[], comprehension_gaps[]}], dropoff_risk_points: [{after_id, reason}], overall_notes }.
FR-42The skill MUST write the simulated-respondent report(s) to simulation.md and derive a fix list (each fix: {target_question_id_or_survey, problem, proposed_fix}). It MUST compare estimated_minutes to time_budget_min: if over budget, the fix list MUST include concrete cut proposals; if the user keeps over-budget questions, the skill MUST prominently flag the expected drop-off + overage in the summary.
FR-43The skill MUST present the fix list via batched AskUserQuestion (same protocol as FR-33: ≤4/call, one Q per fix, options Apply / Modify / Skip / Defer, recommended = Apply for clear comprehension/length fixes). It MUST then apply chosen fixes to survey.json, re-render, append a "Dispositions" section to simulation.md, and git commit with message survey-design: apply simulation fixes for <slug>.
FR-44The skill's output MUST state, near the simulation results, that this pass is a heuristic stand-in and does not replace cognitive interviews / a soft launch (Non-Goal restated; D-requirements).

9. Data Contracts (survey.json schema + subagent & export contracts)

(There is no HTTP API and no database. This section plays the role of §9/§10 — it specifies the canonical survey.json schema and the subagent/export contracts.)

9.1 survey.json schema

JSON object. Required keys marked (req).

{
  "schema_version": 1,                         // (req) int
  "title": "Trial conversion — exit survey",   // (req) string
  "purpose": "Understand why recent trial users did not upgrade.",  // (req) string — the research goal
  "mode": "generative",                        // (req) "generative" | "evaluative" | "hybrid"
  "audience": "People who started a trial in the last 30 days and did not upgrade.",  // (req) string
  "time_budget_min": 3,                        // (req) int — target completion minutes
  "estimated_minutes": 2.7,                    // (req) number — skill's estimate (FR-21 constants)
  "max_questions": null,                        // int | null
  "intro": {                                   // (req)
    "text": "Thanks for trying . A few quick questions — about 3 minutes. Your answers are confidential and help us improve.",  // (req) string
    "consent_required": false,                 // bool — if true, an explicit "I agree" gate precedes Q1
    "anonymous": false,                        // bool — MUST be false if any PII question exists
    "estimated_seconds": 15                    // number
  },
  "sections": [                                // (req) array, ≥1
    {
      "id": "screening",                       // (req) kebab id, unique
      "title": "First, a quick check",         // (req) string
      "description": null,                     // string | null — signpost text
      "randomize_questions": false,            // bool — never true for screening / ordinal-dependent sections
      "questions": [ /* see below */ ]         // (req) array, ≥1
    }
  ]
}

Each question object:

{
  "id": "q-hoped-to-do",            // (req) kebab id, unique across the whole survey
  "type": "open_long",              // (req) see type list below
  "stem": "What were you hoping to accomplish when you started the trial?",  // (req) string
  "help_text": null,                // string | null — shown under the stem
  "required": false,                // (req) bool
  "reference_period": null,         // string | null — e.g. "in the past 7 days"; required for retrospective/frequency questions
  "screening": false,               // bool — true => answer drives skip logic; such questions come first
  "skip_logic": null,               // null | { "on_value": , "action": "skip_to" | "end_survey", "target_section_id":  }
  "randomize_options": false,       // bool — true only for nominal (unordered) option lists; never for ordinal scales

  // type-specific fields:
  "options": [                      // for single_select / multi_select / forced_choice_grid (rows) / ranking
    { "value": "price", "label": "The price was too high" }
  ],
  "other_option": false,            // bool — appends an "Other (please specify)" free-text option
  "opt_out_options": [],            // array of {value,label} appended & visually separated, e.g. [{ "value":"na","label":"Not applicable" },{ "value":"dk","label":"Don't know" },{ "value":"pnts","label":"Prefer not to say" }]

  "scale": {                        // for rating / nps
    "points": 5,                    // int — 5 or 7 default; NPS implies 11 (0..10)
    "min": 1, "max": 5,             // ints — NPS: 0..10
    "labels": { "min": "Not at all satisfied", "mid": "Neither", "max": "Extremely satisfied" },  // pole labels; "mid" only for odd scales
    "balanced": true                // bool — equal #positive/#negative around midpoint (MUST be true unless purpose forces a forced-choice even scale)
  },

  "rows": [ { "id":"r-ease","label":"Ease of use" } ],   // for forced_choice_grid / matrix — the items being rated

  "constant_sum_total": 100         // for constant_sum

Schema invariants (the skill and any consumer enforce these): all ids kebab-case and unique; anonymous: true forbids any PII question; rating/nps scales balanced: true unless purpose explicitly justifies a forced even scale; ordinal types never have randomize_options: true; skip_logic.target_section_id (when action is skip_to) MUST reference an existing later section; retrospective/frequency stems MUST set reference_period; required: true on a sensitive item MUST be accompanied by an opt_out_options entry.

9.2 Reviewer subagent contract

Inputs (in the prompt): survey context object; survey.json as text; reference/survey-best-practices.md + reference/question-antipatterns.md contents (or paths if the subagent can read files); [mode: …] first line in non-interactive runs.
Output (two bodies):

  • question-eval.md body: ## <question-id> per question, each containing the stem (quoted), a verdict line (Verdict: no issues or Verdict: N finding(s)), then a findings table with columns Severity | Defect | Message | Proposed fix per D17.
  • survey-eval.md body: sections for Structure & order, Length vs. budget, Mode fit, Scale balance, Accessibility, Ethics/PII, Intro/consent — each with the same findings-table shape; plus an overall verdict line.

Parent-side validation: count of ## <id> sections in question-eval.md == count of questions in survey.json; on mismatch, one re-dispatch naming the missing ids, then surface-and-proceed.

9.3 Simulated-respondent subagent contract

Inputs: the rendered question text + options (or survey.json as text); the persona description; a friction-walk rubric (walk Q-by-Q; for each note friction/confusion/comprehension gaps; estimate cumulative minutes; flag drop-off-risk points); [mode: …] first line in non-interactive runs.
Output (JSON or markdown that the parent can parse): { persona, estimated_minutes, per_question: [{id, friction:[…], confusion:[…], comprehension_gaps:[…]}], dropoff_risk_points: [{after_id, reason}], overall_notes }.

10. Frontend Design (generated artifacts)

10.1 survey.html — readable rendering

Substrate-compliant. Uses template.html shape (toolbar with Copy-Markdown/Copy-link, <main> body, footer). Body structure: a <section id="intro"> rendering the intro/consent block; one <section id="<section-id>"> per survey section with an <h2> = section title; within each, each question rendered as an <h3 id="<question-id>"> = stem, the help text as a <p>, options as a <ul> (with opt-out items visually separated by a rule), scales as a small <table>, matrix/grid as a <table>; metadata footer-ish line ("Mode: generative · Target: ~3 min · Estimated: 2.7 min · 7 questions"). Companion survey.sections.json enumerates all section + h2 + h3 ids. No inline <script>/<style>.

10.2 preview.html + survey-preview.js — fillable preview

Self-contained. preview.html: a minimal HTML page with an inline <style> (small, mobile-first, accessible: label-adjacent controls, ≥4.5:1 contrast, visible focus, "Question X of Y" text), an inline <script type="application/json" id="survey-data"> holding the full survey.json, and a <script src="survey-preview.js"></script>. (Note: preview.html is intentionally NOT substrate-compliant — it's a standalone interactive page, not a pmos artifact; that's why it's a sibling, not survey.html — D4.)

survey-preview.js (ASCII-only, no deps, no CDN): on load, parse #survey-data; render the intro screen (with the consent gate if consent_required); then render one section at a time (or one question at a time on narrow viewports) with Next/Back; maintain answer state in a plain JS object (in-memory only — no storage); evaluate skip_logic on advance (skip to target section / end survey); render each question type with the appropriate control (radio / checkbox / Yes-No-per-row grid / scale buttons / number / textarea / ranking via up-down or numbered selects / matrix table); show a running "Question X of Y" indicator (Y = count of non-statement questions on the active path); on the final screen show the thank-you text and an optional "Show my answers (JSON)" debug toggle. Required-question validation gates Next. No network calls.

14. Rollout Strategy

  • No feature flag — a new skill is purely additive; it's auto-discovered from plugins/pmos-toolkit/skills/.
  • Release: minor version bump in BOTH plugins/pmos-toolkit/.claude-plugin/plugin.json and plugins/pmos-toolkit/.codex-plugin/plugin.json (kept byte-identical — pre-push hook enforces). Add a README row under the appropriate section (e.g., "Standalone utilities" — /survey-design is not a pipeline stage). Add a ## /survey-design header to ~/.pmos/learnings.md (idempotent). Changelog entry describing what the system can now do.
  • Rollback = revert the commit; no migration, no data, no flag to flip.
  • Qualtrics export is shipped (or not) independently; if not shipped in v1, the only "rollout" cost later is adding the transformer + un-gating the export option.

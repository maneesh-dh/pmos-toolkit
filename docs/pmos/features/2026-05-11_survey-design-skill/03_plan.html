Constraints discovered

  • Canonical skill path is load-bearing and silent on failure (repo CLAUDE.md): SKILL.md MUST be at plugins/pmos-toolkit/skills/survey-design/SKILL.md — anywhere else and the skill simply doesn't register, with no error.   • Manifest version sync is pre-push-hook-enforced: plugins/pmos-toolkit/.claude-plugin/plugin.json and plugins/pmos-toolkit/.codex-plugin/plugin.json must carry the identical version on every release; T9 edits both in one commit (R4).
  • Release entry point is /complete-dev, not the legacy /push — SKILL.md's "Release prerequisites" section must point at /complete-dev (T8).
  • preview.html must work on double-click (file://) — no fetch(), no CDN, no external refs (NFR-02, D12): the full survey.json is inlined into preview.html as <script type="application/json" id="survey-data"> and survey-preview.js reads it from the DOM. survey-preview.js source is ASCII-only (no smart quotes / non-ASCII identifiers).
  • Two distinct HTML conventions in play: survey.html IS a pmos artifact → substrate-compliant (heading ids, no inline script/style, cache-busted local assets, sections.json companion); preview.html is intentionally NOT a pmos artifact → it's a standalone interactive page with an inline <style> and the inlined JSON + a <script src="survey-preview.js">. Don't conflate them (D4).
  • The shell's ls in this environment can emit nothing in non-tty — verification steps use /bin/ls or find, never bare ls, when their output is asserted on.
  • No plugin.json skills-array edit: skills are auto-discovered from plugins/pmos-toolkit/skills/; touching the array is wrong.

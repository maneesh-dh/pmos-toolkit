## STREAM 1 — Survey design best practices

### Summary
- Keep it short: <5 min / ~10–15 questions for general audiences; >12 min ≈ 3× the dropout of sub-5-min; in-app/intercept 1–3 questions max. Email ≈ 3% response; in-context microsurveys 40–50%.
- Order matters and creates bias: funnel general→specific (specific items prime general ones); warm up with easy non-sensitive items; group by topic; open-ended "biggest problem" before related closed items; demographics + sensitive items at the end (unless screening/routing).
- Closed-ended for measurement, open-ended for discovery. Generative surveys lean open/broad; evaluative lean closed/comparable/quantitative. Don't ask people to predict future behavior — ask recent concrete time-anchored behavior.
- Scale design is deliberate trade-offs: 5-point = familiar default; 7-point = more reliability for nuanced/employee research; even-point (4/6, forced choice) only when you genuinely want to push off the fence — and even then add a separate "Don't know"/"N/A" rather than relying on a midpoint to absorb non-opinions. Always label poles (+ midpoint if odd); balanced scales; logical order — never randomize an ordinal scale (but may reverse for half the sample).
- Avoid agree/disagree formats — acquiescence bias (worse for lower-education respondents, with an interviewer present). Use construct-specific options ("How easy was X? Very easy … Very difficult") or a choice between two competing statements.
- Randomize to spread bias, not eliminate it: randomize/rotate answer-option order for nominal lists; randomize question/block order where no logical sequence required. Avoid select-all-that-apply — forced-choice yes/no per item is more accurate.
- Build in data-quality guards: multiple short attention checks, time-to-complete thresholds (speeders), straightlining detection on grids — but treat as one signal, not the sole gate (over-screening biases toward higher-education respondents).
- Pre-test before launch: expert review (3–5) → iterative cognitive interviews (3 rounds of ~5 beats 1 round of 15; 5–7 catch 75–85% of major problems) → revise → soft launch ~50–100 (or 5–10% of target). Mobile-first (one question per screen, large tap targets), WCAG 2.1 AA, "Question 3 of 10" over graphical progress bars. Informed consent; minimize PII; state anonymity vs. confidentiality honestly; GDPR/CCPA.

### Detail

**Structure & flow.** Intro/consent screen: state sponsor, purpose, accurate completion-time estimate, what's collected/how used/stored, anonymous vs. confidential (don't conflate), voluntariness, contact; for research, an affirmative consent action. Warm-up: simple, interesting, low-effort, non-sensitive first; common Pew pattern = open "most important problem" before related closed items. Logical grouping: group by topic, signpost transitions, place instructions right before the question they govern (Dillman), use item-in-a-series/matrix carefully. Ordering effects: *contrast* (specific-before-general makes respondents exclude the specific topic — Pew: presidential approval before "satisfaction with country" pushed dissatisfied 78%→88%); *assimilation* (consecutive similar items pull toward consistency — "Dems work with Reps?" before reverse moved agreement 66%→81%); *priming*; don't cluster difficult items. Funnel: broad first, narrowing. Sensitive/demographic late (exception: screeners early). Closing: optional open catch-all; thank-you screen.

**Question types.** Open-ended (discovery, surface vocabulary; costly; under-reports vs. prompted list — Pew: economy 35% unprompted vs. 58% listed; build closed list from pilot open responses) vs. closed-ended (workhorse; "mostly closed-ended with a smattering of open-ended" — NN/g). Single-select radio: MECE options, ~4–5 for attitudinal, include opt-out. Multi-select "select all that apply": lower data quality (satisficing, ambiguous blanks) — prefer forced-choice Yes/No grid; randomize options if used. Ranking: relative preference, short list (≤5–7), cognitively heavy; Top-3 pick easier. Rating scales: Likert (unipolar construct; prefer construct-specific over agree/disagree; 5 or 7 pts); semantic differential (bipolar, brand/concept across dimensions); NPS ("likely to recommend?" 0–10; promoters 9–10, passives 7–8, detractors 0–6; NPS=%P−%D; pair with open follow-up; single-item — don't over-interpret); star ratings (quick, coarse, ceiling effects); slider (interactive; starting-position bias, mobile non-response risk — handle "untouched"). Matrix/grid: compact but top cause of straightlining/break-offs; keep rows ≤5–7, split, randomize rows, one-item-per-screen on mobile. Dichotomous Yes/No: clear binaries, eligibility, forced-choice grids; add "Don't know" when uncertainty plausible. Constant-sum: forces explicit trade-offs ("allocate 100 points"); demanding; small item count; running total.

**Scale design.** Odd (5/7/9): midpoint = genuine neutrality — use when "no opinion/neutral" is legitimate. Even (4/6): forces a lean — only when you deliberately want to push to a side; caveat: removing midpoint doesn't make ambivalent people informed — better to keep odd + add a separate visually-offset "Don't know"/"N/A". 5-pt = best default for general/customer; 7-pt = more nuance for research/employee; >7 = diminishing returns. Always label the two poles + midpoint (odd); no strong evidence labeling every point beats labeling some, but fully verbal labels aid interpretation. Endpoint intensity matters: extreme labels ("Extremely satisfied") → peaked distribution; moderate labels ("Very satisfied") → people use extremes more. Provide an opt-out (N/A / Don't know / Prefer not to answer / None) — don't bundle multiple opt-outs into one; visually separate from the scale. Balanced: equal positive/negative, equivalent intensity. Direction: logical order, consistent within survey; reverse for a random half to distribute primacy/recency; numeric 0–10 low-to-high left-to-right. Acquiescence: replace agree/disagree with forced choice between opposing statements OR construct-specific options; if you must use batteries, mix reverse-coded items.

**Length & burden.** <5 min for general audiences; quality + completion degrade past ~10–15 min; >12 min ≈ 3× drop-off. Email/web survey ~10–15 questions sweet spot; in-app/intercept/microsurvey 1–3 (ideally 1–2) in context. Burden is psychological — respondents rate it by # questions + perceived length, not actual time; don't pad. Drop-off curve: micro/1-question ≈ 85% completion; 1–3 ≈ 83%; falls with each added question, steepest early. Progress indicators: show progress ("Question 3 of 10" or bar) to reduce abandonment — but a bar that crawls early can increase drop-off; numeric "X of Y" more screen-reader-friendly; don't show a bar if heavy skip logic makes "% complete" misleading. Pagination: one question per page on mobile; logical section breaks; one-question-per-page also helps screen readers + skip logic, at the cost of more clicks.

**Sampling & audience.** Define target audience precisely first (role, tenure, usage, geography, plan tier) — screener + quota logic flow from it. Screening questions: short, up front, neutral wording so people can't game in, don't reveal which answer qualifies, avoid over-screening. Quota logic: cell quotas (N per segment or proportional), close cells as they fill, decide interlocking vs. marginal in advance, track during fielding. Sample size: driven by precision needed on key estimates + subgroup analysis; account for break-off + disqualification when sizing the invite list.

**Generative vs. evaluative.** Generative/discovery ("what problems to solve? what's the landscape?"): heavy open-ended + broad questions; minimal pre-imposed structure; closed items exploratory not confirmatory; goal = breadth, surfacing the unknown (vocabulary, JTBD, unanticipated needs); surveys are a weak primary tool here — use to size/prioritize themes interviews surfaced. Evaluative/validation ("how well does X work? is H true? how big is the effect?"): heavy closed-ended, comparable, quantitative; pre-defined options for cross-respondent comparison + stats; tight hypothesis-targeted questions; rating scales, NPS/CSAT, A/B wording tests, benchmarking; wording/order/context must be locked + held constant across waves. Decide which mode you're in *before* designing — it changes question-type mix, scale use, length, and whether you randomize (exploration) or freeze (comparability).

**Reducing bias & error.** Question-order randomization (independent blocks where no logical sequence required — spreads order/context bias). Answer-option randomization/rotation for nominal lists (counters primacy in visual/self-administered, recency in aural/phone) — don't randomize ordered scales; reverse for a random half. Avoid priming (no loaded preambles; general before specific). Social-desirability mitigation: self-administered modes reduce it; word sensitive items non-judgmentally; normalize the undesirable answer ("Many people don't get around to … did you happen to …?"); provide several low/none options; offer "Prefer not to answer"; list experiments/randomized response for highly sensitive topics. Known: under-report alcohol/drugs/tax evasion/prejudice; over-report voting/church/charity/exercise. Satisficing/straightlining detection: 2+ short attention checks (instructed-response items), flag speeders via time vs. realistic floor, flag straightliners via low/zero grid-row variance, flag inconsistent reverse-coded pairs — treat as one indicator; aggressive exclusion correlates with demographics so pre-register exclusion rules; designing *out* satisficing (shorter, fewer grids, varied types, clear language) matters more than detecting it. Recall bias/timeframe anchoring: don't ask "how often do you usually…" — anchor to a concrete recent window ("In the past 7 days, approximately how many times…"); use "approximately" + ranges; shorter recall windows for frequent/mundane events; never ask people to predict future behavior.

**Writing the questions.** Plain language (as you'd ask aloud; vocabulary for the least-educated respondent; no jargon/acronyms/double negatives). One concept per question (split "easy and intuitive", split "domestic and foreign policy"; "and"/"or" joining two ideas in the stem = suspect double-barrel). Concrete & specific (ambiguous "regularly"/"recently"/"this area"/"good value" — define or replace; word choice swings results — Pew: "military action in Iraq" 68%→43% with "even if it meant thousands of U.S. casualties"; "welfare" vs. "assistance to the poor"; A/B-test wording on random halves when stakes high). MECE answer options (no overlapping ranges — "20–29"/"30–39" not "20–30"/"30–40"; cover every plausible answer or include "Other (specify)"/opt-out). Consistent time references (same window across related items; state it in the stem). Avoid hypotheticals ("Would you use X if we built it?" / "How likely are you to…?" unreliable — reframe to past behavior, current needs, revealed preference). Neutral/non-leading (no pleading wording, no loaded premises, balanced options). Respectful/inclusive (current demographic terminology; gender beyond binary + "prefer not to say"; objective phrasing for conditions/behaviors).

**Pre-testing/piloting.** Sequence: (1) expert review 3–5; (2) cognitive interviews (think-aloud/probing with target users); (3) revise; (4) soft launch/field pilot; (5) analyze + iterate; (6) full launch. Cognitive interview sample: 5–10 per round; 5–7 catch ~75–85% of major problems; 10–15 catch most interpretation problems; iteration beats size (3 rounds of ~5 with revisions > 1 round of 15); cognitive interviews reliably *identify* problems but don't guarantee the *fix* improves data — re-test revisions. Soft launch/pilot: ~50–100 responses or 5–10% of target (whichever smaller); check completion rate, drop-off points, item non-response, straightlining, time-to-complete, skip-logic correctness, whether open-ended answers reveal missing closed options. Also pilot the plumbing (device/browser rendering, skip logic, quotas, data export, randomization).

**Accessibility & mobile-first.** Mobile-first: design for small screens first; one question (or small group) per screen; avoid wide matrices (reformat per-item); large tap targets; minimal typing (prefer choices); thumb-reachable; no horizontal scroll; test on real devices. Accessibility (WCAG 2.1 AA): radio/checkbox immediately adjacent to label + proper label association; don't rely on a graphical progress bar (use "Question X of Y" + state total up front); self-paced (no timeouts); text equivalents/alt text/transcripts for images in questions/answers; sufficient color contrast, don't encode meaning in color alone; keyboard-navigable; logical focus order; works with screen readers + zoom; sliders problematic for keyboard/SR — provide accessible alternative or ARIA. Run the platform accessibility checker; manually test with a screen reader + keyboard-only.

**Ethics/consent/PII.** Informed consent: plain-language statement before data collection — who you are, purpose, what's collected, how used/stored, retention, who sees it, voluntary + can stop, risks, incentive terms, contact; for research require affirmative action. Anonymity vs. confidentiality (distinct, don't conflate): anonymous = responses can't be linked to identity by anyone (don't promise if you collect email/IP/account ID/device fingerprint); confidential = identity known but protected, not disclosed. PII minimization: collect the minimum necessary; avoid identifiers you won't use; if you must collect PII, justify it, store separately/encrypted, restrict access, set retention/deletion schedule, anonymize/aggregate for analysis + sharing. GDPR: consent freely given/specific/informed/unambiguous, by affirmative action; data-subject rights (access, deletion); plain-language notices. CCPA/CPRA: disclosure/opt-out obligations. Sensitive topics (health, finances, sexuality, immigration, illegal behavior, protected characteristics): make optional, offer "prefer not to answer", consider whether you need them at all.

## STREAM 2 — Survey question anti-pattern catalog

### Summary
- Wording moves numbers more than people realize (Pew split-sample: single-word swaps shift agreement 7+ points; closed vs open framing 35%→58%). A reviewer rubric must treat wording as a measurement instrument, not cosmetics.
- The big four framing defects — leading/loaded, double-barreled, vague/ambiguous, absolutes — are the most cited and the easiest to detect mechanically (presupposition phrases, "and/or" conjunctions, unanchored adverbs, universal quantifiers).
- Response-option defects are as damaging as question-text defects: unbalanced scales, missing/forced midpoints, overlapping ranges, non-exhaustive options, no opt-out, question/answer-scale mismatch. Review stem + options together, always.
- Format choices induce systematic bias independent of content: agree/disagree → acquiescence (worse for less-educated, per Pew); long grids → straight-lining; "select all that apply" → under-reporting vs forced-choice yes/no; option order → primacy/recency.
- Respondent capability is a hard constraint: don't ask about behavior/products they may not have, events they can't recall precisely, percentages they can't compute, futures they can't predict ("Would you buy this?" is notoriously non-predictive).
- Context contaminates: earlier questions prime later ones (assimilation/contrast); buried instructions and long preambles add load that amplifies acquiescence + satisficing.
- Always pretest — every major methodology source treats it as non-negotiable; a static rubric complements, doesn't replace, pretesting.
- Ethics is part of quality: unnecessary PII, pseudo-anonymity claims, forced-answer questions with no "prefer not to say" reduce response rates, increase break-offs, bias the sample toward compliant respondents.

### Catalog (each: name | category | harm | BAD | FIXED | detection heuristic)

**A. Question-stem framing defects**

- **A1. Leading question** | Framing/bias | Steers toward an answer; measures the wording not the opinion. BAD: "How short was Napoleon?" / "Should concerned parents use infant car seats?" FIXED: "How would you describe Napoleon's height?" / "Do you think infant car seats should be required for infant passengers?" Heuristic: flag evaluative adjectives/adverbs in the stem ("short", "concerned", "generous", "excessive"); any phrasing naming only one side.
- **A2. Loaded question / presupposition** | Framing/bias | Embeds an assumption the respondent may not share; forces accepting the premise to answer ("Have you stopped…?"). BAD: "Where do you enjoy drinking beer?" / "What do you like most about our new dashboard?" FIXED: add a screening/filter question first (skip logic) or neutralize: "Have you used the dashboard? If yes, what is your overall impression?" Heuristic: does the stem assert a fact (behavior, feeling, prior usage, shared value) before asking? Look for "our excellent…", "since you enjoy…", "what do you like about…".
- **A3. "Don't you agree…" / rhetorical pressure** | Framing/bias (leading + acquiescence) | Signals the "correct" answer; acquiescence compounds. BAD: "Don't you agree that our checkout process is easy to use?" FIXED: "How easy or difficult was the checkout process?" (balanced very easy→very difficult). Heuristic: flag stems starting "Don't you…", "Wouldn't you say…", "Isn't it true that…", "Most people think X — do you?"
- **A4. Emotionally charged / loaded wording** | Framing/bias | Affect-laden terms ("suicide", "slaughter", "handout", "freedom") move responses by framing. BAD: "Do you support doctors helping terminally ill patients commit suicide?" FIXED: "Do you favor or oppose allowing doctors to give terminally ill patients the means to end their lives?" (test both wordings). Heuristic: flag strongly-connotated terms where a neutral synonym exists; flag asymmetric framing ("favor" without "or oppose").
- **A5. Double-barreled question** | Structure | Two distinct asks in one item; a respondent who feels differently about each can't answer; uninterpretable. BAD: "How satisfied are you with the pay and benefits of your job?" / "confidence in the President to handle domestic and foreign policy?" / "Was the textbook useful for students and young professionals?" FIXED: split into separate items. Heuristic: flag "and"/"or" joining two concepts/objects/time periods/audiences in the stem; "X and Y" where someone could rate X high and Y low.
- **A6. Double negative / confusing negation** | Clarity | Cognitive load + misreads; a "yes" may mean the opposite of intent. BAD: "Do you oppose not allowing employees to work from home?" / "Congress should not fail to act." FIXED: "Do you think employees should be allowed to work from home?" Heuristic: flag ≥2 negations ("not", "no", "never", "oppose", "fail", "prohibit", "without", "un-") in one sentence; "not un-" constructions.
- **A7. Ambiguous / undefined terms** | Clarity | Respondents silently interpret differently → answers not comparable. BAD: "How often do you exercise?" / "Was the room clean?" (vs spotless?) FIXED: define in the stem: "In a typical week, how many days do you do at least 30 minutes of moderate physical activity (e.g., brisk walking, cycling)?" Heuristic: flag any key noun/verb 3 people could define 3 ways ("regularly", "household", "income", "use", "recently"); require an inline definition or example.
- **A8. Jargon / acronyms / insider language** | Clarity | Respondents who don't know the term guess or drop out; biases the sample toward insiders. BAD: "How satisfied with our NPS and CSAT touchpoints?" / "Do you own a tablet PC?" FIXED: plain language + examples: "Do you own a tablet, such as an iPad or Android tablet?" Heuristic: flag acronyms not expanded on first use; domain/product/feature terms without a gloss; reading-level check.

**B. Respondent-capability violations**

- **B1. Assumes knowledge/experience the respondent may lack** | Capability/presupposition | Respondents who never used the thing still answer (to be agreeable) → non-attitudes recorded as attitudes. BAD: "How satisfied are you with our new dashboard?" (asked of everyone). FIXED: gate it: "Have you used the new dashboard in the past 30 days?" → if yes, "How satisfied are you with it?" with an explicit "Haven't used it" backstop. Heuristic: does the stem assume usage/awareness/possession? Is there a screening question and/or an "I haven't used this / not familiar" option? If not, flag.
- **B2. Recall / memory overload** | Capability | Can't accurately recall distant or high-frequency mundane events; precision is fabricated; longer recall periods → larger, biased error. BAD: "How many times did you visit our website in the last 12 months?" / "How much did you spend on groceries last year?" FIXED: shorten the reference period and/or coarsen: "In the past 7 days, how many times did you visit our website?" or bands ("0, 1–2, 3–5, 6+"). Heuristic: flag exact-count questions about frequent/mundane behaviors; reference periods >~30 days for routine events or >~12 months for salient ones; prefer bands over open numeric for recalled counts.
- **B3. Asking for precision the respondent can't give** | Capability | "Best-guess" precision masquerades as data; respondents anchor on round numbers or quit. BAD: "What percentage of your work time is spent in meetings?" / "How many emails did you receive yesterday?" FIXED: ranges or coarse scale: "Roughly how much of your work time is spent in meetings? (None / A little — under 25% / Some — 25–50% / A lot — over 50%)." Heuristic: flag requests for percentages, exact counts of hard-to-track items, dollar figures to the dollar, "how many minutes" — unless the respondent plausibly has a record.
- **B4. Hypotheticals / future-prediction** | Capability/validity | Stated intentions (esp. purchase intent) are weak predictors; respondents over-claim interest; the real-decision context is absent. BAD: "Would you buy this product if we launched it?" / "Would you use a feature that did X?" FIXED: anchor to past behavior/revealed preference: "In the past 6 months, have you bought a product like this? Which one?" — or if you must ask intent, add price/effort + a calibrated scale ("definitely will / probably will / might or might not / probably won't / definitely won't") and treat as directional only. Heuristic: flag "Would you…", "If we built…, would you…", "How likely are you to [future action]" without realistic constraints; flag any decision being made on raw intent data.

**C. Response-option defects**

- **C1. Unbalanced response scale** | Scale design | More positive than negative options (or stronger positive anchors) pulls the mean up; the scale is leading. BAD: "Outstanding / Excellent / Very good / Good / Fair" (4 positive, 1 weak low). FIXED: symmetric: "Very satisfied / Satisfied / Neither satisfied nor dissatisfied / Dissatisfied / Very dissatisfied." Heuristic: count positive vs negative options — must match in number and roughly intensity; flag asymmetric anchor strength.
- **C2. Missing midpoint when neutrality is real** | Scale design | Forces neutral respondents to pick a side → random noise; inflates extremity. BAD: "Satisfied / Somewhat satisfied / Somewhat dissatisfied / Dissatisfied" (no neutral) for an attitude many are genuinely neutral on. FIXED: add "Neither satisfied nor dissatisfied" (+ "No opinion / Not applicable" separately if applicable). Heuristic: is neutral a meaningful real-world stance for this item? If yes and there's no midpoint, flag.
- **C3. Unwanted midpoint / forced fence-sitting** | Scale design | When you need a direction (a go/no-go, a recommendation), a midpoint becomes a satisficing dump. BAD: 5-pt agree scale with "Neither agree nor disagree" for "I would recommend this to a colleague" when the business needs lean. FIXED: even-numbered scale to force a lean, or keep odd deliberately. (Judgment call — flag only when the doc's stated purpose requires direction.) Heuristic: if the survey's goal is directional and the scale has a soft midpoint, raise as a tradeoff to confirm.
- **C4. Overlapping / non-mutually-exclusive ranges** | Options | Respondents at a boundary don't know which to pick; results not cleanly bucketable. BAD: "0–10 / 10–20 / 20–30 / 30+" (10 and 20 appear twice). FIXED: "0–9 / 10–19 / 20–29 / 30 or more." Heuristic: scan numeric bands for shared endpoints; verify every value maps to exactly one bucket.
- **C5. Non-exhaustive options / missing "Other", "None", "N/A", "Don't know"** | Options | Respondents whose true answer isn't listed pick the "closest" wrong option or abandon → fabricated/biased distribution. BAD: "Which browser do you use? Chrome / Safari / Firefox" (no Edge, no "Other", no "I don't know"). FIXED: add "Other (please specify) ___" and, where relevant, "None of these" / "Not applicable" / "Don't know" as distinct options. Heuristic: could a real respondent's honest answer be absent? Is there an escape hatch? Are "None"/"N/A"/"Don't know" conflated when they mean different things? Flag.
- **C6. Question/answer-scale mismatch** | Structure | Stem asks one thing, options measure another; respondents re-map (inconsistently) or answer the wrong question. BAD: stem "How often do you use feature X?" options "Strongly agree / Agree / Disagree / Strongly disagree." FIXED: match: frequency stem → frequency options ("Daily / A few times a week / A few times a month / Rarely / Never"). Heuristic: does the option set's dimension (frequency/agreement/satisfaction/likelihood/quantity) match the stem's verb? If not, flag.
- **C7. Vague quantifiers in options (or stem)** | Clarity/options | "Often"/"sometimes"/"regularly"/"occasionally" mean wildly different frequencies → uncomparable data. BAD: "How often do you use it? Always / Often / Sometimes / Rarely / Never." FIXED: anchor to time: "Every day / 3–6 days a week / 1–2 days a week / Less than weekly / Never" (calibrate buckets to the behavior's cadence). Heuristic: flag adverbial frequency labels with no numeric anchor; "a few"/"several"/"many" as response options.
- **C8. Absolutes forcing false choice** | Framing/options | "Always/never/all/every/ever" + Yes/No leaves no room for the common "usually" reality; few can honestly say yes → under-detects the behavior. BAD: "Do you always read the terms before agreeing? (Yes / No)" FIXED: "How often do you read the terms before agreeing? (Always / Most of the time / About half the time / Occasionally / Never)." Heuristic: flag universal quantifiers ("always", "never", "all", "every", "everyone", "none", "ever") in the stem, esp. paired with a binary answer set.

**D. Format & cognitive-load bias**

- **D1. Acquiescence bias (agree/disagree format)** | Format bias | People — esp. less-educated/less-engaged/fatigued (Pew) — tend to agree regardless of content; "agree" is the low-effort default. BAD: "Agree or disagree: 'The new layout is easy to navigate.'" FIXED: rewrite item-specific/forced-choice: "How easy or difficult is the new layout to navigate? (Very easy … Very difficult)" — or two competing statements, ask which is closer. Heuristic: flag any "[statement] — agree/disagree" battery; recommend conversion to construct-specific scales or forced choice between alternatives.
- **D2. Social-desirability triggers** | Response bias | Over-report "good" behavior (voting, exercise, donations, recycling); under-report "bad" (drinking, drug use, prejudice, skipping work); the stem's framing signals what's "approved". BAD: "How often do you exercise to stay healthy?" / "Do you always recycle?" FIXED: normalize the full range + lower stakes: "Some people exercise regularly, others rarely or never. In the past 7 days, on how many days did you exercise for at least 30 minutes?"; for sensitive topics prefer self-administered mode + acknowledge barriers ("Did things come up that kept you from voting, or did you happen to vote?"). Heuristic: flag stems that moralize ("to stay healthy", "to do your part"); sensitive behaviors asked bluntly with no face-saving framing or "haven't/didn't" option weighted into the scale.
- **D3. Order / priming effects baked into wording (or sequence)** | Context | An earlier item primes a frame that distorts a later one (assimilation: consistency pressure; contrast: differentiation). Within an item, an example/preamble primes a category. BAD: "How worried are you about data breaches?" right before "How do you feel about our new data-sharing feature?" / a preamble listing only downsides before the question. FIXED: separate sensitive/priming items; general before specific when measuring overall sentiment; randomize where order shouldn't matter; keep preambles neutral + minimal. Heuristic: look at adjacent items — does one establish a frame (worry, value, comparison) the next inherits? Does an item's lead-in nudge toward certain answers? Flag + recommend reorder/randomize.
- **D4. Option-order effects (primacy/recency)** | Format bias | Self-administered → primacy (top over-chosen); phone → recency. Fixed order systematically biases which option wins. BAD: a long brand/feature list always shown in the same order, untreated. FIXED: randomize (or reverse for half the sample) for non-ordinal lists; keep ordinal scales in logical order but consider reversing for half. Heuristic: flag long non-ordinal lists with no randomization noted; ordinal scales presented in only one direction in long instruments.
- **D5. Matrix/grid overload & straight-lining** | Format bias | Big grids invite picking one column down the page; missing data + break-offs rise with visual complexity; straight-lining is largely a design failure. BAD: a 15-row × 7-column agree/disagree matrix on one screen. FIXED: break into ≤5–7-row blocks; vary item polarity to defeat auto-pilot; one-item-per-screen on mobile; attention checks sparingly. Heuristic: flag matrices with >~7 rows or >~5 columns; all-positively-worded rows; multiple large grids in sequence.
- **D6. "Select all that apply" pitfalls vs forced-choice yes/no** | Format | CATA produces lower endorsement than asking yes/no per item — respondents satisfice + stop scanning; non-selection is ambiguous ("no" vs "didn't read it"). BAD: "Which of these features have you used? [check all that apply]" when accurate prevalence matters. FIXED: for each feature "Have you used [feature]? Yes / No" — forces consideration of every item + disambiguates blanks. Reserve CATA for low-stakes quick-scan contexts. Heuristic: flag CATA used where you need per-item prevalence; recommend forced-choice grid.
- **D7. Ranking too many items** | Format/capability | People reliably rank only a handful; beyond ~5–7 they guess at the tail → noisy lower ranks. BAD: "Rank these 12 features from most to least important." FIXED: pick top 3 (or MaxDiff / a "most important" + "least important" pick), or rank a shortlist of ≤5. Heuristic: flag rank-order tasks with >~5–7 items; recommend top-N or MaxDiff.

**E. Structure, length & logic**

- **E1. Compound stems / long preambles / buried instructions** | Cognitive load | Long lead-ins + mid-sentence instructions get skimmed; load amplifies satisficing + acquiescence; respondents answer a question they didn't fully parse. BAD: "Given that our company has recently undergone a major reorganization and launched several new initiatives across multiple regions, and bearing in mind your own role, please rate, on a scale where 1 is low and 5 is high, but only if you've been here over a year, how supported you feel." FIXED: screen for tenure first; then "How supported do you feel in your role? (1 = not at all supported … 5 = fully supported)."; put any instruction on its own line before the question. Heuristic: flag stems >~20–25 words, subordinate clauses stacking conditions, instructions buried after the question or mid-sentence.
- **E2. Biased or missing reference period** | Clarity/structure | No timeframe → each respondent picks their own → uncomparable. A loaded period ("during the holidays, when service is slowest") biases recall. BAD: "How satisfied have you been with our support?" (no period) / "How was support during our recent outage?" (cherry-picked bad window). FIXED: "Thinking about your experience with our support in the past 3 months, how satisfied are you?" — neutral, explicit, fixed window. Heuristic: flag any retrospective/frequency question with no reference period; reference periods chosen to coincide with unusually good/bad conditions.
- **E3. Forced answers / no opt-out** | Ethics/data quality | Required questions with no "Prefer not to say"/"Don't know"/"N/A" → respondents fabricate or abandon; biases the sample toward the compliant + toward those with an opinion. BAD: mandatory income question; mandatory "How satisfied with feature X?" with no "haven't used it." FIXED: make sensitive items optional, or always include "Prefer not to say"/"Don't know"/"Not applicable" as selectable options; reserve "required" for genuinely essential, low-sensitivity items. Heuristic: flag required questions on sensitive topics (income, health, demographics, politics) lacking an opt-out; attitude/usage questions marked required with no "don't know / haven't used" option.
- **E4. Branching mistakes / no skip logic** | Structure/logic | Respondents get irrelevant questions (e.g. rate a product they said they don't own) → frustration, break-offs, meaningless data; or relevant follow-ups skipped for everyone. BAD: "Do you own a car? No." → next: "How satisfied are you with your car's fuel economy?" FIXED: add skip logic — non-owners bypass car-specific items; owners get follow-ups; pipe prior answers ("the [brand] you mentioned…"). Heuristic: trace each path — does a "No"/"Never"/"N/A" answer ever lead to a question presupposing the opposite? Are filter questions wired to downstream items? Flag any orphaned follow-up.
- **E5. Pseudo-anonymity claims / unnecessary PII** | Ethics/trust | Claiming "anonymous" while collecting email/IP/employee ID, or asking for PII not needed for the analysis, suppresses honest answers + erodes trust → social-desirability spikes, response rate drops, data-protection liability. BAD: "This survey is completely anonymous." [then] "Please enter your work email so we can follow up." / asking full DOB when an age band would do. FIXED: either truly anonymize (collect no identifiers; say "confidential" not "anonymous" if you can re-identify) or be transparent ("Your responses are confidential; your manager will not see individual answers"); collect only the PII the analysis requires; use bands (age range, not DOB). Heuristic: flag "anonymous" if any identifier (email, name, ID, IP, identifying free-text) is also collected; PII fields not used by any stated analysis goal; exact DOB / full address / SSN-like fields unless clearly justified.
- **E6. Non-attitude harvesting / no "Don't know" on knowledge or opinion items** | Capability/options | On topics respondents know nothing about, omitting "Don't know / No opinion" manufactures opinion data out of noise. BAD: "Do you approve or disapprove of the Bricker Amendment?" with only Approve/Disapprove. FIXED: include "Don't know / No opinion"; consider a prior filter ("Have you heard of…?"). Heuristic: for opinion/knowledge questions on niche/technical topics, is "Don't know / No opinion" offered? If not, flag.

## STREAM 3 — Survey platform export/import

### Summary
- Recommend the skill support 3 platforms: Typeform, SurveyMonkey, Google Forms. Cleanest "generate-an-artifact → import" stories. Qualtrics (QSF) is a strong honorable mention but enterprise-only → stretch goal. Microsoft Forms is weakest (no real programmatic import) → deprioritize.
- Typeform → emit `typeform.json` (Create API request body). Full form definitions POST as JSON to `https://api.typeform.com/forms`. Best "import a designed survey" path of any platform — rich field types, logic jumps, screens, validations in one JSON. Needs a personal access token (`forms:write`). UI import exists but is AI-text-based + lossy.
- SurveyMonkey → emit `surveymonkey.json` (POST /v3/surveys body with nested `pages[].questions[]`). One request creates the whole survey. Needs OAuth token with "Create/modify surveys" scope; **API access requires a paid (Team/Enterprise) plan**. No-code UI import limited to copy/paste of plain text (multiple choice + single textbox only) → also emit a "paste-ready" plain-text block as a fallback.
- Google Forms → emit a Google Apps Script `.gs` file (FormApp builder script). No native bulk-import UI. Realistic path: generate a self-contained Apps Script calling `FormApp.create()` + `addMultipleChoiceItem()`/`addCheckboxItem()`/`addScaleItem()`/`addParagraphTextItem()`/`addPageBreakItem()`/`addSectionHeaderItem()` etc. User pastes into script.google.com, runs once. (Alternative: Forms REST API `forms.create` + `forms.batchUpdate` with `createItem` — but needs OAuth setup, so Apps Script is lower-friction.) Third-party add-ons exist but unreliable.
- Qualtrics (stretch) → emit a `.qsf` file. QSF is plain JSON; importable via UI ("Import survey") or `POST /survey-definitions?format=qsf` with `X-API-TOKEN`. Schema well-documented but verbose (SurveyEntry + SurveyElements array of Questions/Blocks/Flow/SurveyOptions). Enterprise-only.
- Microsoft Forms → no recommended artifact. Only import path = "Quick Import" of a Word/PDF document, only recognizes titles, multiple-choice, open-text. No Excel import. Graph `forms` API undocumented/unsupported for creation. If you must support it, emit a structured `.docx` + warn it's lossy.
- Jotform / Tally / Fillout (optional): Jotform has a public REST API (`POST /form` with `questions` map) — viable artifact if demand exists. Tally + Fillout have no public form-creation API; skip.

### Per-platform detail

**A. Typeform — BEST import target.**
Import mechanisms ranked: (1) Create API `POST https://api.typeform.com/forms` — accepts a complete form definition as JSON; returns 201 + `Location` header. Recommended. (2) UI import from Google Forms / "Typeform AI" — paste text into Typeform AI; lossy (section headers → Statement blocks, images dropped). No CSV import. (3) Import from another Typeform = just "Duplicate" in UI.

Artifact = `typeform.json`. Example:
```json
{
  "title": "Customer Feedback Survey",
  "type": "branching",
  "settings": { "language": "en", "progress_bar": "proportion", "is_public": true },
  "welcome_screens": [ { "ref": "welcome", "title": "We'd love your feedback", "properties": { "show_button": true, "button_text": "Start" } } ],
  "fields": [
    { "ref": "satisfaction", "title": "How satisfied are you with our product?", "type": "opinion_scale", "properties": { "steps": 5, "start_at_one": true, "labels": { "left": "Not at all", "right": "Extremely" } }, "validations": { "required": true } },
    { "ref": "features_used", "title": "Which features do you use?", "type": "multiple_choice", "properties": { "allow_multiple_selection": true, "allow_other_choice": true, "choices": [ { "ref": "c_dash", "label": "Dashboard" }, { "ref": "c_reports", "label": "Reports" }, { "ref": "c_api", "label": "API" } ] }, "validations": { "required": false } },

    { "ref": "improve", "title": "What should we improve?", "type": "long_text", "validations": { "required": false } },
    { "ref": "recommend", "title": "Would you recommend us?", "type": "yes_no", "validations": { "required": true } }
  ],
  "logic": [ { "type": "field", "ref": "recommend", "actions": [ { "action": "jump", "details": { "to": { "type": "field", "value": "improve" } }, "condition": { "op": "is", "vars": [ { "type": "field", "value": "recommend" }, { "type": "constant", "value": false } ] } } ] } ],
  "thankyou_screens": [ { "ref": "thanks", "title": "Thanks — your feedback helps!", "properties": { "show_button": false } } ]

```
Field types & key per-type properties: `short_text`/`long_text` (`properties.description`, `validations.max_length`); `multiple_choice` (`properties.choices[]` each `{ref,label}`, `allow_multiple_selection`, `allow_other_choice`, `randomize`, `vertical_alignment`; `validations.min_selection`/`max_selection`); `picture_choice` (choices carry `attachment`); `dropdown` (`choices[]`, `alphabetical_order`); `opinion_scale` (`steps` 5–11, `start_at_one`, `labels.{left,center,right}`); `rating` (`steps` 1–10/11, `shape` star/heart/up/cat/circle); `yes_no`; `ranking` (`choices[]`); `number`, `email`, `phone_number`, `website`, `date` (`structure`, `separator`), `file_upload`, `legal`, `statement`, `payment`, `matrix`, `contact_info`, `group` (nested fields), `nps`. Every field: `ref` (stable id, used for logic targets), `title`, `type`, `properties`, `validations.required`, optional `attachment`/`layout`. Screens: `welcome_screens[]` + `thankyou_screens[]` each with `ref`, `title`, `properties` (`show_button`, `button_text`, `button_mode`, `redirect_url`). `logic[]`: `{type:"field", ref, actions:[{action:"jump"|"add"|"set", details, condition:{op, vars}}]}`; also `hidden[]` (URL params) + `variables` (`score`, `price`).
Auth/limits: Bearer token (personal access token from Settings → Personal tokens, or OAuth 2.0); needs `forms:write`. Create API works on free plans but premium-only *settings* (remove branding, custom thank-you redirect, some logic richness) are ignored/rejected on free. Images must be pre-uploaded via `POST /images`. Rate ~2 req/sec. README snippet: `curl -X POST https://api.typeform.com/forms -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" -d @typeform.json`.

Import mechanisms ranked: (1) `POST https://api.surveymonkey.com/v3/surveys` with nested `pages[].questions[]` — creates the entire survey in one call. Best programmatic path. (2) `POST /v3/surveys` with `from_template_id`/`from_survey_id` — clone, then PATCH. (3) UI "Import Questions → Paste your content" — paste plain text; only Multiple Choice + Single Textbox parsed. Useful low-tech fallback.

**C. Google Forms.**
Import mechanisms ranked: (1) Google Apps Script (`FormApp`) — generate self-contained `.gs`; user → script.google.com → New project → paste → Run. Most reliable, no OAuth app setup, full type control. Recommended artifact. (2) Google Forms REST API (`forms.create` then `forms.batchUpdate` with `createItem` requests, `https://forms.googleapis.com/v1/forms`) — same power, needs OAuth client + token. (3) Third-party add-ons ("Form Builder for Sheets", "Quiz Importer", "Form Creator") read a Google Sheet — unreliable across versions, limited type mapping. Mention only. (4) No native "import questions" UI feature (you can only import from another Google Form).

Artifact = `build-google-form.gs`. Example:
```javascript
function buildSurvey() {
  var form = FormApp.create('Customer Feedback Survey');
  form.setDescription('We appreciate your feedback.');
  form.setProgressBar(true);
  // form.setIsQuiz(true); // uncomment for quiz mode
  form.addSectionHeaderItem().setTitle('About your experience');
  form.addMultipleChoiceItem().setTitle('How satisfied are you overall?').setChoiceValues(['Very satisfied','Satisfied','Neutral','Dissatisfied','Very dissatisfied']).setRequired(true);
  var cb = form.addCheckboxItem();
  cb.setTitle('Which features do you use?').setChoiceValues(['Dashboard','Reports','API']).showOtherOption(true).setRequired(false);
  form.addListItem().setTitle('Your role').setChoiceValues(['IC','Manager','Director','Exec']);
  form.addScaleItem().setTitle('How likely are you to recommend us?').setBounds(1,5).setLabels('Not at all likely','Extremely likely').setRequired(true);
  form.addPageBreakItem().setTitle('Open feedback');
  form.addParagraphTextItem().setTitle('What should we improve?').setRequired(false);
  form.addTextItem().setTitle('Email (optional)');
  form.addGridItem().setTitle('Rate the following').setRows(['Ease of use','Support']).setColumns(['Poor','Fair','Good','Excellent']);
  Logger.log('Form URL: ' + form.getEditUrl());
  Logger.log('Published URL: ' + form.getPublishedUrl());
}
```
Key FormApp surface: `FormApp.create(title)` → Form; `form.setTitle/setDescription/setProgressBar/setCollectEmail/setIsQuiz/setAllowResponseEdits/setConfirmationMessage`. Item adders: `addMultipleChoiceItem()` (radio), `addCheckboxItem()` (multi-select), `addListItem()` (dropdown), `addTextItem()` (short answer), `addParagraphTextItem()` (long answer), `addScaleItem()` (linear scale), `addRatingItem()` (star/heart rating), `addGridItem()` (radio matrix), `addCheckboxGridItem()` (checkbox matrix), `addDateItem()`, `addDateTimeItem()`, `addTimeItem()`, `addDurationItem()`, `addSectionHeaderItem()`, `addPageBreakItem()`, `addImageItem()`, `addVideoItem()`. Common item methods: `setTitle()`, `setHelpText()`, `setRequired(bool)`. Choice items: `setChoiceValues([...])` or `setChoices([item.createChoice(...)])`, `showOtherOption(bool)`; navigation jumps via `createChoice(value, FormApp.PageNavigationType...)` / `setGoToPage()`. Scale: `setBounds(min,max)` (min 0 or 1; max ≤10), `setLabels(lower,upper)`. Grid: `setRows([...])`, `setColumns([...])`. Quiz extras: `setPoints()`, `setFeedbackForCorrect()`, `setFeedbackForIncorrect()`, `setCorrectAnswers()`/`createFeedback()`.
Auth/limits: Apps Script runs as the signed-in user; first run prompts for Drive/Forms authorization; no API keys. REST path needs OAuth 2.0 with `https://www.googleapis.com/auth/forms.body`. Apps Script daily quotas apply (form creation cheap). Coverage gaps vs other platforms: no native NPS/ranking types (map ranking → grid; NPS → scale 0–10), limited conditional logic (only "go to section based on answer" on multiple-choice/dropdown). Optionally also offer `forms-api.json` (batchUpdate payload) for REST users.

**D. Qualtrics (stretch, enterprise only).**
Import mechanisms ranked: (1) UI: Survey → Tools → Import/Export → Import Survey → upload `.qsf` (or advanced-format TXT). Primary intended path. (2) API: `POST /API/v3/survey-definitions?format=qsf` with the QSF JSON body, `X-API-TOKEN` header. Also `POST /API/v3/survey-definitions` (leaner non-QSF) then `POST .../questions`, `POST .../blocks`, `PUT .../flow`, `PUT .../options`. (3) Copy survey within Qualtrics.

Artifact = `survey.qsf` (plain JSON; abbreviated):
```json
{
  "SurveyEntry": { "SurveyID": "SV_PLACEHOLDER", "SurveyName": "Customer Feedback Survey", "SurveyStatus": "Inactive", "SurveyLanguage": "EN" },
  "SurveyElements": [
    { "Element": "BL", "PrimaryAttribute": "Survey Blocks", "Payload": [ { "Type": "Default", "Description": "Default Block", "ID": "BL_1", "BlockElements": [ { "Type": "Question", "QuestionID": "QID1" }, { "Type": "Question", "QuestionID": "QID2" } ] } ] },
    { "Element": "FL", "PrimaryAttribute": "Survey Flow", "Payload": { "Flow": [ { "Type": "Block", "ID": "BL_1", "FlowID": "FL_1" } ], "Properties": { "Count": 1 } } },
    { "Element": "SO", "PrimaryAttribute": "Survey Options", "Payload": { "BackButton": "true", "ProgressBarDisplay": "Text" } },
    { "Element": "SQ", "PrimaryAttribute": "QID1", "Payload": { "QuestionText": "How satisfied are you overall?", "QuestionID": "QID1", "QuestionType": "MC", "Selector": "SAVR", "SubSelector": "TX", "Choices": { "1": {"Display":"Very satisfied"}, "2":{"Display":"Satisfied"}, "3":{"Display":"Neutral"}, "4":{"Display":"Dissatisfied"}, "5":{"Display":"Very dissatisfied"} }, "ChoiceOrder": ["1","2","3","4","5"], "Validation": { "Settings": { "ForceResponse": "ON" } } } },
    { "Element": "SQ", "PrimaryAttribute": "QID2", "Payload": { "QuestionText": "What should we improve?", "QuestionID": "QID2", "QuestionType": "TE", "Selector": "ESTB" } }
  ]
}
```
QSF essentials: `SurveyEntry` object + `SurveyElements` array; entries have `Element` code (`SQ`=survey question, `BL`=blocks, `FL`=flow, `SO`=survey options, `RS`=response set, `PL`=playerOptions, `QC`=quota, `SCO`=scoring, etc.). Question types: `MC` (Selector `SAVR`/`MAVR`/`DL`), `TE` (text entry; Selector `SL`/`ML`/`ESTB`/`FORM`), `Matrix` (Selector `Likert`/`TE`/`Bipolar`, `Answers` rows + `Choices` columns), `Slider`, `RO` (rank order), `NPS`, `DB` (descriptive text), `CS` (constant sum), `SBS` (side-by-side). The API "definitions" export is leaner + omits display/skip/flow logic the QSF includes — generate QSF if you want logic to survive.
Auth/limits: `X-API-TOKEN` header (user's API token from Account Settings → Qualtrics IDs); only if the org's license enables API. Datacenter-specific base URL. Enterprise/institutional — no consumer free tier. Recommend QSF + manual UI import for the skill rather than API upload.

**E. Microsoft Forms — not recommended.** Only import path = "Quick Import" of a Word (.docx) or PDF (Forms → New → Quick Import); OCRs/parses the document; only recognizes titles/subtitles, multiple-choice, open-text — no rating/ranking/Likert/date. No Excel import (workaround: paste Excel → Word → Quick Import). No supported public API for creating forms (form-creation Graph endpoints undocumented/unsupported). If you must support it: emit a `.docx` (one question per block: question line, then choices) + warn it's lossy. Otherwise skip.

**F. Jotform / Tally / Fillout (optional).** Jotform — documented REST API: `POST https://api.jotform.com/form` with `questions` (map keyed by order index, each `{type:"control_radio"|"control_checkbox"|"control_textbox"|"control_textarea"|"control_dropdown"|"control_scale"|"control_matrix"|..., text, options, required}`) + `properties` + `emails`; auth via API key (`?apiKey=` or `APIKEY` header); free plan allows API with monthly limits → `jotform.json` artifact viable if demand. Tally — no public form-creation API (read-only submissions/webhooks); no import-from-file; skip. Fillout — API can read forms/submissions but cannot create forms; skip.

### Flagged gaps (Stream 3)
- SurveyMonkey paid-plan gate is the biggest practical limitation — API/JSON artifact works only on Team/Enterprise plans with API enabled; the skill should detect/warn; the plain-text "paste" fallback loses all types except MC + single-textbox.
- Typeform Create API `logic[]` schema is the most complex/under-documented piece — many operators (`is`, `is_not`, `greater_than`, `contains`, `begins_with`, `on`...) varying by field type; generate logic conservatively + validate against the live API. Premium-only settings silently fail / 4xx on free.
- Google Forms real type-coverage gaps vs a generic survey design: no native NPS, no ranking, limited conditional logic — needs a documented mapping (ranking→grid, NPS→0-10 scale, matrix→grid).
- Apps Script vs REST API for Google Forms — Apps Script recommended for low friction; if the skill is part of a hosted product with OAuth, REST `batchUpdate` is cleaner. Decide by runtime context.
- Typeform "import from Google Forms" is shifting toward "Typeform AI" (LLM text paste, non-deterministic) — don't build the artifact around it; use the Create API.
- Qualtrics datacenter-specific base URLs + per-org API enablement — a generated `.qsf` is portable but API upload needs org-specific config. Recommend QSF + manual UI import.
- MS Forms / Graph — no supported documented form-creation endpoint; community reverse-engineered `forms.office.com` internal APIs are unsupported + break. Treat MS Forms creation as not-programmatically-doable.
- Jotform API question schema (`control_*` types, `options` as pipe-delimited strings in some types) is quirky + version-sensitive — verify against live docs before building.
- Not every official doc page fetched in full — field-type lists + method names corroborated across multiple sources but validate exact property names against the live API reference + a test request before shipping each exporter.

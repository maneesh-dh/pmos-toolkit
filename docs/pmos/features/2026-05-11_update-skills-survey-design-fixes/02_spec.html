survey-design hardening — Spec

tier: 3 · type: enhancement · feature: update-skills-survey-design-fixes · date: 2026-05-11 · status: Draft · requirements: docs/pmos/features/2026-05-11_update-skills-survey-design-fixes/01_requirements.html

Adapted Tier-3 template: this is a skill-prose change (no compiled code, no API/DB/frontend). §9 = reviewer subagent return contract; §10 = survey.json schema v2; §11 = rendered-artifact changes. Architecture/sequence diagrams (§6) are textual — the "system" is the phase flow inside one SKILL.md.

1. Problem Statement

The first production run of /survey-design shipped a fieldable survey but the Phase-4 reviewer caught only methodology nits — an experienced PM had to manually surface six structural problems (predictable answers, non-load-bearing questions, scope-mismatched framing, redundancy). Plus: the generated intro doesn't motivate respondents; the skill silently degrades on missing substrate assets; cosmetic findings crowd the dispositions batch; and there's no question type for "metrics by cadence" items. Primary success metric: a re-run on the retro's original brief has the Phase-4 loop flag-and-handle the predictable / non-load-bearing / scope-mismatched questions on its own (vs. 0 of 6 caught today).

D8Score dimension weights: product-fit 30, structure/funnel 15, length-vs-budget 10, mode-fit 10, scale-balance 10, accessibility 10, ethics/PII 10, intro/consent 5 (= 100). Per-dimension score = 100 × (1 − weighted-deduction); weighted-deduction tuned in reference/survey-best-practices.md (blocker = large deduction, should-fix = medium, nit = small). The composite is informational only.(a) equal weights; (b) product-fit-heavy weighting(b) — product-fit is the headline gap this change exists to close, so it carries the largest single weight; the rest are roughly even because they were already adequately handled. Since the score doesn't gate (D7), exact calibration is low-stakes — these are defaults the implementer may tune.
D9The loop's regeneration step is targeted: the generator regenerates only the flagged questions (plus directly-coupled ones — an nps follow-up, a skip_logic target), keeping the rest of survey.json byte-stable except where edits cascade (id renames, skip-logic retargets); it re-runs §3.5 trim-to-budget afterward.(a) full Phase-3.4 re-run each iteration; (b) targeted regeneration of flagged questions(b) — a full re-run would churn questions the reviewer was happy with and make the changelog unreadable; targeted keeps diffs minimal and the changelog precise.
D10Question provenance is tracked in transient skill state (not persisted in the final survey.json): a question is author-origin (the author supplied or insisted on it) or generated. Inside the loop, author-origin questions are rewrite-only — never auto-killed; if one still FAILs product-fit after 2 iterations it's surfaced as a kill candidate in Phase 5. generated questions may be auto-killed by the loop (removed from survey.json), recorded in the changelog so the author can revert in Phase 5.(a) loop may kill anything; (b) loop never kills, only rewrites; (c) loop may kill generated questions but never author-origin ones(c) — protects coverage the author explicitly wanted while still letting the loop drop a genuinely dead generated question; the changelog + Phase-5 review keep it reversible.
D11The refinement changelog is a ## Refinement loop changelog section appended to question-eval.md — one ### Iteration N subsection with a table | Q id | action (rewrote / killed / added) | reviewer reason | score Δ |. Not a new file.(a) new refinement-log.md; (b) section in question-eval.md(b) — consistent with D1 (the eval artifacts are the single place to look); the changelog is meaningless without the per-question verdicts it sits next to.
D12Loop iteration produces an invalid survey.json → re-derive once; if still schema-invalid, abandon that iteration's regeneration, keep the last-valid survey.json, note iteration N: regeneration produced invalid survey.json (<invariant>), reverted in the changelog, exit the loop early to Phase 5.(a) hard-abort the skill; (b) revert the bad iteration, keep last-valid, exit loop early(b) — the author should still get the best valid survey produced so far; aborting throws away good work over a regeneration bug.
D13Reviewer returns malformed output mid-loop (missing a product-fit verdict, or no score) → re-dispatch once naming the gaps (same as today's per-question-count-mismatch handling); if still malformed, skip the auto-regenerate for the run and proceed to Phase 5 with whatever verdicts/findings did come through; non-interactive → log and proceed.(a) hard-abort; (b) re-dispatch once then degrade-gracefully(b) — mirrors the skill's existing reviewer-failure posture (keep the committed draft, proceed with a warning).
D14multi_field_open time cost = 30s per field (each field counted like an open_short); plus the question's own intro/read overhead is zero (the stem read is amortized into the first field).(a) 30s/field; (b) a smaller per-field value (shared context)(a) — keeps the constant consistent with open_short and errs toward over-estimating burden, which is the safer direction for budget trimming.
D15multi_field_open export: native on platforms that have a multi-textbox-style question type (SurveyMonkey's multiple_textbox family if confirmed at implementation time); Typeform and Google Forms have no equivalent → downgrade to N open_short items preceded by a statement/section-header that names the original grouped question. Every downgrade noted in export/README.md and commented in the artifact.(a) downgrade everywhere; (b) native where available, downgrade elsewhere(b) — preserve the grouped UX where the platform allows it; the implementer verifies SurveyMonkey support during /execute and downgrades if it doesn't pan out.

6. System Design — the Phase-4 loop

6.1 Loop flow (textual)

Phase 3  generate survey.json  (per reference/survey-best-practices.md; no product-fit self-grade)
   │
   ▼
Phase 4  iteration i := 1
   │   reviewer subagent (fresh context) evaluates survey.json →
   │     - composite 0–100 score + per-dimension sub-scores
   │     - per-dimension findings, each {target, severity ∈ {blocker,should-fix,nit}, defect, message, proposed_fix}
   │     - per-question product-fit verdict {predictable: PASS|FAIL + predicted-answer line, load_bearing: PASS|FAIL, scope_match: PASS|FAIL}
   │     - recommended edits (rewritten stems / "kill q-id" / added opt-outs)
   │   write survey-eval.md (leads with score + research-goal-coverage section)
   │   write question-eval.md (each ## q-id leads with the product-fit verdict + prediction)
   │
   ├── exit condition met?  (no product-fit FAIL on any question  AND  no blocker-severity methodology finding)
   │      yes ─────────────────────────────► go to Phase 5
   │      no, and i < 2:
   │          generator regenerates the flagged questions (targeted, D9), respecting D10 (author-origin = rewrite-only),
   │          re-runs §3.5 trim-to-budget, appends "### Iteration i" to question-eval.md's changelog
   │          → re-derive survey.html / preview.html / etc.  (if invalid: D12)
   │          i := i + 1 ; loop back to "reviewer subagent ..."
   │      no, and i == 2:  (cap hit)
   │          go to Phase 5 with the residual FAILs as top-of-batch kill/rewrite candidates
   │
   ▼
Phase 5  open with the changelog summary → dispositions batch (product-fit FAILs first, then blocker→should-fix→nit; cosmetic-only in a trailing "Noted, not asked:" line)

6.2 Reviewer dispatch contract

One reviewer subagent per iteration (Task tool, or sequential inline when subagents are unavailable — same prompt, same return contract, just no fresh-context isolation; the existing Platform-Adaptation note already covers this). The prompt MUST contain: the survey context object {purpose, audience, response_impact, time_budget_min, mode, max_questions, estimated_minutes}; the full survey.json as text; the contents (or paths, if the subagent can read files) of reference/survey-best-practices.md + reference/question-antipatterns.md; and (iteration ≥ 2) the prior iteration's question-eval.md so the reviewer can see what changed. In non-interactive runs the prompt's first line is [mode: non-interactive]. The reviewer does NOT write any file and does NOT mutate survey.json — it returns the two markdown bodies (survey-eval.md, question-eval.md) plus a machine-readable findings/verdicts/recommended-edits block the parent parses to drive the loop.

7. Functional Requirements

7.1 F1 — Phase-4 refinement loop & product-fit rubric

IDRequirement

9. Reviewer subagent return contract (the "API" of this feature)

The reviewer returns three things to the parent (as markdown bodies + one fenced block the parent parses):

## Research-goal coverage / product fit
<does the question set answer `purpose`? coverage gaps? redundancy? scope drift? — findings table, same shape as below>
## Structure & order   (sub-score X/15)   …findings table…
## Length vs budget    (sub-score X/10)   …
## Mode fit            (sub-score X/10)   …
## Scale balance       (sub-score X/10)   …
## Accessibility       (sub-score X/10)   …
## Ethics / PII        (sub-score X/10)   …
## Intro / consent     (sub-score X/5)    …  (flags missing WIIFM sentence as should-fix)
<overall verdict line>

# 2. question-eval.md body  — one section per question
## q-recent-moment
**Product fit:** predictable: FAIL — predicted: ~75% → "the analytics team"; load-bearing: PASS; scope-match: PASS  →  kill/rewrite candidate
Verdict: 2 finding(s)
| Severity | Defect | Message | Proposed fix |
| blocker | product-fit:predictable | … | rewrite to ask <X> / or cut |
| nit | A3 | … | … |
…(one section per question)…
## Refinement loop changelog        ← present, written/extended by the GENERATOR each iteration, not the reviewer
### Iteration 1
| Q id | action | reviewer reason | score Δ |
| q-recent-moment | rewrote | predictable answers | +6 |

# 3. machine block the parent parses to drive the loop
```yaml
score: 87
sub_scores: { product_fit: 24, structure: 13, length: 9, mode: 9, scale: 8, accessibility: 9, ethics: 9, intro: 5 }
product_fit_fails: [ q-recent-moment ]                # questions with any product-fit FAIL
blockers: [ q-recent-moment ]                          # questions/sections with a blocker-severity methodology finding
recommended_edits:
  - target: q-recent-moment
    action: rewrite          # rewrite | kill | add_opt_out | …
    detail: "Ask: 'In the last 30 days, which of these did you do when you hit a wall?' with options […]"
```

Exit condition (parent-side): product_fit_fails == [] AND blockers == [] → done. Else if iteration < 2 → generator applies recommended_edits (honoring D10), re-derive, append changelog, re-dispatch. Else → Phase 5 with product_fit_fails as top-of-batch kill/rewrite questions.

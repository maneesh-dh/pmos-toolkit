Grill Report — 01_requirements (survey-design hardening)

Depth: standard · Questions asked: 10 · Target: docs/pmos/features/2026-05-11_update-skills-survey-design-fixes/01_requirements.html

Resolved — decisions pinned this session

#BranchDecision
1F1 — Phase 4 architecturePhase 4 becomes a scored generate↔review refinement loop. Phase 3 generates → Phase 4 reviewer evaluates only (returns a numeric score + per-dimension findings + recommended edits; never mutates survey.json) → if the exit condition isn't met and iterations < 2, control returns to the generator, which regenerates the offending questions incorporating the reviewer's notes → re-review → repeat → on exit, Phase 5 surfaces the residual findings to the author for disposition as today. Single writer = the generator; the reviewer never writes questions.
2F1 — scored rubricThe reviewer produces a weighted score out of 100 across all eval dimensions (product-fit, structure/funnel, length-vs-budget, mode fit, scale balance, accessibility, ethics/PII, intro/consent). The existing blocker/should-fix/nit severities still tag individual findings (used for Phase-5 ordering and to prioritize what the loop fixes first). Exact weights ⇒ /spec.
3F1 — loop exit conditionThe loop exits on categorical conditions, not the score: when (a) zero product-fit FAILs AND (b) zero blocker-severity methodology findings remain — OR after 2 refinement iterations. The 0–100 score is computed and shown (progress signal: "72 → 91", and the loop uses point-weight to pick which findings to fix first) but never gates termination — a single LLM-judge score near a hard cutoff is too flappy to gate on. (Supersedes an earlier "≥90/100 gate" answer.)
4F1 — predictability checkApplies to both question kinds. Closed-ended: the reviewer writes a one-line predicted answer distribution; FAIL iff 1–2 options would plausibly absorb ~80%+. Open-ended: a yes/no judgment on whether multiple distinct high-signal answers are realistically expected; FAIL iff the open question would also collapse to 1–2 predictable themes. The written prediction is the evidence and goes in the product-fit block.
5F1 — product-fit FAIL ordering in Phase 5A product-fit FAIL is the top tier in the dispositions batch: one "Kill or rewrite Q<id>? — <rubric reason>" question per FAIL (options Kill / Rewrite-as-proposed / Keep-with-reason / Defer), batched before blocker → should-fix → nit methodology findings.
6F3 — fail-loud, but tieredAbort with the reinstall message only on hard-required assets (survey-preview.js, style.css, viewer.js — load-bearing for the artifacts the skill promises). Warn-and-skip-the-extra on convenience assets (serve.js, html-to-md.js + turndown bundle): missing those ⇒ no preview server / no .md sidecar, run continues. Keeps F3 consistent with the skill's warn-and-continue posture.
7F6 — keep the first-class typemulti_field_open stays in scope as a real schema type (schema_version → 2), driving estimated_minutes, the preview render, and the per-platform export mapping (native where supported; N open_short items + a section-header explanation where not). Not downgraded to "just document the pattern."
8F8 — get the impact from the authorAdd a Phase-2 intake variable: what happens to the responses / what does the audience get. The generated intro.text motivation line is built from that answer (honest by construction). Fallback when the author doesn't supply it ⇒ /spec (likely: restate purpose in respondent-benefit terms; no fabricated impact claim).
9Scope / tierAll 5 changes (F1, F3, F6, F7, F8) ship as one Tier-3 batch, as previously decided — same skill, mostly one file; /spec and /plan sequence F1 and F6 as the big tasks.

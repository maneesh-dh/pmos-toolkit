Problem

Engineers driving multi-stage agentic dev cycles (the same audience that lives in /feature-sdlc) inherit, scaffold, and review codebases whose architectural quality sits in the gap between syntactic lint (ESLint / Prettier / Ruff) and post-impl behavioural verification (/verify). There is no consistent, opinionated, book-grounded architecture check that fires at the moments where architectural drift compounds: scaffolding a new repo, reviewing a /spec, inheriting a legacy codebase, or merging agent-generated code.

The OpenAI “Harness Engineering” (2025) thesis is the structural argument: opinionated, uniform repo structure is the dominant lever for coding-agent productivity — agents do dramatically better in repos with one test runner, one formatter, one deps file, folder-by-feature, explicit layer boundaries. Without a skill that operationalizes that thesis, every repo accretes drift, and every agent run pays the cost.

Who experiences this?

The pmos-toolkit primary persona: a senior engineer / PM-engineer driving agentic development through /feature-sdlc, reviewing agent-produced code, and frequently inheriting or starting repos. They have read Clean Architecture, Pragmatic Programmer, DDD, and Tidy First? at some point — but cannot hold all the principles in working memory on demand during a 30-minute review window.

Why now?

Three converging signals:

  1. The harness-engineering thesis is fresh (May 2025) and not yet operationalized as tooling — the window to ship an opinionated reference implementation is open.
  2. /feature-sdlc is in production at v2.46.0 and Phase 6 (/execute) generates substantial new code per run. The architectural-drift surface is real and measurable.
  3. Matt Pocock’s improve-codebase-architecture exists as prior art but is single-mode, TS-only, and ruleset-in-prompt — leaving room for a data-driven, multi-tier, citation-rich, multi-stack successor.

Goals & Non-Goals

Goals

  • Consistent, book-grounded architectural feedback on existing repos — measured by: ≥1 substantive, user-agreed-with violation per /architecture audit run on a real repo within the first 30 days.
  • Citations make every flag a micro-lesson — measured by: a non-author rating ≥4 of 5 sample violation outputs as “educational, not preachy”.
  • Principles file accretes by user accretion, not edict — measured by: ≥1 user-authored L3 override appearing in a real repo’s principles.yaml within 30 days of first audit.
  • Wraps, does not reimplement, declarative AST tooling — measured by: ≥70% of v1 declarative checks delegate to Dependency Cruiser / ts-arch / ruff (gap-map artifact in 02_spec).
  • Conforms to pmos skill conventions — measured by: passes feature-sdlc/reference/skill-eval.md (39-check binary rubric) with zero residuals at Phase 6a.

Non-Goals (explicit scope cuts)

  • NOT shipping /architecture scaffold in v1 — because the scaffold flavor packs need their own design pass (TS variants, Python variants, Node vs Bun, etc.) and v1 vertical-slice principle says ship the highest-standalone-value mode first.
  • NOT shipping /architecture spec-check in v1 — because it requires careful integration with /feature-sdlc Phase 2 and a separate FR pass; defer to v2.
  • NOT shipping /architecture code-check in v1 — because the Phase 6.5 wiring needs the audit-mode prompt + ruleset to be battle-tested first; defer to v3.
  • NOT auto-wiring any /architecture mode as a PostToolUse / SessionStart hook — because hooks are too easy to disable in noise; user-invoked only in v1+.
  • NOT reimplementing AST-based dependency / layer / cycle detection — because Dependency Cruiser (TS) and equivalent Python tooling already exist; delegate.
  • NOT supporting Go / Rust / Java in v1 — because TS + Python covers ≥90% of pmos-toolkit user repos; v2+ as demand emerges.
  • NOT designing a CI-integration story in v1 — because invocation-time use is the unblocked path; CI is a v2 concern.

User Experience Analysis

Motivation

  • Job to be done: “When I’m reviewing this codebase (mine, inherited, or agent-produced) I want a consistent, opinionated, book-grounded architecture check that flags structural violations with rationale, so my codebase stays maintainable and agent-friendly without me re-deriving every principle from memory.”
  • Importance / urgency: Medium-high. Architectural drift is silent and cumulative; it isn’t urgent on day one but compounds painfully by month six. Today the user uses ad-hoc memory of books + occasional ESLint configs; neither operationalizes the harness-engineering thesis.
  • Alternatives: (a) Run Matt Pocock’s improve-codebase-architecture directly (TS-only, audit-only, prompt-embedded ruleset, no citations, no ADR loop); (b) read the books and freelance the review (high cognitive load, inconsistent); (c) run Dependency Cruiser / ts-arch / ArchUnit standalone (deterministic but no semantic checks, no citations); (d) do nothing and let /verify catch what it catches (general-purpose, not architecture-focused).

Satisfaction Signals

  • User reads the cited rationale and recognizes it (“ah yes, Clean Architecture ch. 11”) — feels educated, not nagged.
  • User accepts a draft ADR within the same session — the loop closed productively.
  • User authors a project-level L3 override within the first week — the principles file is something they own, not endure.
  • The next /verify run on the same repo cites fewer architecture-debt findings than the pre-/architecture audit baseline.

Solution Direction

Composite of ideation Finalists #1 + #2 + #3 + #8 + #10 + #11. Surface = single skill with sub-modes (F2). Substance = principles-as-data (F2) + tiered rulebook (F3) + harness-engineering scaffold opinion (F8) + ADR-driven persistence (F10) + citation-on-every-violation (F11).

v1 ships only the audit mode. The skill skeleton, principles substrate, and ADR loop are designed forward-compatible with spec-check, code-check, scaffold in v2-v4 — but only audit is presented as a user surface in v1.

User-observable behaviour

$ /architecture audit [path]
  → reads principles.yaml (L1 ship + L2 stack-detected + L3 project-overrides)
  → walks repo (or path if given), detects stack(s)
  → delegates declarative checks to Dependency Cruiser (TS) / ruff (py) / ts-arch
  → runs semantic checks via LLM against L1+L2 rules
  → emits prioritized violation report:
       [block]  L1-04 Dependency Inversion (Clean Architecture, ch. 11)
                src/services/email.ts:14 — domain layer depends on infra layer
                → draft ADR queued at docs/architecture/decisions/0007-...
       [warn]   L2-TS-03 No barrel files (improve-codebase-architecture)
                src/utils/index.ts — barrel file re-exports detected
       [info]   L1-09 Tidy First (Beck, 2024) — refactor candidates listed
  → user accepts / rejects / edits each draft ADR
  → accepted ADRs become L3 project overrides in principles.yaml

Principles data model (high-level)

principles.yaml ships as a sibling reference file in the skill (v1 ships L1 + a starter L2 per stack). A repo’s own .pmos/principles.yaml (if present) supplies L3 overrides — same schema, project-scoped.

schema_version: 1
rules:
  - id: L1-04
    tier: universal           # universal | stack | project
    statement: "Dependencies point inward toward higher-level policy."
    rationale: "Lower-level details (infra, frameworks) should depend on..."
    source:
      book: "Clean Architecture"
      author: "Robert C. Martin"
      chapter: 11
      url: null
    severity: block            # info | warn | block
    applies_to: [ts, python, all]
    applies_in: [audit]        # which modes the rule fires in

ADR format (decision D-ADR1)

Nygard-style ADRs (lightweight): Title / Status / Context / Decision / Consequences. Written to docs/architecture/decisions/NNNN-<slug>.md per Nygard convention. NNNN is the next available 4-digit sequence in that directory.

Tier precedence (decision D-PREC1)

When two rules with the same id appear at different tiers, project (L3) wins over stack (L2) wins over universal (L1). Precedence is printed in every violation output (“resolved tier: L3 — see .pmos/principles.yaml line 42”).

Delegation strategy (decision D-DELEG1)

Declarative architectural checks (forbidden-dependency, layer-violation, circular-import) delegate to existing AST tooling:

  • TS: Dependency Cruiser (cycles, forbidden deps), ts-arch (layer violations).
  • Python: ruff (deps), pylint --disable-all --enable=cyclic-import, optional import-linter for explicit contracts.

If a delegated tool is missing, the audit reports the gap as info-severity (“install Dependency Cruiser to enable L1-04 deterministic check”) — never blocks the run.

User Journeys

Primary Journey (Happy Path)

  1. User runs /architecture audit in a TS or Python repo (no arg → audits cwd).
  2. Skill loads its shipped principles.yaml (L1 + L2-ts + L2-python), merges in .pmos/principles.yaml (L3) if present.
  3. Skill detects the stack(s) (via repo-shape-detection: package.json → ts, pyproject.toml/requirements.txt → python).
  4. Skill invokes delegated AST tools — captures their JSON output, maps each violation to the relevant L1/L2 rule by id.
  5. Skill runs semantic checks via LLM against L1+L2 rules not covered by AST tools.
  6. Skill emits the violation report grouped by severity (block first), each row showing [severity] rule-id rule-title (citation) — file:line — rationale.
  7. For each block-severity violation, the skill drafts an ADR under docs/architecture/decisions/ (max 5 per run) and lists them for user disposition.
  8. User accepts / rejects / edits each draft ADR. Accepted ADRs are written to disk; their Decision sections become L3 overrides appended to .pmos/principles.yaml with a back-link to the ADR file.
  9. Skill commits the new ADRs + L3 changes as a single commit: arch: accept N ADRs from /architecture audit run YYYY-MM-DD.

Alternate Journeys

  • Scoped audit: /architecture audit src/services/ — same flow, scoped to one path; only rules that fire within scope are reported.
  • Stack auto-detection ambiguity: repo has both package.json AND pyproject.toml — skill prompts user (AskUserQuestion) to confirm which stack(s) to audit, defaulting to both with their union of L2 rules.
  • Monorepo with per-area stack markers (e.g., the book-companion validation target — backend/pyproject.toml + frontend/package.json): skill walks the repo, detects each area’s stack from the nearest enclosing marker, runs delegated tools scoped to that area, and emits a per-area section in the report (“backend/ (python): N violations · frontend/ (ts+vue): M violations”).
  • Vue SFCs: .vue files are treated as TS for L1 semantic checks; declarative TS-tooling delegation (Dependency Cruiser) is run on .ts/.tsx only with a per-rule info note flagging the SFC blind spot (v2 candidate: a Vue-SFC dependency extractor).
  • No L3 file yet: first audit run creates .pmos/principles.yaml with an empty rules: [] + the schema_version, so accepted ADRs have somewhere to land.
  • Re-run after acceptance: the next /architecture audit reads the new L3 overrides — previously-block violations now resolved appear in the report as “previously gated by ADR-NNNN, now passes”.

Error Journeys

  • No delegated tool installed: skill reports a per-rule info-severity gap (“install Dependency Cruiser to enable L1-04 deterministic check; falling back to semantic check only”). Run continues; semantic-check verdict is marked “degraded”.
  • Repo has no package.json or pyproject.toml: skill emits “no supported stack detected — v1 supports TS / Python; see reference/modes-roadmap.md for v2+ stacks” and exits clean (exit 0).
  • Delegated tool errors: skill captures stderr, surfaces the rule as info-severity with the tool error inline, continues with semantic check.
  • ADR directory has un-numbered ADRs: skill warns and uses next-numeric-after-max-found; never overwrites.
  • .pmos/principles.yaml has a schema mismatch: skill aborts with a clear “run /architecture migrate to upgrade” message (the migrate sub-command is v2; v1 lists the manual edits in the error).

Design Decisions

#DecisionOptions ConsideredRationale
D1 v1 ships only audit mode; spec-check, code-check, scaffold are forward-compatible roadmap items. (a) All four modes v1, (b) audit + code-check v1, (c) audit only v1 (chosen). Vertical-slice principle: ship the highest-standalone-value mode first; battle-test the principles substrate + ADR loop before wiring deeper integrations. Ideation Next Steps #6.
D2 Skill name = /architecture (directory plugins/pmos-toolkit/skills/architecture/). (a) /arch (ideation default — short), (b) /architecture (chosen — clear), (c) /principles, (d) /house-style. User decision: clarity over brevity; /arch reads as jargon, /architecture reads as intent.
D3 v1 stack scope = TS + Python from day one. (a) TS-only v1 (ideation default), (b) TS + Python v1 (chosen), (c) L1-only v1. User decision: pmos-toolkit user repos are roughly evenly split TS / Python; shipping one without the other front-loads churn. Doubled authoring load is acceptable given the L1 universal rules generalize.
D4 ADR format = Nygard (lightweight): Title / Status / Context / Decision / Consequences. (a) Nygard (chosen), (b) MADR, (c) ship both as templates. Lower friction; matches majority of OSS ADR adoption; user can override per-repo by replacing reference/adr-template.md in v2.
D5 Three-tier rulebook with fixed precedence: project (L3) > stack (L2) > universal (L1). (a) Flat ruleset, (b) two-tier (universal + project), (c) three-tier (chosen). Universal rules generalize but per-stack idioms (no barrels in TS, prefer dataclasses in Python) must override cleanly. Project overrides close the loop with ADRs.
D6 Hard cap on L1 universal at 15 rules. (a) No cap, (b) cap at 10, (c) cap at 15 (chosen), (d) cap at 25. Hedges the F1 premortem H-risk (principles.yaml bloats to 2,000 lines). 15 is enough to cover SOLID + DDD basics + Tidy First + harness-eng essentials; pressure to fit forces rule quality.
D7 Delegate declarative checks to existing AST tools (Dependency Cruiser, ts-arch, ruff); do not reimplement. (a) Reimplement, (b) wrap (chosen), (c) skip declarative entirely. Reimplementing collapses into rebuilding F3 (premortem: 3 H-risk failure modes). Wrapping leverages mature tooling; v1’s job is the citation-rich semantic layer + ADR loop, not yet-another-architecture-linter.
D8 Only block-severity violations spawn draft ADRs; warn stays inline; info is muted unless --verbose. (a) Every violation spawns ADR, (b) only block (chosen), (c) user opt-in per rule. Hedges F1 premortem H-risk (ADR-draft fatigue). 5-ADR-per-run cap is the secondary backstop.
D9 Citation is mandatory on every rule (source: { book, chapter | url }). (a) Optional, (b) mandatory (chosen). Cross-finalist riskiest assumption: users prefer cited violations over terse rule IDs. Making citation a schema invariant forces the “micro-lesson” UX even as the rulebook grows.
D10 v1 invocation is user-explicit; no auto-wiring as PostToolUse / SessionStart hook. (a) Hook on day one, (b) opt-in hook v1, (c) manual-only v1 (chosen), (d) /feature-sdlc Phase 6.5 wiring v1. Hooks default-on are noise risk (F2 premortem M-risk); user choice = invocation-time only v1; /feature-sdlc integration is a v2 design item.
D11 Skill conforms to feature-sdlc/reference/skill-patterns.md §A–§F as the standing acceptance criteria; Phase 6a /skill-eval gates merge. (a) Custom criteria, (b) standard pmos criteria (chosen). Every pmos skill conforms; the eval rubric is the binary forcing function. Citations and unique skill content do not exempt the structural rules.

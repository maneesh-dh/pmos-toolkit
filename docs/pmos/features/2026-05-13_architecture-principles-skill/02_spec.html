1. Problem Statement

LLM coding agents — and the humans collaborating with them — accrete architectural debt one PR at a time because no agent-readable checklist of project-wide principles is available at code-review or design time. The /architecture skill ships a tiered, citation-rich rulebook (≤15 L1 universal rules · stack-specific L2 rules · per-repo L3 overrides), runs an audit pass that delegates declarative checks to existing AST tools (Dependency Cruiser, ruff) and grep/regex for semantic checks, and turns block-severity findings into Nygard ADRs (cap 5/run) so architectural decisions become first-class versioned artifacts. v1 ships only the audit mode — scaffold / spec-check / code-check are explicitly v2+.

Primary success metric: a single /architecture audit <repo> run against a representative TS+Python monorepo (e.g., book-companion: 690 .py · 408 .ts · 260 .vue) returns within the runtime budget, produces a JSON report citing rules by ID, writes ≤5 ADRs for block-severity findings, and surfaces declarative-rule coverage % honestly.

3. Non-Goals

  • scaffold / spec-check / code-check modes — explicitly deferred to v2+. v1 ships only audit. The four-mode shape in the ideation artifact is the v2+ horizon.
  • Pre-commit / CI hooks — v1 is manual-invoke only (D10). Adding a Git hook or CI wrapper is a v2 concern.
  • ts-arch tool — dropped from v1 (Dep Cruiser's required + forbidden rules cover the same circular-import + layer-boundary use cases). May re-enter at v2 if Dep Cruiser gaps emerge.
  • Vue SFC <script> extraction — v1 ships with .vue files getting only L1-semantic checks. SFC extractor deferred to v1.1+; the report's frontend_declarative_coverage metric makes the gap visible.
  • Auto-fix — audit reports only; never edits user source. (ADR write is the sole side-effect, and even that is opt-out via --no-adr.)
  • Network calls at audit time — runs offline. Citation URLs in principles.yaml are documentation, not fetched.
  • Custom rule authoring UX — v1 users hand-edit YAML. A /architecture add-rule sub-command is a v2 concern.
  • Cross-language graph (TS importing Python or vice-versa) — out of scope; tools are per-stack.
  • Severity-tier customization — info/warn/block tier mapping is fixed in v1; user can demote a rule via L3 override but cannot redefine the tier semantics.

5. User Personas & Journeys

Personas + journeys are fully specified in requirements §user-journeys. The spec carries forward without restatement; key flow recapped here for sequence-diagram traceability.

5.1 Primary journey — architect running first audit on a TS+Python monorepo

  1. User runs /architecture audit /Users/me/repo (or no-arg → cwd).
  2. Skill loads plugin L1+L2 rules from its skill dir; tries to load <repo>/.pmos/architecture/principles.yaml (L3 + exemptions); on missing, proceeds with L1+L2 only and notes l3_present: false in report.
  3. Skill enumerates source files via git ls-files minus .gitignore matches minus hardcoded deny-list.
  4. For each rule: grep/regex (L1 semantic) or shell-out to Dep Cruiser/ruff (L2 declarative); collect findings keyed by rule_id.
  5. Reconcile findings against L3 overrides + exemptions: tier-precedence + per-site mute.
  6. Promote ≤5 block-severity findings to ADR files at <repo>/docs/adr/ADR-NNNN-*.md; truncate rest with explicit count.
  7. Emit JSON report to stdout + human summary to stderr.

7.3 Rule loading, precedence, and L1 cap (F1)

IDRequirement
FR-20Audit loads rules in order: plugin L1 → plugin L2 (filtered by detected stack) → project L3. Precedence on collision (same rule.id): project (L3) > stack (L2) > universal (L1). Lower-tier rule fields override higher-tier; missing fields inherit.
FR-21L1 cap: at most 15 rules at tier=1. On load, if plugin principles.yaml has >15 entries with tier: 1, abort schema-load: write to stderr exactly ERROR: L1 has <N> rules; cap is 15. Demote rules to L2 or remove. Exit 64. (F1 / D12.)
FR-22Detect stack from the scanned repo: presence of package.json + tsconfig.json → ts; pyproject.toml OR requirements*.txt OR setup.py → python. A repo may carry both stacks; audit runs both rule sets independently. No stack detected → run L1 only; report stacks_detected: [].
FR-23Project L3 file missing → proceed with L1+L2 only; report l3_present: false. Project L3 file malformed YAML → stderr ERROR: <repo>/.pmos/architecture/principles.yaml malformed: <parser-error>, exit 64.
FR-24Every rule loaded into the merged set MUST have non-empty source: (D9). Plugin-shipped rules failing this fail tools/check-citations.sh at release time; user-authored L3 rules failing this → warn at audit time, do not abort.

FR-63ADR cap: at most 5 ADRs written per audit run. Block-severity findings beyond the cap are listed in the report's adrs_truncated array with their rule_id + file + line. Audit prints to stderr note: 7 additional block findings not promoted to ADR (cap 5/run). See report.adrs_truncated.
FR-64Severity-to-action: info → log to report only · warn → log to report + stderr summary · block → log to report + stderr summary + ADR file (subject to cap). Only block spawns ADRs (D8 / D22).
FR-65Exemption reconciliation. At end of finding-collection, before report emission: for each exemptions: row in project principles.yaml, locate the corresponding ADR file at <adr_path>/ADR-<exemption.adr-suffix>-*.md. Three outcomes:
  • matching pair (exemption row + ADR file with ## Suppresses block listing same {rule, file, line?}): suppress matching finding silently from report.
  • orphan exemption (exemption row, no matching ADR file): suppression still applied (user-explicit intent), BUT report carries warn-level entry orphan exemption: <rule_id> <file>[:<line>] (no ADR file at <path>).
  • informational ADR (ADR file with ## Suppresses block, no matching exemption row): findings NOT suppressed (ADR alone is documentation; the project principles.yaml row is what actually mutes); report carries info-level note: ADR-0009 documents suppression but no exemption row in principles.yaml — finding still surfaced.
FR-66Exemption auto-expiry: an exemption row with expires: <ISO-8601> in the past is treated as not-present (finding surfaces). Audit stderr summary includes note: <N> exemption(s) expired; suppression lifted. Re-affirm via ADR or remove the row.
FR-67Audit supports --no-adr flag: collect findings, emit report, do NOT write any ADR files. Useful for dry-run / CI inspection. report.adrs_written = [].

9. Skill invocation contracts (replaces §9 API)

This is a CLI skill; no HTTP API. The invocation contracts are: (a) the slash command, (b) the delegated-tool shell-out invocations, (c) the file-write contracts for ADRs.

9.1 Slash command surface

/architecture audit [path] [--no-adr] [--non-interactive]

Args:
  path         Repository root to audit. Defaults to cwd.
Options:
  --no-adr             Collect findings + emit report; do not write ADR files.
  --non-interactive    Suppress chat prompts; degrade per _shared/non-interactive.md contract.

Exit codes:
  0    Audit completed (findings may exist).
  1    Internal error (parser crash, unexpected exception).
  64   Schema/config error (L1 cap exceeded, malformed principles.yaml, unknown selector).

9.2 Delegated-tool shell-out contracts

# Dependency Cruiser (TS L2)
npx --no-install depcruise \
    --output-type json \
    --config <skill-dir>/tools/.depcruise.cjs \
    <scan-root>

# ruff (Python L2)
ruff check \
    --output-format=json \
    --select=TID252,F401,F403,F405,B006 \
    <scan-root>

Both commands run with: cwd=<scan-root>, env preserved, timeout 60s, stderr captured. On ENOENT (command not found) → skip + warn per FR-32. On non-zero exit with stderr → record in tools_errored, exit 0.

9.3 ADR write contract

Write path: <scan-root>/<adr_path>/ADR-NNNN-<kebab-title>.md. Atomic write (temp-then-rename). Never overwrite existing ADR files — collision (same NNNN) → numeric increment until free slot. Parent directory created if missing. File mode 0644.

11. CLI output format (replaces §11 Frontend)

No web frontend. CLI output split across stdout (machine-readable JSON, FR-71) and stderr (human-readable summary, FR-72). Designer-role concerns reduce to two:

  • Stderr summary tone: direct, lowercase, no emoji, ≤ 30 lines. Severity counts up front. Block findings listed verbatim; warn/info summarised by count + sample. Closing notes (exemption / SFC coverage / ADR cap) appear last.
  • Color & ANSI: stderr respects NO_COLOR env. When color is on, severity tiers are tinted (block: red bold, warn: yellow, info: dim). Stdout JSON is never tinted.

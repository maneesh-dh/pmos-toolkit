/artifact — HTML output parity — Requirements

Problem

The /artifact skill produces structured PM/eng artifacts (PRD, Experiment Design, Engineering Design, Discovery). Its SKILL.md describes HTML output via the _shared/html-authoring/ substrate (added during the v2.33.0 html-artifacts migration), but the contract has eight concrete gaps that prevent its outputs from matching the look-and-feel of other pipeline skills (/requirements, /spec, /plan, /wireframes, /prototype). The result: an /artifact run can emit a valid {slug}.html file, but the reviewer subagent receives raw HTML it was never prompted to handle, the refine flow defaults the output extension to .refined.md regardless of primary format, post-edit sections.json drifts from the on-disk HTML, and the four templates lack explicit MD→HTML authoring guidance — so two different runs of /artifact prd can produce structurally divergent HTML even with the same input.

Who experiences this?

Product managers, engineering managers, and discovery practitioners using /artifact inside a pmos-toolkit pipeline (host-repo developer surface). They expect the artifact viewer (the style.css + viewer.js toolbar, the section-link button, the inlined index, the GFM-backed MD sidecar) to behave identically across every pipeline stage. Drift across stages forces them to mentally translate between styles — or, worse, leaves stale sections.json companions that break cross-doc anchor resolution from 02_spec.html#fr-....

Why now?

The v2.33.0 html-artifacts migration converted ten pipeline skills + /feature-sdlc to HTML-primary with the substrate. /artifact was migrated partially in the same release (SKILL.md describes the HTML write contract) but the surrounding skill plumbing — reviewer subagent dispatch, refine flow, update flow, frontmatter shape, sections.json idempotence — was not. The v2.40.0 polish-skill release exposed similar HTML-fidelity gaps for /polish and resolved them; /artifact is the last pipeline-adjacent skill carrying drift. Closing it before any downstream consumer (a future /artifact → /spec handshake, or a Notion/Confluence export) ships avoids cascading rework.

Goals & Non-Goals

Goals

  • End-to-end HTML parity: An /artifact prd full run produces an HTML artifact visually indistinguishable from a sibling 02_spec.html — same toolbar, same fonts, same section-link affordance, same sidebar — measured by: zero new CSS classes outside the substrate; zero new asset files; diff of the rendered <header> + <footer> regions byte-identical to /spec's.
  • Reviewer-input-contract parity (FR-50/51/52): Phase 3's reviewer subagent receives chrome-stripped HTML body + the eval rubric, returns findings with verbatim quote + section_id matching the live {slug}.sections.json; orchestrator hard-fails on any miss. Measured by: a fresh /artifact prd run on a synthetic draft with a planted bad-quote finding produces a hard-fail with the canonical error message format.
  • Refine + update flows format-aware: /artifact refine <path> defaults the safer output to <path>.refined.html when primary is HTML (not .refined.md); /artifact update appends to <section id="comment-resolution-log"> with an HTML table, not a markdown one. Measured by: refine on a prd.html writes prd.refined.html; update emits a valid <section> wrapper with a <table>.
  • Post-edit fidelity: After Phase 3 edits the HTML draft and after Phase U.3 update appends rows, the skill re-emits {slug}.sections.json and (when output_format=both) the MD sidecar. Measured by: jq on the regenerated sections.json shows ids in the live HTML; diff of the MD sidecar before/after edit reflects the changes.
  • Explicit MD-template → HTML authoring contract: Phase 2.7 carries a paragraph spelling out that the LLM authors the HTML body directly using template.md for section ordering and per-section guidance (matching how /spec and /plan author HTML from outline), with no MD-to-HTML conversion step at write time. Measured by: a new contributor reading Phase 2.7 can describe the rendering path in one sentence.
  • Frontmatter JSON shape: Phase 2.7's frontmatter example shows the JSON shape inside <script type="application/json" id="pmos-frontmatter"> (not a YAML triple-dash block). Measured by: example block in SKILL.md parses as JSON via jq.
  • assert_heading_ids hook at write time: Phase 2.7 runs plugins/pmos-toolkit/skills/_shared/html-authoring/assets/ heading-id validator (or inline equivalent) before rename(2). Measured by: a planted <h2> without id hard-fails the write with the canonical error.
  • Substrate-only assets: No new files added to _shared/html-authoring/assets/. Re-use what's already there (style.css, viewer.js, html-to-md.js, chrome-strip.js). Measured by: git diff --stat of the merge shows zero new files under that path.

Non-Goals

  • NOT changing the template store at ~/.pmos/artifacts/templates/<slug>/template.md — templates stay MD-shaped (runbook edge case row 4 carve-out). Because the templates are the human-editable authoring surface; HTML rendering happens at write time, not in storage.
  • NOT introducing a new md-to-html.js renderer in the substrate. Because LLM-direct HTML authoring matches all other pipeline skills and avoids a new dependency surface.
  • NOT rewriting reviewer-prompt.md from scratch. Because most of its contract (severity rubric, JSON return shape, eval-item-by-item discipline) is sound — only the input-shape and quote-validation contract change.
  • NOT migrating the four built-in templates' eval.md files to HTML. Eval criteria stay MD — they are reviewer input, not render output.
  • NOT introducing a separate "look-and-feel preset" knob. Because the substrate is the look-and-feel; the four content presets (Concise / Tabular / Narrative / Executive) stay orthogonal.
  • NOT changing template management flows (template add|list|remove) or preset management flows. Because they operate on the ~/.pmos/artifacts/ store, not the feature-folder render path.

User Experience Analysis

Motivation

  • Job to be done: "I'm running my pipeline; /artifact produced a PRD; I want to share its URL with my team in the same way I share 02_spec.html or 01_requirements.html — same viewer, same anchors, same look."
  • Importance / urgency: Mid-cycle. Users have noticed the visual drift but worked around it (opening the MD sidecar instead, or copy-pasting into Notion). Each instance of "I'll just open the .md" is a slow erosion of the HTML-primary contract.
  • Alternatives: Continue using the MD sidecar (loses cross-doc anchors); open the .html and tolerate the visual drift (cognitive friction); fork a private renderer (fragments the toolchain).

Solution Direction

Eight targeted edits to plugins/pmos-toolkit/skills/artifact/, plus a focused update to reviewer-prompt.md and SKILL.md text. No new substrate assets; no template-store changes; no new reference/ dir on /artifact (it ships with presets/ and templates/ only — no need to add). Concretely:

  1. SKILL.md Phase 2.7 — add an explicit paragraph: "Author the HTML body directly using template.md for section ordering and per-section guidance comments. Wrap each ## §N as <section id="..."><h2 id="..."> per _shared/html-authoring/conventions.md §3. No MD→HTML conversion happens at write time."
  2. SKILL.md Phase 2.7 — replace the YAML-shape frontmatter example with the JSON shape that actually lands in <script type="application/json" id="pmos-frontmatter">.
  3. SKILL.md Phase 2.7 — add the heading-id assertion step before the atomic rename: node assets/assert_heading_ids.js {slug}.html.tmp (or an equivalent inline grep), hard-fail with canonical error.
  4. SKILL.md Phase 3 step 1 — insert a chrome-strip preamble: "Run node {feature_folder}/assets/chrome-strip.js {slug}.html > /tmp/artifact-reviewer-input.html; pass the stripped HTML inline to the reviewer prompt."
  5. SKILL.md Phase 3 step 2 — add reviewer-return validation (FR-52 parity): (a) section ids returned must subset of {slug}.sections.json ids; (b) every finding's quote (new required field) must be a ≥40-char verbatim substring of the source HTML; (c) hard-fail with canonical error message on either miss.
  6. SKILL.md Phase 3 step 3 and Phase U.3 — at end of edit-applying step, re-emit {slug}.sections.json from the post-edit HTML (parse <h2>/<h3> ids); when output_format=both, re-run the html-to-md.js sidecar pipe.
  7. SKILL.md Phase 3.x refine flow (line 336) — replace "Overwrite <path> or write to <path>.refined.md?" with a primary-format-aware version: read the file extension; offer .refined.html when primary is HTML, .refined.md only when primary is MD-only legacy.
  8. SKILL.md Phase U.4 (update flow Comment Resolution Log) — replace the markdown table with the HTML emission contract: append/extend a <section id="comment-resolution-log"> block with an HTML <table>; the same row shape; same per-row dates and resolutions.
  9. reviewer-prompt.md — (a) change "you receive ... a markdown document" to "you receive a chrome-stripped HTML <h1>+<main> body"; (b) add the quote field as required in the JSON output shape, with the contract "quote must be a ≥40-char verbatim substring of the source"; (c) require section ids match the live sections.json.

Two alternatives considered and rejected:

  • Add a md-to-html.js renderer to the substrate. Rejected: introduces a vendored dependency, doesn't match how /spec + /plan author HTML, and the LLM-direct path is already proven across 10 pipeline skills.
  • Convert the template store to HTML-shape too. Rejected: templates are the human-editable authoring surface; HTML in storage hurts diffability and pull-request reviews; the runbook §edge-case-row-4 carve-out exists precisely for this.

User Journeys

Primary Journey — "Generate a PRD that looks like every other pipeline doc"

  1. User in a feature folder runs /artifact prd full --preset narrative.
  2. Phase 0 resolves output_format: html (source: settings).
  3. Phase 2.1–2.6: template + tier + preset + gathered context resolve (unchanged).
  4. Phase 2.7: LLM authors prd.html body directly per the new contract paragraph; sections.json built from the in-memory tree; assets copied (idempotent); heading-id assertion passes; atomic rename. Index regen pulls in prd.html.
  5. Phase 3: chrome-strip prd.html → pass to reviewer; reviewer returns findings with quote + section; orchestrator validates; Edit applies high+medium; sections.json + MD sidecar re-emitted at loop end.
  6. Phase 4: user runs npx serve docs/pmos/features/<feature>/, opens prd.html — visually identical to the sibling 02_spec.html.

Alternate — "Refine an existing PRD"

  1. User runs /artifact refine docs/pmos/features/X/prd.html.
  2. Skill reads the primary extension: .html.
  3. AskUserQuestion offers: "Overwrite prd.html or write to prd.refined.html?" — default .refined.html.
  4. Phase 3 runs against the chosen file; post-edit re-emission.

Error journey — "Reviewer returns a hallucinated quote"

  1. Phase 3 reviewer returns a finding with quote: "lorem ipsum dolor sit amet consectetur".
  2. Orchestrator substring-greps quote against source HTML — miss.
  3. Hard-fail with canonical error: [/artifact] reviewer quote not found in <file>: lorem ipsum dolor sit amet conse...
  4. Soft-phase failure dialog: Retry / Pause / Abort.

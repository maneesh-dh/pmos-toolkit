Problem

pmos-toolkit users have fuzzy ideas arriving at all hours — feature concepts, product bets, internal-tool sketches, "what if we…" hunches. Today, those ideas either get lost in a chat scrollback, or get pushed straight into /requirements (which is too heavy and assumes the idea is already worth shaping into a feature) or /grill (which assumes a committed plan exists). There is no lightweight, repeatable skill for the pre-requirements step: structured ideation that generates idea variants, pressure-tests the strongest candidate, and saves a single per-idea artifact the user can revisit, share, or promote into the pipeline.

Who experiences this?

The pmos-toolkit user (a product/eng practitioner) in two contexts: (a) capturing a half-formed thought before it evaporates; (b) deciding whether a vague idea is worth the cost of /requirements. Cross-repo by intent — the user may be in any project when an idea arrives.

Why now?

Adjacent skills (/brainstorming in superpowers, /creativity in pmos-toolkit, /grill) cover overlapping-but-distinct surfaces, and the gap is becoming visible as users invoke /requirements for ideas that should have died (or been sharpened) earlier. A standalone ideation skill gives the user a cheap front door and clarifies the seam between "have an idea" and "have a feature worth speccing".

Goals & Non-Goals

Goals

  • Turn a fuzzy idea into a structured one-page artifact in ~10–15 minutes — measured by: a single end-to-end run produces a complete {docs_path}/ideation/{YYYY-MM-DD}_<idea-slug>.html with all 12 sections populated (or marked Open), no skill-internal retries.
  • Generate breadth before depth — measured by: the Expand phase produces ≥8 distinct idea variants before any convergence step; the user (not the skill) picks the 1–3 finalists.
  • Make pressure-testing the default, not an opt-in — measured by: every artifact contains a non-empty Premortem table and a ranked Riskiest-Assumptions list; running the skill without pressure-test is a deliberate --no-stress-test escape, not the happy path.
  • Stay simple and repeatable — measured by: the SKILL.md uses a 3-phase loop (Frame → Expand → Pressure-test) with one optional Refine pass; no internal sub-frameworks, no required references the user must read, no ceremonial "hats" / "roles".
  • Hand off cleanly — measured by: the artifact's Next Steps section names the candidate downstream skill (/requirements, /grill, /backlog add) by exact slash-command name, so a follow-up invocation is one click.

Non-Goals

  • NOT a workshop-shaped framework — no Six Thinking Hats, no Disney Method, no 6-3-5 Brainwriting, no Design Sprint week. Those assume multiple humans, multiple sessions, or facilitator setup. Because: the user explicitly asked for "simple and repeatable, not a complex framework-driven thing".
  • NOT a replacement for /requirements — the artifact is intentionally pre-requirements (no acceptance criteria, no scope decomposition, no per-tier templates). Because: that's the seam the pipeline already covers, and conflating the two muddies both.
  • NOT a replacement for /grill — the pressure-test phase is a batch structured battery (premortem + inversion + assumption-map) emitted in one pass, not a turn-by-turn decision-tree interview. Because: /grill targets committed plans; /ideation targets uncommitted ideas. Different input, different cadence, different output.
  • NOT folding /creativity in as a sub-phase — /creativity stays standalone (Tier-3-requirements enhancer per its own description). Because: tight coupling would erase its standalone surface, and the Expand phase has its own auto-picked techniques.
  • NOT loading workstream context — like /diagram, /polish, /design-crit, this skill is a standalone utility. Because: ideation is cross-repo by intent and workstream pollution would bias the variant generation.
  • NOT auto-promoting to backlog / requirements — the handoff is explicit (the artifact suggests next steps; the user invokes them). Because: auto-promotion floods /backlog with half-baked ideas; manual promotion preserves the bar.

Solution Direction

A single Claude Code skill at plugins/pmos-toolkit/skills/ideation/SKILL.md, invoked as /ideation <free-form idea text>. The skill runs a 3-phase loop with one optional Refine pass:

  /ideation "what if we tracked decisions like commits?"
       │
       ▼
  ┌─────────────────────────────────────────────────────────┐
  │ Phase 1 — FRAME                                         │
  │   • HMW statement                                       │
  │   • Target user + job-to-be-done                        │
  │   • Success signal                                      │
  │   • Idea-type classification (new / extend / fix)       │
  └─────────────────────────────────────────────────────────┘
       │ (one-line "using X+Y because ; override?")
       ▼
  ┌─────────────────────────────────────────────────────────┐
  │ Phase 2 — EXPAND  (auto-picked 2 techniques)            │
  │   • Generate 8–15 raw variants                          │
  │   • User picks 1–3 finalists                            │
  └─────────────────────────────────────────────────────────┘
       │
       ▼
  ┌─────────────────────────────────────────────────────────┐
  │ Phase 3 — PRESSURE-TEST  (always-on, batch)             │
  │   • Premortem (12-mo failure modes)                     │
  │   • Munger inversion ("what guarantees failure?")       │
  │   • Assumption mapping (impact × uncertainty)           │
  └─────────────────────────────────────────────────────────┘
       │
       ▼ (optional Refine — clean rewrite of artifact)
       ▼
  docs/pmos/ideation/2026-05-13_decision-commits.html
       + Next-steps handoffs: /requirements · /grill · /backlog add

Phase 1 — Frame

One short conversational turn (or auto-derived from a rich seed). Pins:

  • HMW statement — "How might we <verb> <outcome> for <user> given <constraint>?"
  • Target user + JTBD — who + what they're trying to do + when.
  • Success signal — one sentence: "we'll know this worked if <observable>".
  • Idea-type classification — new / extend / fix (rules below) — drives auto-pick in Phase 2.

Phase 2 — Expand

The skill auto-picks 2 ideation techniques based on idea-type, then emits a one-line override escape:

Idea-typeAuto-picked techniquesRationale
newFirst Principles + Crazy 8sRadical reframing + breadth burst
extendSCAMPER + Analogous InspirationStructured variation + cross-domain transfer
fixPremortem-as-generator + InversionRoot-cause hunting + "what guarantees recurrence?"
ambiguous(ask one disambiguation question)Bad framing wastes both phases

The skill emits, e.g.: "Using SCAMPER + Analogous because this looks like an extension — override?". User accepts, picks one, or names a different pair from the supported set (HMW riffs, SCAMPER, Crazy 8s, First Principles, Analogous Inspiration, Premortem-as-generator, Inversion). Output: a flat numbered list of 8–15 one-line idea variants, deliberately diverse. The user then picks 1–3 finalists (default-recommended: the strongest single one).

Phase 3 — Pressure-test (always-on)

Single non-interactive batch pass against the chosen finalist(s). Three sub-batteries, each independent:

  1. Premortem — "it's 12 months from now and this idea failed; what happened?" Output: a 3–6 row failure-modes table (mode / likelihood / mitigation).
  2. Munger Inversion — "what set of choices would guarantee this fails?" Output: 3–5 inverted-action bullets.
  3. Assumption Mapping — enumerate load-bearing assumptions, rank by (impact × uncertainty). Output: a ranked table with "cheapest test to validate" per assumption.

Pressure-test is always-on by default. Disabling it requires a deliberate --no-stress-test flag, surfaced as a one-line warning in the artifact's TL;DR.

Phase 4 — Refine (optional)

If the user opts in (or passes --refine), the skill rewrites the artifact as a clean, voice-consistent brief — trimming working notes, tightening the TL;DR, and cross-referencing the failure-modes table. Default: skip; the working artifact is itself shippable.

User Journeys

Primary journey — "I have a half-formed idea"

  1. User types /ideation what if we tracked decisions like commits?
  2. Skill reads .pmos/settings.yaml for docs_path + output_format; derives slug decision-commits; confirms slug via AskUserQuestion (Recommended: Use this slug).
  3. Skill emits Phase-1 Frame prompt — auto-fills HMW + target user + success signal from the seed if possible; asks the user (one consolidated AskUserQuestion with up to 4 sub-questions) to confirm/edit each.
  4. Skill classifies idea-type as new; announces "Using First Principles + Crazy 8s — override?"; user accepts (Recommended).
  5. Skill generates 10 variants (mix of "decision-as-commit", "decision graph", "decision review queue", inversions, analogues like git-blame-for-decisions); presents the list; asks user to pick 1–3 finalists (Recommended: strongest single).
  6. Skill runs Pressure-test batch — emits failure-modes table (5 rows), inversion bullets (4 lines), assumption map (6 ranked rows). No questions in this phase.
  7. Skill writes docs/pmos/ideation/2026-05-13_decision-commits.html (atomic, via _shared/html-authoring/ substrate with ?v=<version> cache-bust, kebab-case heading IDs, sections.json companion). Mixed-format sidecar (.md) when output_format=both.
  8. Skill prints chat summary with absolute file path and three suggested next-step commands (/requirements decision-commits, /grill docs/pmos/ideation/2026-05-13_decision-commits.html, /backlog add). User chooses whichever (or none).

Alternate journey — rich seed (no Frame questions needed)

User passes a multi-paragraph seed that already contains HMW + JTBD signal. Skill detects sufficient context, auto-derives Frame fields, surfaces them as a confirm (not fill) prompt, and proceeds. Saves one turn.

Alternate journey — user overrides technique pick

Skill announces "Using SCAMPER + Analogous — override?"; user replies "use First Principles instead". Skill replaces the second technique and proceeds. Override is one-shot — no further negotiation.

Alternate journey — --no-stress-test

User passes --no-stress-test (e.g., for a deliberately throwaway brainstorm). Skill emits a one-line warning into the TL;DR ("⚠ Stress-test skipped — failure modes and assumptions are NOT validated"); writes the artifact without Phase 3 sections; Next Steps recommends running /ideation --resume <artifact> later to add the pressure-test pass.

Error journey — empty / one-word seed

User invokes /ideation with no arg or a single word ("dashboards"). Skill issues a single AskUserQuestion with three options: (a) Use as a HMW seed and ask Frame questions interactively, (b) Provide more context (free-form reply), (c) Cancel. Recommended: (b).

Error journey — interrupted run

If the user kills the run between Phase 2 and Phase 3, the partial artifact has been atomically written through Phase 2 (Frame + Expand sections present, Pressure-test section marked Pending). Re-invoking with /ideation --resume <artifact-path> picks up at Phase 3. Re-invoking without --resume warns about the existing artifact (overwrite / pick-new-slug / cancel).

Design Decisions

#DecisionOptions ConsideredRationale
D13-phase loop (Frame → Expand → Pressure-test)(a) 3-phase, (b) 4-phase (add Refine as mandatory), (c) 2-phase (Generate → Critique)3-phase is the shortest loop that still contains the differentiating value (pressure-test). 4-phase makes Refine mandatory — most ideas don't earn the polish cost. 2-phase folds Frame into Generate, losing the HMW discipline that prevents bad framing. Refine stays available as optional Phase 4.
D2Artifact path: {docs_path}/ideation/{YYYY-MM-DD}_<slug>.html(a) in-repo at {docs_path}, (b) global at ~/.pmos/ideation/, (c) hybrid (in-repo default, --global escape)In-repo matches the user's explicit ask and the /artifact//wireframes//survey-design pattern. Global would parallel /backlog//mytasks, but ideation is intentionally project-shaped (the idea lives near the work). A future --global flag stays an option, not a requirement.
D3Pressure-test is always-on (escape: --no-stress-test)(a) always-on, (b) opt-in flag, (c) opt-out flagPressure-test is the differentiating value vs. /superpowers:brainstorming and /creativity. Making it opt-in means users skip the half that matters. Opt-out flag preserves the escape for deliberate cases (with a warning in the artifact).
D4Technique selection: auto-pick by idea-type + one-line override escape(a) auto-pick, (b) always ask user, (c) always run all techniquesAuto-pick saves a turn and signals the skill has an opinion. The override escape preserves user control. "Always run all" is expensive, redundant for narrow ideas, and dilutes the strongest variants.
D5Seam vs. /grill and /creativity: /ideation stays standalone; suggests handoffs in Next Steps(a) stay standalone, (b) fold /creativity in as Expand, (c) chain into /grill automatically(a) preserves orthogonal surfaces — each skill has one job. Folding /creativity in erases its standalone surface and couples release cycles. Auto-chaining to /grill defeats /grill's contract (it targets committed plans, not uncommitted ideas).
D6Standalone (no workstream load), no pipeline-mode dispatch(a) standalone, (b) workstream-aware, (c) optional --workstreamMatches the /diagram//polish//design-crit//survey-design pattern. Workstream bias is exactly what we don't want at the ideation stage — variant breadth depends on letting in non-workstream framings.
D7HTML primary, MD sidecar via output_format=both(a) HTML primary, MD on opt-in, (b) MD primary, (c) HTML-onlyMatches every other pmos-toolkit artifact-producing skill. _shared/html-authoring/ substrate handles the heavy lifting (template, assets, sections.json, cache-bust, heading IDs).
D8Resumability: partial artifact written at each phase boundary; --resume <path> re-enters the cursor(a) per-phase atomic writes + --resume, (b) single end-of-run write (no resume), (c) full state.yaml like pipeline skills(a) gives the user a recoverable artifact without the pipeline-style state file overhead. The phase cursor lives in the artifact (an HTML data-attr / MD frontmatter field), not a sidecar.

This spec turns the §A–§F constraints in plugins/pmos-toolkit/skills/feature-sdlc/reference/skill-patterns.md into concrete FRs and acceptance criteria. The Phase 6a /skill-eval rubric will score the produced skill against the matching bijective check_ids in reference/skill-eval.md.

Architecture

A single Claude Code skill under plugins/pmos-toolkit/skills/ideate/, structured per the pmos-toolkit standalone-utility shape (model: /polish, /diagram, /survey-design):

plugins/pmos-toolkit/skills/ideate/
├── SKILL.md                       # Entry point — phases, control flow (target ~400 lines)
├── reference/
│   ├── techniques.md              # The 6 supported techniques + prompt templates
│   ├── idea-type-classifier.md    # new / extend / fix rules + disambiguation question
│   ├── pressure-test-battery.md   # Premortem / Inversion / Assumption-map prompts
│   ├── artifact-template.html     # HTML skeleton for the per-idea artifact
│   └── slug-derivation.md         # Idea-slug rules (mirrors feature-sdlc but shorter)
└── assets/                        # (copied lazily from _shared/html-authoring/ at run time)

No scripts/ directory — the skill is prompt-driven; no bundled executables. (Group 10.E of skill-eval.md auto-skips per the group-skip rules.)

Substrate dependency: the skill writes HTML/MD artifacts via the existing _shared/html-authoring/ substrate (template, assets, sections.json, cache-bust, heading IDs). The skill does NOT re-implement HTML emission — it consumes the substrate exactly like /polish, /wireframes, /spec, /plan do.

Codex sidecar: at plugins/pmos-toolkit/skills/ideate/agents/openai.yaml — required for --target codex parity (decision F-codex-sidecar of skill-eval). Matches the existing per-skill sidecar pattern.

Functional Requirements

FR-INVOCATION — argument parsing

  • FR-01: Skill is invoked as /ideate <seed-text> where <seed-text> is free-form (a sentence to several paragraphs). Empty seed → error journey (single AskUserQuestion per requirements §user-journeys).
  • FR-02: Supported flags (enumerated in argument-hint): --format <html|md|both> (overrides .pmos/settings.yaml :: output_format), --no-stress-test (skips Phase 3), --refine (forces Phase 4), --slug <custom> (overrides derived slug), --resume <artifact-path> (re-enters at the partial-write phase cursor), --non-interactive / --interactive (mode override per _shared/non-interactive.md contract).
  • FR-03: Last flag wins on conflict. Unknown flag → stderr warn + exit 64. --non-interactive on a skill without a Recommended default for any required AskUserQuestion → DEFER per the canonical non-interactive block.

FR-FRAME — Phase 1

  • FR-10: Auto-derive HMW + target user + success signal from the seed if it contains a verb + object + audience signal; otherwise emit one consolidated AskUserQuestion with up to 4 sub-questions to gather them.
  • FR-11: Classify idea-type per reference/idea-type-classifier.md. Rules (regex-style, ordered, first-match-wins): new|build|what if|invent → new; improve|extend|better|enhance|add to → extend; fix|bug|broken|complain|wrong|painful → fix; else → ambiguous (issue one disambiguation AskUserQuestion).
  • FR-12: Emit the technique-pair announcement as a single chat line: Using <technique1> + <technique2> because this looks like a <type> — override? with no AskUserQuestion. User may reply with a free-form override or "ok" to continue. (Conversational, not structured-ask — per requirements §design-decisions D4.)

FR-EXPAND — Phase 2

  • FR-20: Generate ≥8 and ≤15 distinct one-line idea variants using the two techniques. Each variant is ≤120 chars. Variants are deliberately diverse (no two paraphrase the same idea).
  • FR-21: Present the variants as a numbered list in chat; emit one AskUserQuestion asking the user to pick 1–3 finalists. Options: "Pick #N (Recommended: strongest single)" + "Pick multiple — reply with numbers" + "Regenerate — different angles".
  • FR-22: Persist a partial artifact (Frame + Expand sections populated; Pressure-test placeholder) after FR-21 resolves, via the atomic-write contract (FR-30).

FR-PRESSURE-TEST — Phase 3 (always-on)

  • FR-30: Run three sub-batteries from reference/pressure-test-battery.md in one non-interactive pass:
    1. Premortem: 3–6-row failure-modes table (mode / likelihood / mitigation).
    2. Munger Inversion: 3–5 inverted-action bullets ("what choices guarantee failure?").
    3. Assumption Mapping: 4–8-row ranked table (assumption / impact / uncertainty / cheapest-test).
  • FR-31: When multiple finalists are picked in Phase 2 (open question OQ1 from requirements): run the battery once per finalist, then emit a final cross-cutting decision table comparing them on a fixed 3-axis schema (risk-density, assumption-load, ease-of-validation).
  • FR-32: --no-stress-test short-circuits Phase 3; the artifact emits a TL;DR warning line ("⚠ Stress-test skipped"); Next Steps recommends a --resume invocation to add the battery later.

FR-REFINE — Phase 4 (optional)

  • FR-40: Default: skip. When --refine is passed OR the user explicitly opts in via an end-of-Phase-3 AskUserQuestion (Recommended Skip), the skill rewrites the artifact for voice consistency, tightens the TL;DR, and cross-links failure-modes/assumptions.

FR-ARTIFACT — output

  • FR-50: Artifact path: {docs_path}/ideate/{YYYY-MM-DD}_<idea-slug>.html. Mixed-format sidecar (.md) when output_format=both. {docs_path} from .pmos/settings.yaml.
  • FR-51: Slug derivation per reference/slug-derivation.md — kebab-case, max 4 words, lowercase, drop stopwords. Collision: append -2, -3, … (idempotent on re-run with the same seed).
  • FR-52: Atomic write: temp-then-rename for both .html and the sections.json companion. Cache-bust ?v=<plugin-version> on all asset URLs. Heading IDs kebab-case per _shared/html-authoring/conventions.md §3. Top-level artifact: asset prefix ../assets/ (the ideation/ dir is one level under docs_path, so its assets/ resolves to {docs_path}/assets/ shared across all ideation outputs).
  • FR-53: Artifact contains the 12 sections from requirements §artifact-section-list, in order. Each section is a <section id="..."> with kebab-case heading IDs.
  • FR-54: Phase cursor stored in a meta tag in the HTML: <meta name="pmos:ideation-phase" content="frame|expand|pressure-test|refine|complete">. --resume reads this tag.

FR-HANDOFF — Next Steps

  • FR-60: The artifact's Next Steps section names 1–3 candidate follow-up commands. Selection rules: idea-type=new → suggest /requirements <slug> + /backlog add; idea-type=extend → suggest /requirements <slug> + /grill <artifact-path>; idea-type=fix → suggest /grill <artifact-path> + /backlog add. All cases include /ideate --refine --resume if Phase 4 was skipped.

FR-PLATFORM — multi-platform

  • FR-70: SKILL.md contains a ## Platform Adaptation section covering: no AskUserQuestion (degrade to numbered free-form per _shared/interactive-prompts.md); no subagents (single-agent execution); no .pmos/settings.yaml (run _shared/pipeline-setup.md §A first-run setup); non-interactive contract (inline canonical block).
  • FR-71: argument-hint in frontmatter enumerates every flag parsed by the body (matches FR-02 list exactly).
  • FR-72: Codex sidecar at agents/openai.yaml with required Codex fields.

FR-SKILL-PATTERNS — §A–§F conformance (Phase 6a gate)

  • FR-80 (§A): frontmatter — name: ideate (matches dir), description ≤1024 chars stating when + what + ≥5 user-spoken trigger phrases + assertive, user-invocable: true, argument-hint.
  • FR-81 (§B): description is third-person, contains "Use when…", no embedded numbered workflow, ≥5 trigger phrases inline. Trigger phrases to include: "help me brainstorm this idea", "stress-test this idea", "I have a half-formed idea", "ideate on X", "what should we build to solve Y", "pressure-test this concept", "/ideate".
  • FR-82 (§C): SKILL.md body ≤500 lines (target: ~400). Reference files in reference/, depth ≤1, each >100-line ref has a ToC in first ~15 lines. No hard-coded absolute paths; portable tokens only.
  • FR-83 (§D): ## Platform Adaptation present, ## Track Progress present, learnings-load line + numbered ## Phase N: Capture Learnings phase. Imperative voice. Rationale given for non-obvious instructions.
  • FR-84 (§E): no bundled scripts — group skips. e-eval-driven satisfied by the per-skill SDLC run producing this artifact stack.
  • FR-85 (§F): user-invocable: true + argument-hint match parsed flags; Codex sidecar present.

Non-functional Requirements

  • NFR-01: Single-pass happy-path runtime ≤ 15 minutes wall-clock for a Tier-2-sized idea.
  • NFR-02: Skill body ≤500 lines (skill-eval c-body-size threshold).
  • NFR-03: Zero external runtime dependencies (no scripts → no interpreters / packages to document).
  • NFR-04: Resumability: any kill between Phase 1 and Phase 4 leaves a re-enterable partial artifact.
  • NFR-05: No workstream load (standalone utility — per requirements D6).

Acceptance Criteria

  1. tools/skill-eval-check.sh --target claude-code plugins/pmos-toolkit/skills/ideate/ returns exit 0 (all 20 [D] checks pass; group 10.E skipped because no scripts).
  2. A reviewer subagent running the 19 [J] checks returns 0 fails (or ≤2 fails with substrings of skill source quoted, captured as accepted_residuals per Phase 6a's cap).
  3. An end-to-end manual run with seed "what if we tracked decisions like commits?" produces docs/pmos/ideate/2026-05-13_decision-commits.html with all 12 sections non-empty, the Pressure-test battery populated, and Next Steps naming concrete slash commands.
  4. The same run with --no-stress-test produces an artifact with a TL;DR warning and Pressure-test sections marked Skipped.
  5. The same run interrupted between Phase 2 and Phase 3, then resumed via --resume, completes successfully and produces an identical Phase-3 onwards.
  6. README has a /ideate row under "Utilities"; both plugin.json manifests bump 2.42.0 → 2.43.0 in sync; changelog has a 2.43.0 entry.

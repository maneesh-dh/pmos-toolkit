Requirements — /plan vertical-slice enforcement

Tier: 2 (enhancement) · Target skill: /plan (plugins/pmos-toolkit/skills/plan/SKILL.md) · Mode: interactive · Feedback source: <inline-text> via /skill-sdlc --from-feedback · Standing acceptance criteria: reference/skill-patterns.md §A–§F

Problem

Plans generated by /plan currently have no mechanism for enforcing vertical-slice task decomposition. The skill body never mentions vertical slices, tracer bullets, or end-to-end slicing, and — worse — the worked example shipped in the ## Phase N groupings section trains plans toward the wrong shape: "Phase 1: Schema and Migration / Phase 2: API Layer" is literally horizontal layering. The consequence is plans that defer integration risk: a developer can complete 80% of the tasks before any user-observable behavior exists, blast radius of bugs widens, and the "does this design actually work end-to-end" question is answered last instead of first.

Proposed solution

Add an explicit, citable mechanism in /plan/SKILL.md that biases task decomposition toward thin vertical slices and pivots the existing horizontal example. Specifically (approved in Phase 0c triage):

  1. New sub-section in Phase 3 — "Vertical-Slice Decomposition" — defining the rule (each task = a thin slice through every layer it needs), the acceptance criteria, tracer-bullet T1 convention, the "many thin over few thick" preference, and the refactor/spike exception path.
  2. Pivot the ## Phase N: <name> worked example from horizontal layering to a vertical-slice example, with a one-line caveat that horizontal phases are an anti-pattern.
  3. Phase 4 structural-checklist item enforcing the rule (and a finding when violated).
  4. New Anti-Pattern entry that explicitly names horizontal-layer decomposition as forbidden.
  5. Phase 2 step earmarking the tracer-bullet candidate from the spec.

User journey (skill author / pipeline operator)

An engineer hands a spec to /plan. The plan output now opens with a tracer-bullet T1 (e.g., "save one record end-to-end via the new endpoint, render it on the page") and widens from there. The engineer running /execute ships a working slice within the first commit cycle; integration risk surfaces immediately rather than at TN. If the planner drifts toward horizontal layering, Phase 4 review catches it as a finding and the user gets a structured disposition (Fix / Modify / Skip / Defer).

Acceptance criteria

  • The phrase "vertical slice" appears in /plan SKILL.md in at least: Phase 3 (rule), Phase 4 (review check), Anti-Patterns (forbidden shape).
  • The phrase "tracer bullet" appears in Phase 3 (T1 convention) and ideally Phase 2 (candidate identification).
  • The existing ## Phase N worked example does NOT contain "Schema and Migration / API Layer / UI" as phase names (the horizontal example is pivoted).
  • The new Anti-Pattern entry explicitly forbids layer-wise grouping ("all migrations, then all endpoints, then all UI").
  • The new Phase 4 check has a concrete pass condition (a finding fires when ≥1 task is a single-layer cut that has no refactor/spike justification).
  • Refactor and spike tasks have a documented exception path (they declare why they aren't vertical).
  • skill-patterns.md §A–§F still passes (the change must satisfy the standing skill-eval rubric — no broken frontmatter, no progressive-disclosure regression, no missing-Capture-Learnings phase, etc.).
  • No regression in the existing Tier-1/2/3 emission rules, Phase 4 cap-of-4 loops, Skip List sidecar contract, or non-interactive halt protocol — the change is additive.

Out of scope

  • Changes to /execute, /verify, or /spec — the change lives entirely in /plan/SKILL.md.
  • A separate reference/slicing-patterns.md — defer; if the in-body rule is clear, no sidecar is needed. Tracked as OQ if Phase 4 review judges otherwise.
  • Automated detection of horizontal slicing (an LLM-judge or regex on task names) — Phase 4 review is human-disposed; pure-prose rule is sufficient.
  • Re-running /plan against historical plans to detect drift — backfill is not in scope.

Constraints

  • Canonical skill path (CLAUDE.md §"Canonical skill path"): edits target only plugins/pmos-toolkit/skills/plan/SKILL.md.
  • Manifest version sync (CLAUDE.md §"Plugin manifest version sync"): both plugin.json manifests bump to the same new version in one commit (handled by /complete-dev).
  • Release entry point: /complete-dev, not /push.
  • Standing rubric: reference/skill-patterns.md §A–§F — body content must remain progressive-disclosure-friendly (no dumped reference content), frontmatter intact, Phase Capture-Learnings unchanged, Anti-Patterns list intact.
  • Insertion strategy (per 2026-05-13 learnings): prefer adding a new sub-section in Phase 3 over renumbering Phase 4..7. Use a fractional phase number only if a wholly new phase is required (it is not in this case).

Risks

  • Rule too rigid — pure refactors, schema-only spikes, and CSS-only changes genuinely can't be vertical. Mitigation: explicit exception path declared in Phase 3 rule + Phase 4 check honors the declared exception.
  • Example pivot under-specified — replacing horizontal example with a vague vertical example creates a different kind of training defect. Mitigation: spec phase produces the literal replacement text, reviewed in /grill.
  • Anti-pattern wording too strong — saying "NEVER" could overshoot; some plans legitimately need a non-trivial schema task before any slice. Mitigation: anti-pattern targets "pure layer-wise grouping with no end-to-end slice until the end", not "any task that touches one layer."

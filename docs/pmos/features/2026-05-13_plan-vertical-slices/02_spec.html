Functional requirements

FR-VS-01: Vertical-Slice Decomposition rule in Phase 3

Add a sub-section titled "Vertical-Slice Decomposition" immediately after the existing "Task Design Rules" anchor in Phase 3 (around current line 505), BEFORE the "Optional: ## Phase N Groupings" sub-section. The sub-section MUST contain, in order:

  1. Rule: "Each task is a thin vertical slice that cuts through every layer it needs (schema, API, UI, tests) to deliver one user-observable behavior end-to-end. A horizontal cut (one layer only) is the wrong shape."
  2. Tracer bullet: "T1 is a tracer bullet — the minimal end-to-end path that proves the architecture works. Prefer the narrowest, dullest, hardcoded-where-needed slice that still exercises every integration layer involved."
  3. Preference: "Prefer many thin slices over few thick ones."
  4. Done-when: "A completed slice is independently demoable or verifiable on its own; it doesn't require T+1 to land before you can show something works."
  5. Exception: "Refactor, schema-only spike, pure-CSS, and config/IaC tasks cannot be vertical. Such tasks MUST declare the exception with a one-line rationale (e.g., **Slice shape:** refactor-prep — extracts the layer T2 needs before its vertical cut.)."
  6. Field convention: "Each task MAY include a **Slice shape:** vertical | refactor-prep | spike | css-only | config field. When omitted, vertical is assumed; non-vertical tasks MUST include it."

Total addition: ≈25-35 LOC of prose under a single ### Vertical-Slice Decomposition heading.

FR-VS-04: Anti-Pattern entry

Append a new bullet to the existing ## Anti-Patterns (DO NOT) list. Wording:

- Do NOT decompose by layer — "all migrations first, then all endpoints, then all UI." Each task is a thin vertical slice that cuts through every layer it needs to deliver one user-observable behavior. T1 is a tracer bullet: the narrowest end-to-end path that proves the architecture. Refactors, spikes, CSS-only, and config-only tasks are the only legitimate exceptions and MUST declare **Slice shape:** with a rationale.

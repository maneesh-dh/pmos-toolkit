Requirements — /polish: source-format honoring + editorial reduction pass

Mode: skill-feedback · interactive · Tier 2 · output_format=html
Target skill: /polish — plugins/pmos-toolkit/skills/polish/
Seed: 0c_feedback_triage.html (3 approved findings: F1 format honoring, F2 editorial pass, F3 ordering)
Standing acceptance criteria: the revised skill must conform to plugins/pmos-toolkit/skills/feature-sdlc/reference/skill-patterns.md §A–§F and this repo's CLAUDE.md ## Skill-authoring conventions (canonical path, manifest version sync, /complete-dev as release entry).

Problem

/polish today does two things that this change loosens:

  1. It is markdown-only on output. Phase 1 ingest normalizes every source (file, URL, Notion) to markdown; Phase 7 writes <original>.polished.md regardless of what the input was. A user who hands in an .html document gets markdown back — wrong shape for their workflow, and the replace prompt can't safely overwrite the original.
  2. It only does span-local rewrites. The 14-check rubric flags clarity/concision/AI-slop violations and Phase 4 generates a patch per flagged span. There is no holistic "this doc is 40% too long — cut it" capability. Users who want a draft shortened (PRDs, blog posts, READMEs that bloated over time) have no lever for it.

Goals

  • G1. When the input is a local .md/.markdown file, output is markdown (unchanged behavior). When the input is a local .html/.htm file, output is HTML — same shape in, same shape out. URL and Notion inputs continue to normalize to markdown (their HTML is page chrome, not an authored artifact); inline-text input continues to be treated as markdown prose.
  • G2. A new, opt-in editorial reduction pass: the user picks a reduction target (preset range or custom), an editor subagent critiques the doc ruthlessly and emits structured notes to editor_notes.json, a rewriter subagent applies those notes to produce a reduced doc.
  • G3. The editorial pass runs before the existing binary rubric, so the rubric (and the budget estimate, patch passes, findings protocol, and metrics) all operate on the reduced doc.
  • G4. Voice preservation, lock zones, the 2-iteration rubric cap, and the "no silent high-risk edits" guarantee all still hold. The editor pass does not consume a rubric iteration; high-risk editor notes are surfaced for approval, not auto-applied.
  • G5. The revised skill stays conformant with skill-patterns.md §A–§F and this repo's release conventions (manifest version bump in both plugin.json files, README row unchanged-but-accurate, changelog entry).

Non-goals

  • Not adding support for any input format beyond markdown and HTML (no DOCX, RST, AsciiDoc, PDF).
  • Not making the editor pass mandatory or default-on — Skip is the recommended default.
  • Not changing the 14 rubric checks themselves, the preset semantics, or the voice-sampling algorithm.
  • Not building a "grow this doc" / expansion mode — reduction only.
  • Not round-tripping HTML through markdown internally — HTML docs are polished as HTML (prose nodes extracted in place), not converted-and-converted-back.
  • Not changing how URL/Notion inputs are handled (they remain markdown-normalized).

User journeys

J1 — Shorten a bloated PRD (markdown)

User: /polish docs/prd.md. Skill ingests markdown, classifies it as a PRD, asks for a preset. Then the new editor gate: "Reduction target?" → user picks ~30-40% (substantial cut). Editor subagent returns 22 notes (8 cuts, 6 merges, 5 tightens, 3 reorders), estimated ~36% saved; writes editor_notes.json. Rewriter applies the 19 low-risk notes, surfaces the 3 reorders for approval; user approves 2. Reduced doc (≈33% shorter) flows into the rubric, which now finds only 2 failures (vs 9 on the original). Final prd.polished.md is shorter and cleaner; Phase 7 metrics show Words: 4,200 → 2,710 (-35%) anchored to the original.

J2 — Polish an HTML doc, get HTML back

User: /polish report.html. Skill detects HTML, builds HTML-aware lock zones (tags, attributes, <script>/<style>, comments, <pre>/<code>), chunks on <h1>/<h2> if large. Rubric runs on the extracted prose text; patches are applied to prose nodes only, never to markup. Editor gate offered as usual. Output is report.polished.html; replace prompt offers to overwrite report.html (git is the safety net; .bak if not in a repo).

J3 — Skip the editor pass (default)

User: /polish notes.md, just wants cleanup. At the editor gate, the recommended option is Skip — no reduction; user accepts. The phase is a no-op; the pipeline behaves exactly as it does today. No editor_notes.json is written.

J4 — Dry-run with a reduction target

User: /polish draft.md --dry-run, picks ~50%+ at the editor gate. The editor subagent still runs and writes editor_notes.json; the rewriter does not run; Phase 4 stops and the report includes both the editor notes (with the target-vs-estimate reconciliation) and the rubric results — so the user can see what would be cut and what would be fixed before committing.

J5 — Non-interactive run

User: /polish draft.md --non-interactive. The editor gate is a structured ask with a Recommended (Skip) → auto-picked per the canonical non-interactive block; the deferral is logged to the OQ section. The format-detection is deterministic (no prompt). Everything else as today.

Requirements

Source-format honoring (F1)

  • FR-1. Phase 1 resolves doc_format ∈ {markdown, html} from the input: local file with extension .html/.htm → html; local file with .md/.markdown/.mdown/.txt or no recognized extension → markdown; URL / Notion / inline-text → markdown (unchanged). The detection is deterministic — never an LLM call, never a prompt.
  • FR-2. When doc_format == html: lock zones additionally cover all HTML tags and their attributes, <script>/<style> contents, HTML comments, and <pre>/<code> contents — the rubric and all patches operate only on the extracted prose text of element bodies. (Markdown lock zones unchanged.)
  • FR-3. When doc_format == html and the doc is large enough to chunk, chunk boundaries are <h1>/<h2> elements (the HTML analogue of the existing H1/H2 markdown chunking). Stitching must produce byte-identical markup outside the rewritten prose nodes.
  • FR-4. Phase 7 writes <original-basename>.polished.html for HTML inputs and <original-basename>.polished.md for markdown inputs. The replace prompt, the in-repo mv, and the .bak-then-write fallback all use the matching extension. URL/Notion/inline outputs are still printed (no file), as markdown.
  • FR-5. Wording: the SKILL.md description, argument-hint, "File map", and anti-patterns are updated to say "single document" / "single doc" where they currently say "single markdown document" — the skill is no longer markdown-only. The argument-hint input description notes that local .html files round-trip as HTML.
  • FR-6. The 14 rubric checks are unchanged — they evaluate prose text, which is extracted identically regardless of format. No new checks; no check is HTML- or markdown-specific.

Editorial reduction pass (F2 / F3)

  • FR-7. A new phase — call it Phase 2.5 "Editorial reduction" — runs after the preset phase (current Phase 2) and before the binary rubric (current Phase 3). All subsequent phases (rubric, budget, patches, findings, apply, metrics) operate on the phase's output document.
  • FR-8. The phase opens with an AskUserQuestion reduction-target gate. Options: Skip — no reduction (Recommended) / ~10-20% (light trim) / ~30-40% (substantial cut) / ~50%+ (aggressive), with a custom free-text option (a single % or a range like 25-35%). The recommended/default is Skip — the editor pass is opt-in.
  • FR-9. Skip ⇒ the phase is a no-op: no subagents dispatched, no editor_notes.json written, the ingested doc is the working doc, and the rest of the pipeline is byte-for-byte today's behavior.
  • FR-10. On any non-Skip target, dispatch the editor subagent with: the ingested doc (verbatim), the voice markers from Phase 1, the lock-zone map, and the reduction target. It returns structured critique notes — it does not rewrite. It must respect lock zones (no note may target a locked span). It produces a target-vs-estimate reconciliation (sum of est_words_saved vs target band; if it can't reach the target without touching protected content or harming meaning, it says so and proposes the best achievable band).
  • FR-11. editor_notes.json schema (validated against schemas/editor-notes.schema.json): top-level {doc_format, target: {kind: "range"|"point", low, high}, reconciliation: {estimated_pct, achievable: bool, note}, notes: [...]} where each note is {id, kind ∈ "rephrase"|"merge"|"reorder"|"tighten"|"cut", locator: {heading_path: [...], quote: "<≥20-char verbatim substring of the source>"}, rationale, est_words_saved: int, risk ∈ "low"|"high"}. No line numbers anywhere (they go stale — same rule as defer comments). The phase writes editor_notes.json next to the other run artifacts.
  • FR-12. Dispatch the rewriter subagent with: the original ingested doc (verbatim), editor_notes.json, the voice markers, and the lock-zone map. It applies the low-risk notes, honoring lock zones (a note whose quote can't be located verbatim, or that would touch a locked span, is reported as skipped, not force-applied). It emits the reduced doc plus an applied/skipped log (per-note: applied | skipped + reason). high-risk notes (structural reorders, anything ambiguous) are not auto-applied — they are surfaced to the user through the Phase 5 findings protocol (same "Fix as proposed / Modify / Skip / Defer" shape), and only then applied.
  • FR-13. Re-using the determinism contract already in the skill: both subagents run at temperature: 0 with structured output; a cut/tighten note whose quote is empty or not a verbatim substring of the source is dropped (no evidence → no action) — the same rule the rubric already applies to cited_spans.
  • FR-14. The editor pass does not count toward the 2-iteration rubric cap (it is "iteration 0"). The cap text in Phase 6 and the anti-patterns is clarified accordingly.
  • FR-15. Phase 4's budget-estimate block gains a conditional line: when the editor pass produced output, "+~2 subagent calls (editor critique + rewrite)". The Surgical/Comprehensive shape is still not introduced — this is purely an additive note to the existing estimate block.
  • FR-16. Phase 7 before/after metrics anchor the "Words" delta to the original ingested doc (not the editor-reduced doc), so the headline % reflects the full reduction (editor cut + rubric tightening). The summary block gains an "Editorial pass:" line: target band, estimated %, actual %, notes applied/skipped/surfaced — or "skipped" when the gate was Skip.
  • FR-17. --dry-run with a non-Skip target: the editor subagent runs and writes editor_notes.json; the rewriter does not run; Phase 4 stops as today and the dry-run report includes the editor notes + reconciliation alongside the rubric report.
  • FR-18. In --non-interactive mode the reduction-target gate is a Recommended-bearing structured ask → auto-picked (Skip) per the canonical non-interactive block; the deferral is appended to the OQ buffer. (No defer-only tag on this gate — it has a Recommended option and is safe to auto-pick.)

Risks & open questions

  • HTML prose extraction fidelity. Re-stitching a rewritten HTML doc so that everything outside the prose nodes is byte-identical is fiddly (whitespace, self-closing tags, entity encoding). Mitigation: the rewriter operates on text-node boundaries only and the lock-zone map records exact byte ranges; a post-write check (analogue of the existing "chunk boundary lines byte-identical" check) compares all non-prose bytes. If fidelity can't be guaranteed for a given doc, fall back to "refuse HTML rewrite for this doc, polish-as-markdown-and-warn" rather than emit broken HTML — to be decided in spec.
  • Editor target unachievable. A doc that's mostly code blocks / tables can't be cut 50% by prose edits alone. The reconciliation field surfaces this; the rewriter cuts what it safely can and the Phase 7 summary reports actual vs target. No hard failure.
  • Interaction with chunking. If both the editor pass and the rubric chunk a large doc, the editor pass should chunk-and-reduce first, then the (now smaller) reduced doc is re-bucketed for the rubric. Spec to nail the ordering.
  • Voice drift under aggressive reduction. At ~50%+ the rewriter is making big cuts; voice markers must still be injected into the rewriter prompt and a PRESERVE_VOICE_CONFLICT from the rewriter is promoted to a high-risk finding, same as the patch passes.

Grill resolutions (2026-05-13)

  • GR-1 (refines FR-4 / risks). HTML rewrite fidelity fallback = best-effort HTML + warn. If the rewriter cannot guarantee byte-identical non-prose bytes, it still emits the rewritten .polished.html but the Phase 7 summary and the chat output carry a loud warning ("markup outside prose nodes may have shifted — review before replacing"); the replace prompt is not auto-defaulted to yes in that case. Never refuse, never hard-fail.
  • GR-2 (refines "interaction with chunking" risk). Editor pass and rubric chunking = reduce first, then re-bucket. The editor pass chunks (on H1/H2 markdown headings or <h1>/<h2> HTML headings) and reduces; the resulting reduced doc is then re-measured against the size thresholds for the rubric/patch phase (it may now be small enough to skip chunking).
  • GR-3 (refines FR-10/FR-12). Editor re-critique = one re-critique, capped at 1. If the rewriter's actual reduction lands below the low end of the target band, the editor subagent gets a single second pass (handed the rewriter's output + applied/skipped log + the shortfall), producing a delta editor_notes.json (appended, not replacing); the rewriter applies the delta once more. After that one re-critique, whatever the actual reduction is, the pipeline proceeds — no further looping. This is independent of the 2-iteration rubric cap.
  • GR-4 (new, extends FR-18/FR-21). Add a --reduce <pct|range> CLI flag (e.g. --reduce 30-40, --reduce 25). When present it sets the reduction target and the Phase 2.5 gate is not shown (parallels how --preset suppresses the Phase 2 prompt). This makes the editor pass usable in --non-interactive runs. The flag goes in argument-hint; a malformed value (not a % or a low-high range, or low>high, or >90%) → print the offending value and continue with the editor pass skipped (do not abort).

Acceptance criteria (summary)

  1. An .html input produces an .html output; an .md input produces an .md output; URL/Notion/inline still produce markdown; format detection is deterministic.
  2. HTML inputs: markup is never rewritten; the rubric/patches touch prose text only; non-prose bytes are byte-identical in the output.
  3. The editor gate offers Skip (Recommended) + 3 preset ranges + custom; Skip is a true no-op.
  4. Non-Skip target ⇒ editor_notes.json is written, validates against schemas/editor-notes.schema.json, contains structured notes with verbatim quotes and no line numbers; the editor subagent does not rewrite.
  5. The rewriter applies low-risk notes, surfaces high-risk ones via the findings protocol, honors lock zones, and produces a measurably shorter doc + an applied/skipped log.
  6. The editor pass runs before the rubric; the rubric/budget/patches/metrics all see the reduced doc; the pass doesn't consume a rubric iteration.
  7. Phase 7 metrics anchor the word-count delta to the original; the summary has an "Editorial pass:" line.
  8. --dry-run + target ⇒ editor notes written, rewriter skipped, report includes both editor notes and rubric results.
  9. SKILL.md says "10 phases" everywhere; reference/editorial-pass.md and schemas/editor-notes.schema.json exist; fixtures + expected.yaml cover HTML round-trip and a reduction run.
  10. Skill conforms to skill-patterns.md §A–§F; release prereqs (synced version bump in both manifests, changelog) are met for /complete-dev.

Spec — /polish: source-format honoring + editorial reduction pass

Tier 2 · skill-feedback · target: plugins/pmos-toolkit/skills/polish/
Source: 01_requirements.html (FR-1…FR-22 + GR-1…GR-4). Standing acceptance criteria: plugins/pmos-toolkit/skills/feature-sdlc/reference/skill-patterns.md §A–§F + this repo's CLAUDE.md ## Skill-authoring conventions.

1. Overview of the change

Two capability additions to /polish, plus the bookkeeping they imply:

  • Format honoring — a doc_format ∈ {markdown, html} dimension threaded from Phase 1 ingest to Phase 7 write. HTML inputs get HTML-aware lock zones + chunk anchors and an .polished.html output. Everything else stays markdown.
  • Editorial reduction pass — a new Phase 2.5 between preset selection (current Phase 2) and the binary rubric (current Phase 3): a reduction-target gate (or the new --reduce flag), an editor subagent (critique → editor_notes.json), a rewriter subagent (apply → reduced doc), and a capped (1×) editor re-critique if the target band is missed.

Phase numbering shifts: the skill goes from 9 numbered phases to 10. The new phase is "Phase 2.5" conceptually but is rendered as a full phase (existing Phase 3→7 keep their numbers; the new one is inserted as "Phase 2.5 — Editorial reduction" to avoid renumbering churn in references, OR all phases renumber 0…9 — implementer's choice, but the "create one TodoWrite task per phase" count and the anti-pattern wording must agree with whatever scheme is chosen; **decision: keep existing numbers, insert "Phase 2.5", say "10 phases" in Track Progress**).

2. Format detection & the doc_format dimension

2.1 Detection (FR-1)

In Phase 1, after resolving the input source, set doc_format:

Input shapedoc_format
Local file, extension (case-insensitive) .html or .htmhtml
Local file, extension .md / .markdown / .mdown / .mkd / .txt / none / anything elsemarkdown
http(s)://… URLmarkdown (WebFetch strips HTML chrome to markdown — unchanged)
notion://<id>markdown (unchanged)
Quoted inline textmarkdown (unchanged)

Pure extension check — no content sniffing, no LLM call, no prompt. If a file has no extension or a weird one, it's treated as markdown (today's behavior; the new rule only adds the HTML branch).

2.2 HTML-aware lock zones (FR-2)

When doc_format == html, the Phase-1 lock-zone computation (reference/chunking.md) gains these HTML-specific zones, recorded as byte ranges exactly like the markdown ones:

Lock zone (HTML)Detection rule
All element tags + attributesEvery <tag …> / </tag> — the markup itself, including all attribute names and values. Only text between tags is polishable.
<script> / <style> contentsEverything between the open and close tag (and the tags).
<pre> / <code> contentsSame as fenced code blocks in markdown — locked.
HTML comments<!-- … --> spans (this also covers any <!-- POLISH: … --> defer markers — they stay put).
<a href> URLs, <img src>Covered by "tags + attributes" — link text (the element body) is still polishable.
Doctype, <head> contents, frontmatterThe whole <!DOCTYPE> + <head>…</head> region is locked (titles/meta are not prose to polish). Any leading ---…--- YAML frontmatter stays locked as in markdown.
Short table cells (<8 words)Same rule as markdown — <td>/<th> bodies under 8 words are locked.

Consequence (unchanged from the existing contract): the rubric never fires inside a locked zone, and any patch that intersects one is rejected. "Polishable words" for size bucketing = total words − words inside locked zones, computed identically; for HTML this is the visible prose text of body elements.

2.3 HTML chunking (FR-3 / GR-2)

When doc_format == html and polishable words ≥ 4,000, chunk on <h1> / <h2> elements (the analogue of markdown H1/H2). Boundaries land immediately before an <h1>/<h2> open tag. Oversized sections split on <h3>, then on block-element (<p>) boundaries with a 200-word read-only overlap — mirroring the markdown rules. Stitch-back (Phase 7) verifies that all non-prose bytes (the entire markup skeleton + locked-zone contents) are byte-identical to the original; only text-node bytes inside polished prose elements may differ. (See §6 for the fidelity fallback.)

2.4 Output extension & replace (FR-4)

Phase 7: the polished file is <original-basename-without-ext>.polished.html when doc_format == html, else <…>.polished.md. The replace prompt, the in-git mv, and the not-in-git <original>.bak-then-write fallback all use the matching extension. URL / Notion / inline inputs are still printed (no file, markdown text) — the replace prompt is skipped for them, as today.

3. Phase 2.5 — Editorial reduction (FR-7…FR-17, GR-1…GR-4)

3.1 Placement (FR-7 / F3)

Runs after "Phase 2 — Pick preset" and before "Phase 3 — Run binary eval rubric". Its output document (the reduced doc if the pass ran, else the ingested doc unchanged) becomes the working document for Phase 3 onward — rubric, budget estimate, patch passes, findings, apply, re-run, metrics. The editor pass does NOT count toward the Phase-6 2-iteration rubric cap (FR-14).

3.2 Resolving the reduction target (FR-8 / GR-4)

  1. If the --reduce <value> flag was passed: parse <value> as a single percent (25 → point target 25%) or a low-high range (30-40 → range target 30–40%). Validation: numbers only, 0 < low ≤ high ≤ 90. On a malformed value → print --reduce: invalid value '<v>'; skipping the editorial pass and treat as Skip (do NOT abort, do NOT prompt). The Phase 2.5 gate is NOT shown when --reduce is present (parallels --preset suppressing the Phase 2 prompt).
  2. Else, AskUserQuestion (interactive) — this gate has a Recommended option and is auto-pickable in --non-interactive (no defer-only tag): 
    question: "Run an editorial reduction pass before polishing? Target reduction:"
options:
  - "Skip — no reduction (Recommended)"   # → no-op, FR-9
  - "~10-20% (light trim)"                # → range target 10–20%
  - "~30-40% (substantial cut)"           # → range target 30–40%
  - "~50%+ (aggressive)"                  # → point target 50% (open-ended; rewriter cuts as much as it safely can ≥ 50%)
  # platform note: out-of-options reply / "Other" → treat free text as a custom target,
  # parsed by the same rule as --reduce; unparseable → Skip with a note.
    The custom path: a free-text answer like 25% or 25-35% is parsed identically to --reduce; unparseable → Skip-with-note.
  3. Skip (or unparseable custom, or malformed --reduce) ⇒ no-op (FR-9): no subagents dispatched, no editor_notes.json written, the ingested doc is the working doc, Phase 3+ is byte-for-byte today's behavior. The Phase 7 summary "Editorial pass:" line reads skipped.

3.3 The editor subagent (FR-10, FR-13)

Dispatched via the Task tool (or inline if no subagent tool — see Platform Adaptation). Prompt is the template in the new reference/editorial-pass.md; inputs: the verbatim ingested doc (chunked per GR-2 if ≥4,000 polishable words — one editor call per chunk, notes merged), the Phase-1 voice markers, the lock-zone byte-range map, doc_format, and the parsed target. temperature: 0. The editor critiques only — it does not rewrite. It returns a JSON object conforming to schemas/editor-notes.schema.json (§3.6). Constraints baked into the prompt:

  • No note may target a span that intersects a locked zone (the map is supplied; the editor must check).
  • Every note's locator.quote must be a verbatim ≥20-char substring of the source; locator.heading_path is the list of enclosing headings (markdown # levels or HTML <hN> text) for disambiguation. No line numbers.
  • kind ∈ {rephrase, merge, reorder, tighten, cut}; risk ∈ {low, high} — reorder is always high; a merge spanning more than two adjacent paragraphs is high; cuts/tightens/rephrases that stay within one paragraph are low.
  • est_words_saved per note (int, ≥0; rephrase may be 0).
  • Target reconciliation: reconciliation.estimated_pct = Σ est_words_saved / polishable_words × 100; achievable = whether estimated_pct reaches the target band without touching protected content or destroying meaning; note = one sentence (e.g. "doc is 60% code blocks — max safe prose cut ≈ 18%").

Per FR-13: a cut or tighten note whose quote is empty or not a verbatim substring of the source is dropped by the orchestrator before the rewriter runs (no evidence → no action) — same rule the rubric applies to cited_spans.

3.4 The rewriter subagent (FR-12, GR-1)

Dispatched after the editor returns and the orchestrator has validated/pruned editor_notes.json. Inputs: the verbatim original ingested doc, the (pruned) editor_notes.json, the voice markers, the lock-zone map, doc_format. temperature: 0. It:

  • Applies every risk: low note. A note whose quote can't be located verbatim in its heading_path context, or whose application would touch a locked span, is skipped (not force-applied) and logged.
  • Does not auto-apply risk: high notes — those are returned to the orchestrator as "surface candidates" and presented to the user through the existing Phase-5 findings protocol (each as a high-risk finding: "Fix as proposed / Modify / Skip / Defer"; structural reorders are individually surfaced per the existing structural-changes rule). Approved high-risk notes are then applied (a second short rewriter call scoped to those notes, or applied inline by the orchestrator — implementer's choice).
  • Injects voice markers; if applying a note would force a voice-marker violation, it emits PRESERVE_VOICE_CONFLICT + JSON (same token + shape as the patch contract) for that note → promoted to a high-risk finding, same as the patch passes (FR — reuse reference/patch-contract.md's conflict handling verbatim).
  • Emits: the reduced doc + an applied/skipped log (per note: applied | skipped: <reason> | surfaced | voice-conflict).
  • HTML fidelity (GR-1): after producing the reduced HTML, the orchestrator runs the non-prose-bytes-byte-identical check (§2.3). If it fails, the output is still written (best-effort HTML + warn): the Phase 7 summary and the chat output carry ⚠ markup outside prose nodes may have shifted — review before replacing, and the replace prompt is shown with NO default-yes. Never refuse, never hard-fail.

3.5 The capped re-critique (GR-3)

After the rewriter's first pass, compute actual_pct = (orig_polishable_words − reduced_polishable_words) / orig_polishable_words × 100. If actual_pct < target.low (for a point target, target.low == target.high == the point; for the open-ended ~50%+, target.low = 50) and the editor has not already re-critiqued (cap = 1): dispatch the editor subagent once more with the rewriter's output, the applied/skipped log, and the shortfall (target band, actual %, words still to cut). It returns a delta editor_notes.json (new notes only; the file is appended to, with the new notes carrying ids continuing the sequence). The rewriter runs once more on those delta notes. After this single re-critique — whatever actual_pct ends up at — the pass is done; the pipeline proceeds. Never loop further. This is fully independent of the Phase-6 2-iteration rubric cap.

3.6 editor_notes.json schema (FR-11)

New file schemas/editor-notes.schema.json (JSON Schema draft-07, mirroring the style of schemas/custom-checks.schema.json). Shape:

{
  "doc_format": "markdown" | "html",
  "target": { "kind": "range" | "point", "low": number, "high": number },
  "reconciliation": { "estimated_pct": number, "achievable": boolean, "note": string },
  "recritique": { "ran": boolean, "shortfall_pct": number | null },
  "notes": [
    {
      "id": "E1",
      "kind": "rephrase" | "merge" | "reorder" | "tighten" | "cut",
      "locator": { "heading_path": [string, ...], "quote": "<≥20-char verbatim substring>" },
      "rationale": string,
      "est_words_saved": integer,
      "risk": "low" | "high"
    }
  ]
}

The orchestrator validates the editor's output against this schema; on a schema error it prints the offending entries and proceeds with the entries that do validate (mirrors the custom-checks "print and continue" behavior — never silently swallow). Written next to the run's other artifacts (same dir the polished file would go, or cwd for non-file inputs). On --dry-run the file is still written.

3.7 --dry-run interplay (FR-17)

--dry-run + a non-Skip target: the editor subagent runs, editor_notes.json is written, the re-critique does not run (no rewriter, so no actual % to compare), the rewriter does not run; Phase 4 stops as it does today and the dry-run report includes the editor notes + reconciliation summary above the rubric results. The "Editorial pass:" summary line reads dry-run — N notes drafted (est ~X%), not applied.

3.8 Budget & metrics touchpoints (FR-15, FR-16)

  • FR-15: Phase 4's budget-estimate block gains a conditional line when the editor pass produced output: + ~2 LLM calls (editorial critique + rewrite) — or + ~4 if a re-critique ran. The Surgical/Comprehensive shape is still NOT introduced.
  • FR-16: Phase 7 before/after metrics anchor the Words: delta to the original ingested doc (not the editor-reduced doc), so the headline % includes editor cut + rubric tightening. New summary line: Editorial pass: target ~30-40% · est ~36% · actual ~33% · 19 applied / 2 skipped / 3 surfaced (2 approved) — or Editorial pass: skipped / the dry-run variant.

5. Testing strategy

/polish has no executable test runner — its tests/fixtures/ + tests/expected.yaml are contract documents the skill (and reviewers) read, not a CI suite. So "testing" here = (a) the new fixtures + expected.yaml contracts above, and (b) a manual dry-run smoke during /verify: run /polish tests/fixtures/html-doc.html --dry-run and confirm the rubric report mentions HTML lock zones; run /polish tests/fixtures/bloated-doc.md --reduce 30-40 --dry-run and confirm editor_notes.json is produced and validates against schemas/editor-notes.schema.json (validate with any JSON-schema CLI, or by inspection). No new bats/pytest is added — consistent with the skill's existing test posture.

6. HTML fidelity decision (GR-1, final)

The rewriter operates on text-node boundaries inside prose elements; the orchestrator holds the lock-zone byte-range map; after the rewrite, it reconstructs the document by splicing the new prose text into the original byte stream at the recorded prose ranges. A post-write check diffs every byte outside the prose ranges against the original. If that diff is non-empty → the output is kept as-is (best-effort), but a warning is surfaced and the replace prompt loses its default. This keeps the common case (well-formed HTML, prose-only edits) exact, degrades loudly rather than silently, and never blocks the user.

Approved changes by skill

polish

  1. Source-format honoring (F1). Detect input format (markdown vs HTML) in Phase 1; carry a doc_format ∈ {markdown, html} through the run. HTML inputs get HTML-aware lock zones (tags, attributes, <script>/<style>, comments) and HTML chunk anchors (<h1>/<h2>); markdown unchanged. Phase 7 writes <original>.polished.html for HTML inputs, <original>.polished.md for markdown; the replace prompt and .bak logic follow the same extension. URL/Notion inputs continue to normalize to markdown (their HTML is page chrome, not an authored document) — call this out explicitly. Update description, argument-hint, file-map, and anti-patterns to say "single document" instead of "single markdown document".
  2. Editorial reduction pass (F2, placed per F3). A new gated phase between current Phase 2 (preset) and current Phase 3 (rubric):
    • Prompt the user via AskUserQuestion for a reduction target — options: Skip — no reduction (Recommended) / ~10-20% (light trim) / ~30-40% (substantial cut) / ~50%+ (aggressive) / custom (free-text % or range). Skip ⇒ phase is a no-op and the pipeline proceeds on the ingested doc unchanged.
    • Editor subagent — receives the (unchanged) ingested doc + voice markers + the reduction target. Critiques ruthlessly: rephrasing, sentence/paragraph merges, section reorders, prose tightening, fluff/throat-clearing/redundancy cuts. Does not rewrite — it emits structured notes to editor_notes.json (one entry per opportunity: {id, kind ∈ rephrase|merge|reorder|tighten|cut, locator (heading path + verbatim quote, no line numbers), rationale, est_words_saved, risk ∈ low|high}) plus a target-vs-estimate reconciliation summary. Locked zones (code, frontmatter, link URLs, short table cells, HTML tags/attrs) are off-limits to editor notes.
    • Rewriter subagent — receives the original ingested doc + editor_notes.json + voice markers; applies the edits, honoring lock zones; emits the reduced doc + an applied/skipped log. high-risk notes (structural reorders, anything the rewriter can't apply without ambiguity) are surfaced to the user via the Phase 5 findings protocol rather than auto-applied.
    • The reduced doc becomes the working document for the rest of the pipeline (rubric, budget, patches, findings, metrics). The editor pass does not consume one of the 2 rubric-polish iterations. Before/after metrics in Phase 7 anchor to the original ingested doc.
    • If --dry-run: still run the editor critique and write editor_notes.json, but do not invoke the rewriter; report the notes alongside the rubric report.
    • Add to the budget-estimate block: when the editor pass is active, note "+~2 subagent calls (editor critique + rewrite)".
  3. Bookkeeping (both). Phase count goes 9 → 10 ("Track Progress" + the "10 per-phase tasks" anti-pattern wording). Add editor_notes.json to the file map. Add a reference/editorial-pass.md (editor + rewriter prompt templates, editor_notes.json schema, target-reconciliation rules) and a schemas/editor-notes.schema.json. Update tests/fixtures/ + tests/expected.yaml for: an HTML input fixture, and a verbose doc fixture with a reduction target.

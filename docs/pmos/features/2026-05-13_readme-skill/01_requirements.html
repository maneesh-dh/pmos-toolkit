/readme skill — Requirements

Date: 2026-05-13 · Last updated: 2026-05-13 · Status: Draft · Tier: 3 — Feature · Mode: skill-new · Seed: docs/pmos/ideate/2026-05-13_readme-skill.html

Standing acceptance criteria: The produced skill MUST conform to plugins/pmos-toolkit/skills/feature-sdlc/reference/skill-patterns.md §A–§F (frontmatter; description & triggering; structure & progressive disclosure; body & content; scripts & tooling; platform-conditional frontmatter). Per FR-61, this is cited, not copied.

Problem

READMEs in repos that pmos-toolkit users maintain — including agent-skills itself — drift into internal-structure dumps: file inventories, directory trees, "Features" sections, and welcome boilerplate that fail the basic adoption question. A stranger landing on these READMEs can't answer what / who / why / how-to-start in 60 seconds, so the project loses adoption to whatever the reader looks at next.

Who experiences this?

  • Maintainer — the person running /readme; initially this repo's author, later any pmos-toolkit user with rotted READMEs. They feel the loss as silent (low star-rate, no contributors, repeated "what is this?" questions in DMs).
  • The README's three readers (whom the maintainer is serving): evaluator (60s — "keep reading?"), adopter (5min — "get to first success"), contributor (30min — "extend / contribute"). All three are blocked by the same structure-dump anti-pattern.

Why now?

Three triggers converge. (1) Multiple shipped pmos-toolkit skills now need their own READMEs to gain users — adoption is the current bottleneck, not capability. (2) Every existing README generator (readme-md-generator, README-AI, GitHub Copilot's generator) is template-filling without an opinionated rubric; the niche for a rubric-gated alternative is uncontested. (3) agent-skills itself is the dogfood case — a plugin-marketplace-root with ~30 plugin-package READMEs, none audit-passing.

Goals & Non-Goals

Goals are observable user outcomes; engineering acceptance criteria belong in /spec.

Goals

  • Rubric pass on emit — generated and audited READMEs pass ≥90% of the binary structural rubric on first write — measured by the rubric script's pass-rate.
  • Cold-reader signal — for each human/simulator pair, the simulator's top-flagged friction lands in the human's top-3 friction list; this holds for ≥70% of pairs (binary per-pair check, removes Jaccard granularity noise at small list sizes). Human baseline is collected by 3 unrelated readers (recruited from the pmos-toolkit user community) each given the same ≥5 READMEs cold; the simulator runs on the same set. Loop-3 revision (was: Jaccard ≥50% on ≥3 READMEs — too noisy at small N, threshold was arbitrary; H1).
  • Maintainer-actionable findings — when run against an existing README, /readme produces ≥5 findings the maintainer agrees with on first audit — measured by per-finding accept/skip ratio in the diff-preview gate.
  • Voice preserved — round-tripping a hand-edited README through audit-mode satisfies BOTH (a) mean-sentence-length delta <15% AND (b) Jaccard of new-tokens-vs-original ≥0.7 — measured by a voice-diff script that wraps /polish's existing voice-preservation substrate. Both metrics must pass; either failure flips the gate. Loop-3: H2 (was: "<15% sentence-length / vocabulary drift" — under-specified; now mean delta + token Jaccard).
  • Monorepo coverage — workspace-discovery enumerates ≥19/20 of a curated public-monorepo test set correctly — measured by the A8 pre-spec test (Babel, Next.js, Turborepo, Nx, React Native, kubernetes/kubernetes, Cargo workspaces, this repo).

Non-Goals

  • NOT doing sentence-level prose polish — because /polish already solves voice preservation with lock zones + per-finding approval; duplicating that machinery is waste. /readme owns structure + content accuracy; it delegates prose to /polish.
  • NOT generating CHANGELOGs — because release-please / git-cliff / pmos's own /changelog already do that. The commit-classifier mechanic is borrowed for the v2 diff-mode but targets READMEs, not CHANGELOGs.
  • NOT auto-enabling the /complete-dev Phase 7.6 hook in v1 — because the section taxonomy needs validation on real repos before patching working trees during release is safe. The hook ships in v1 but is opt-in / default-off until cleared: defined as ≥3 real-repo /complete-dev runs producing zero rubric-failed patches AND a /retro pass with neutral-or-positive disposition. The skill author owns the "cleared" decision after running the 3 dogfood releases on this repo. Users explicitly enable via ~/.pmos/readme/config.yaml or a /complete-dev flag. Loop-3: M1 (was: "≥3 real-repo runs and a /retro pass clear it" — circular; now concrete pass criterion).
  • NOT auto-committing the new README — because /complete-dev is the canonical release entry and audit/scaffold are pre-release operations; the user reviews the diff, the user commits.
  • NOT supporting more than 8 workspace-manifest formats in v1 — these 8 were chosen by deliberate selection across the most-represented JS, Rust, Go, and Python monorepo ecosystems. The 8 are: (1) pnpm-workspace.yaml, (2) package.json#workspaces, (3) lerna.json, (4) nx.json, (5) turbo.json, (6) Cargo [workspace], (7) Go go.work, (8) Python pyproject.toml [tool.uv.workspace]. The pmos-toolkit plugins/*/ + .claude-plugin/marketplace.json pair is a repo-type discriminator (plugin-marketplace-root) layered on top of the 8, not a 9th manifest. Long-tail formats (yarn berry PnP, Bazel BUILD, Bun, Rush.js) fall back to interactive repo-type pick; v2 extension trigger lives in D12. Loop-3: H3 (was uncited "≥95% of public repos"; now deliberate-selection rationale + v2 trigger).
  • NOT running more than one simulated-reader iteration — because more iterations = LLM drift, not better readers (consistent with /polish's 2-iteration cap and /wireframes's reviewer loop ceiling).

User Experience Analysis

Motivation

  • Job to be done: "When I ship or update a repo, I want a README that lets a stranger decide in <60s whether to adopt and start using it in <5min, so I don't lose adoption to a wall of structure prose."
  • Importance / urgency: High and bursty. Importance is steady (adoption is the bottleneck for every new skill / plugin). Urgency spikes at three moments: shipping a new repo, the moment after /complete-dev merges a feature that changed user-facing surface, and the moment someone DMs "what is this?".
  • Alternatives: Hand-write the README (high quality, low frequency — most README rot is "I'll fix it later"); use readme-md-generator (badge-soup, template-filler); use README-AI (AI-slop voice, whole-doc regenerate destroys hand edits); use Copilot's "Generate README" (one-shot, no rubric). All three tool alternatives produce structure-dump output; the hand-written path doesn't scale.

Satisfaction Signals

  • A maintainer says "I'd ship this README" after a single audit pass with no manual cleanup.
  • The diff preview shows ≤20 lines of structural changes for a "good-but-cluttered" existing README (audit-mode is finding-grained, not rewrite-everything).
  • Three unrelated cold readers, given a scaffold-mode output, all answer "what is this?" and "how do I start?" correctly within 60s.
  • The maintainer of agent-skills runs /readme --audit-all on the repo, approves ≥80% of findings on first pass for root + every plugin.

Solution Direction

Overview

One /readme skill with three operating modes and a shared substrate:

  • --audit (default when README.md exists) — score, list findings worst-first, auto-apply mechanical, per-finding approval for structural rewrites, diff preview, write.
  • --scaffold (default when absent) — detect repo type, mine the repo for facts (manifest, entry point, example usage, license, contributors), Q&A only for what can't be detected, emit per-type opening + spine + Diátaxis linkouts, run rubric + simulated-reader, diff preview, write.
  • --update — invoked from /complete-dev Phase 7.6 hook (opt-in, default-off until first 3 real-repo runs validate the section taxonomy); classify merged commits → impacted sections → patch with human confirm per section; never auto-commit. Emits a suggested patch the user applies via /complete-dev rather than touching the working tree directly (per Q4 leaning — re-resolved below).

Mode composition (B1, loop-3). The three modes are not mutually exclusive. A single invocation can compose --audit + --scaffold when the workspace-discovery scope is scaffold-missing — packages with READMEs get audited, packages without get scaffolded, and the diff preview is unified (sectioned by package, with per-package "scaffold" or "audit" labels). --update never composes with the other two in a single invocation; it is /complete-dev-driven and runs alone. The skill-internal contract: scope = {audit: [...paths], scaffold: [...paths]} — both lists may be non-empty; the diff-preview gate operates over the union.

All three share:

  • section-schema.yaml — the spine: standard-readme ordered backbone (Title → Short Description → TOC → Install → Usage → Contributing → License) + Google's Status / Maintainers fields + Diátaxis-style Documentation linkout. Per-repo-type variants for 7 types: library / CLI / plugin / app / monorepo-root / monorepo-package / plugin-marketplace-root.
  • rubric.yaml — ~15 binary checks across hero-line shape, opening order, copy-pasteable quickstart, banned-phrase list, structural cleanliness, install-before-architecture, link-out discipline. Severity-tiered (blocker / friction / nit).
  • simulated-reader.md — three persona subagents (evaluator 60s, adopter 5min, contributor 30min). Each reads top-to-bottom, must quote concrete passages, emits a friction list. Failures feed back as rubric findings.
  • Per-type opening shapes — the 5-block opening (hero / proof row / show-don't-tell / install / why) for library/CLI/plugin/app/monorepo-package; the map+identity shape (hero / proof row / contents-table / canonical-entry quickstart / why-these-pieces) for monorepo-root and plugin-marketplace-root.
  • Workspace discovery — parses the 8 supported workspace manifests at root invocation; classifies repo as monorepo-root / plugin-marketplace-root / single-package; surfaces a single AskUserQuestion choosing scope (audit-all / audit-one / scaffold-missing / root-only).
  • Voice delegation — sentence-level prose polish is never written by /readme; surfaced as "run /polish on this section" suggestions when structural rewrites would brush the line. Boundary definition: structural = section order, section presence/absence, hero-line shape, opening pattern, link-out direction, banned-phrase flagging (token-level — /readme reports "line 14: 'blazing fast' — banned by rubric"). Prose = banned-phrase removal, sentence rewriting, paragraph-level voice, tone, register, register-consistency across sections — those belong to /polish, which receives the flagged ranges as input. /readme flags; /polish removes. Loop-3: H4 (was: "banned-phrase removal at the token level" inside /readme — leaked prose work into structural scope; now /readme flags only).

Monorepo cross-file rules (v1 candidates)

Four candidate cross-file rules, each one a binary check (A9 clarity test applied at /spec time; rules failing the binary test demote to friction-tier advisory):

  1. R1 — Root links to every package. Each detected workspace package's README.md is referenced (by relative link) from the root README's contents-table. Auto-fix: create-or-update the contents-table — when one already exists, append missing entries inline; when none exists, creating the table itself is a structural rewrite that surfaces an explicit AskUserQuestion ("root README has no contents-table — add one? [show preview]") before any write. Loop-3: M2.
  2. R2 — Packages link up to root. Every package README has a relative-link reference back to the root README. Auto-fix: add the link-up line.
  3. R3 — Install / Contributing / License sections live at root only (warn-with-override). Package READMEs that contain these sections either (a) match the root's content byte-for-byte (duplicate — surface as friction; auto-fix replaces with link-up), or (b) diverge from the root (drift — surface as warning with override, not blocker). Auto-fix offers either "replace package section with link-up to root" OR "mark this package as legitimate variance" (persisted in .pmos/readme.config.yaml as a per-package override). Real monorepos (kubernetes/kubernetes, Babel) legitimately diverge per package — blocking would be wrong. Loop-3: H5 (was: divergence as blocker; now warn-with-override).
  4. R4 — No duplicate hero-line text between root and any package. Each README has a unique hero line — root describes the family, each package describes itself. Auto-fix: surface as friction; require manual edit (hero lines are voice-sensitive — delegated to maintainer).

User Journeys

Primary Journey — Audit an existing README (single repo)

  1. Maintainer pauses mid-session, runs /readme in a repo with a README.md.
  2. Skill detects existing README → defaults to --audit mode; logs mode: audit (existing README detected).
  3. Skill reads the README, runs workspace-discovery (returns "single package, repo-type=<detected>"); confirms repo-type if heuristic confidence < threshold.
  4. Skill scores against the rubric variant for the detected type; emits findings worst-first (blockers, then friction, then nits), with a one-line description, the offending section/line, and the proposed fix.
  5. Skill auto-applies mechanical fixes (banned-phrase removal, hero-line trimming) and logs them as a single "auto-applied N mechanical findings" line.
  6. For structural rewrites (re-order sections, replace show-don't-tell, add Status field), one AskUserQuestion per finding — batched ≤4 questions per call: Apply as proposed (Recommended) / Modify / Skip / Defer. If >20 findings, the skill first issues a triage prompt: Filter to blockers + friction only (Recommended) / Review all N.
  7. After dispositions, skill runs the simulated-reader pass; any persona friction that wasn't already a rubric finding surfaces as a final batch.
  8. Skill renders diff preview (proposed README vs current); user confirms.
  9. Skill writes README.md atomically (temp + rename). Suggests /polish for prose touch-up. Does NOT commit.

Primary Journey — Scaffold a README (greenfield repo)

  1. Maintainer runs /readme in a repo without a README.md.
  2. Skill detects absence → defaults to --scaffold mode.
  3. Skill mines the repo: project name from package manifest, primary entry point from main/bin/CLI shebang, example usage from a tests/ or examples/ folder, license from LICENSE, contributors from git log.
  4. Skill runs workspace-discovery; classifies repo-type.
  5. Skill asks Q&A only for what can't be detected (capped at 4 questions per AskUserQuestion call; typically 3 questions total: hero line — user-verb + outcome ≤15 words; why this exists — 2–3 sentences; target user / persona — one phrase). Hard cap: ≤6 questions across the whole scaffold flow; beyond that, skill emits a stub README with TODO markers rather than interrogating further.
  6. Skill emits a draft using the per-type opening shape + the section-schema spine + Diátaxis linkouts.
  7. Rubric + simulated-reader pass.
  8. Diff preview ("here is the README that would land — confirm?"); user confirms; skill writes.

Alternate Journey — Monorepo audit-all

  1. Maintainer runs /readme at the root of a monorepo (e.g., agent-skills).
  2. Workspace-discovery parses every supported manifest; emits "detected N workspaces via <manifest>".
  3. Single AskUserQuestion: audit-all (root + every package) / audit-one (pick from list) / scaffold-missing (READMEs absent in subset) / root-only (default).
  4. On audit-all: each file processed against the rubric variant for its sub-type. Cross-file checks fire: root links to every package, packages link up to root, no duplicate Install/Contributing/License between root and packages (those live at root, packages link up).
  5. Findings rolled up by severity across all packages for the first pass; per-package drill-down on demand.
  6. Per-package approval gates batched (one Y/n per package, not per finding) for non-blocker tiers; per-finding approval for blockers.
  7. Single diff preview spanning all touched files; user confirms; skill writes atomically.

Alternate Journey — /complete-dev Phase 7.6 update-mode hook

Ships in v1; opt-in / default-off until first 3 real-repo runs validate the section-affinity table. Maintainer enables via ~/.pmos/readme/config.yaml or a /complete-dev flag.

  1. After /verify passes and before /complete-dev commits the release, the hook fires (only if opted in).
  2. Skill classifies merged commits via conventional-commit heuristics (feat, fix, docs, etc.) → maps to impacted sections via section-schema.yaml's commit-affinity table. The v0 commit-affinity table (B2, loop-3) lives in section-schema.yaml under key commit_affinity; the skill author owns it; initial content: feat → {Features, Usage, Quickstart}; fix → {Changelog, Troubleshooting (if present)}; docs → {Documentation, Usage}; BREAKING CHANGE in body or feat!/fix! → {Migration, Changelog}; refactor/perf/test/chore → no README sections impacted. Per-repo-type variants override the base table.
  3. Skill emits a suggested patch covering only impacted sections; surfaces one AskUserQuestion per section: Apply / Modify / Skip / Defer.
  4. If user approves, patch is staged but never auto-committed — /complete-dev's existing commit flow picks it up as part of the release commit, preserving authorship and provenance. Patch-rubric abort semantics (H6, loop-3): after staging, the patch is re-scored against the rubric; if the patched README would fail any blocker check, the patch is dropped (not the release), /complete-dev logs the dropped-patch outcome to /retro with a "patch failed self-rubric — manual review" finding, and the release proceeds with the original (un-patched) README. The hook never blocks /complete-dev.
  5. If no sections are impacted, caches the "no README changes needed" verdict so re-runs in the same /complete-dev session are free.
  6. Failure case: commit-classifier confidence below threshold → skip the hook with a warn line ("README update skipped — commit signal ambiguous"), never block /complete-dev.

Error journeys

  • Unknown workspace-manifest format — root has a layout the 8 supported manifests don't catch (Bazel, yarn berry PnP, Bun, Rush). Skill falls back to interactive repo-type pick rather than guessing; surfaces "detected: unknown layout — pick repo-type" with a list.
  • Hand-edited README is dirty (uncommitted) — snapshot-commit before any write, message snapshot: pre-/readme-rewrite (same pattern as /requirements pre-write safety).
  • Audit produces 50+ findings on a monorepo run — auto-filter to blockers-only on first pass; offer "show friction + nit" expand on demand; cap interactive prompts at 4-per-batch.
  • Simulated-reader returns no friction at all — flag suspicious (theater check); re-prompt the personas with "you have alternatives and 90 seconds; what makes you bounce?" injected into the failure-seeking prompt template.
  • Workspace cross-file rule produces ambiguous result — surface as a friction-tier finding with both options, not a blocker; the rule promotes to blocker only if it passes the A9 clarity test (binary, no human judgement) at /spec time.

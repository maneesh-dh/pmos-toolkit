1. Problem Statement

READMEs in repos that pmos-toolkit users maintain drift into internal-structure dumps — file inventories, directory trees, "Features" sections, welcome boilerplate — and fail the basic adoption question (what / who / why / how-to-start in 60 seconds). Three reader personas (evaluator-60s, adopter-5min, contributor-30min) are all blocked by the same structure-dump anti-pattern. We ship a /readme skill that audits or scaffolds READMEs against a rubric + simulated-reader, with three operating modes (--audit, --scaffold, --update) sharing one substrate, voice work delegated to /polish, and monorepo-aware workspace discovery. Primary success metric: rubric pass-rate ≥90% on first emit across the 5-README dogfood set (this repo's plugins).

3. Non-Goals

  • NOT doing sentence-level prose polish — delegated to /polish per D2.
  • NOT generating CHANGELOGs — release-please / git-cliff / /changelog already cover that.
  • NOT auto-enabling the /complete-dev Phase 7.6 hook in v1 — opt-in / default-off until ≥3 real-repo runs + /retro pass per M1.
  • NOT auto-committing the new README — /complete-dev is the canonical release entry per D9.
  • NOT supporting more than 8 workspace-manifest formats in v1 — long-tail (Bazel, yarn berry PnP, Bun, Rush) falls back to interactive repo-type pick per H3.
  • NOT running more than one simulated-reader iteration per D6.

5. User Personas & Journeys

Personas and journey-narrative carry forward verbatim from 01_requirements §User Journeys. This section specifies the technical sequence behind each journey; see §6 for the architecture diagrams.

5.1 Primary Journey — Audit existing README (single repo)

  1. User runs /readme in repo with README.md present.
  2. Mode-resolver detects file → defaults to --audit. Logs mode: audit (existing README detected).
  3. Workspace-discovery runs (FR-WS-1 below); single-package result, repo-type confirmed via heuristic + AskUserQuestion fallback when confidence < threshold.
  4. README loaded, rubric variant for detected type selected, 15 checks scored → findings list emitted worst-first (blockers, friction, nits).
  5. Auto-apply pass: mechanical findings (banned-phrase flagging, hero-line trimming) auto-applied; logged as one "auto-applied N mechanical findings" line.
  6. Structural findings: AskUserQuestion per finding, batched ≤4 per call. If >20 findings, triage prompt first ("Filter to blockers+friction (Recommended)?").
  7. Simulated-reader pass: 3 personas dispatched in PARALLEL subagents (D13); friction findings returned with quote+line; parent substring-validates quotes (FR-SR-3).
  8. Unresolved simulated-reader friction surfaces as a final findings batch.
  9. Diff preview emitted (proposed vs current). User confirms via AskUserQuestion.
  10. Atomic write (temp+rename) → suggest /polish for prose touch-up → DOES NOT COMMIT.

5.2 Primary Journey — Scaffold (greenfield)

  1. User runs /readme in repo without README.md.
  2. Mode-resolver → --scaffold.
  3. Repo-mining subagent dispatched: package manifest name, entry point (main/bin/CLI shebang), example usage (tests/, examples/), license file, contributors via git log --format="%an" | sort -u | head -10.
  4. Workspace-discovery → classify repo-type.
  5. Q&A only for what can't be mined. ≤4 questions per AskUserQuestion call; hard cap ≤6 across whole scaffold flow; beyond that → stub README with TODO markers.
  6. Draft emitted using per-type opening shape + section-schema spine + Diátaxis linkouts.
  7. Rubric pass + simulated-reader pass.
  8. Diff preview ("here is the README that would land") → user confirms → atomic write.

5.3 Primary Journey — Update (Phase 7.6 hook)

Default-OFF; opt-in via ~/.pmos/readme/config.yaml :: phase_7_6_hook_enabled: true AND .pmos/complete-dev.lastrun.yaml :: readme_update_hook: true (both required). Cleared for default-on after ≥3 real-repo runs with zero rubric-failed patches + a /retro neutral-or-positive (M1).

  1. /complete-dev Phase 7.6 detects opt-in → invokes /readme --update <commit-range> where range is <merge-base>..HEAD.
  2. Commit-classifier (FR-UP-1) runs over the commit range: conventional-commit prefix mapped to impacted sections via section-schema.yaml :: commit_affinity (v0 table per requirements; see §7.6).
  3. Per impacted section: AskUserQuestion (Apply / Modify / Skip / Defer), ≤4 per call.
  4. If user approves: patch is re-scored against rubric (FR-UP-3 patch-fail guard). If patched README fails ANY blocker check, patch is DROPPED (not the release); /complete-dev logs "/retro: patch failed self-rubric" finding; release proceeds with un-patched README.
  5. If patch passes rubric: staged in working tree (not committed). /complete-dev's commit flow picks it up as part of release commit.
  6. No impacted sections OR classifier confidence below threshold → no-op with warn line ("README update skipped — <reason>"). Never blocks /complete-dev.

5.4 Alternate Journey — Monorepo audit-all

  1. User runs /readme at monorepo root.
  2. Workspace-discovery parses ALL supported manifests; emits "detected N workspaces via <primary-manifest>"; for multi-stack repos (MS01) appends "+ <secondary-manifest> (M additional packages)".
  3. Workspace-scope AskUserQuestion (FR-WS-4): audit-all / audit-one / scaffold-missing / root-only (Recommended) / multi-stack: include all (only shown when MS01 triggers).
  4. On audit-all: each file processed against the rubric variant for its sub-type (rubric-variant table FR-RUB-2).
  5. Cross-file rules R1–R4 fire (FR-CF-1 to FR-CF-4).
  6. Findings rolled up by severity across packages first pass; per-package drill-down on demand.
  7. Per-package approval gates batched (one Y/n per package, not per finding) for non-blocker tiers; per-finding for blockers.
  8. Single unified diff preview (D15) spanning all touched files with per-file section headers. One final approval → atomic-multi-write.

5.5 Error journeys (summary)

Carry forward from 01_requirements §Error journeys: unknown manifest, dirty tree, 50+ findings, simulated-reader returns no friction, ambiguous cross-file rule. Each handled inline per the FR table; no additional spec-level mechanics.

7.3 Rubric (15 binary checks)

IDRequirement
FR-RUB-1Rubric lives in reference/rubric.yaml as 15 binary checks; severity distribution targets ≈4 blockers / 7 friction / 4 nits (D3).
FR-RUB-2Rubric variants per repo-type (7 types): library / CLI / plugin / app / monorepo-root / monorepo-package / plugin-marketplace-root. 7.5 Workspace discovery

IDRequirement

FR-WS-6Long-tail formats (Bazel WORKSPACE/MODULE.bazel, yarn berry PnP, Bun, Rush rush.json): fallback to interactive repo-type pick with detected: unknown layout warn line.

7.6 Update mode (Phase 7.6 hook)

IDRequirement
FR-UP-1Commit-classifier consumes <commit-range> argument; parses each commit's subject via conventional-commit regex ^(feat|fix|docs|refactor|perf|test|chore)(\([^)]+\))?(!)?:\s; BREAKING CHANGE: body footer detected via git show --format=%b. Maps to impacted sections via section-schema.yaml :: commit_affinity; per-repo-type override applied if present.
FR-UP-2Classifier confidence: minimum quorum is ≥1 conventional-commit-formatted commit in range. Range with zero conv-commit subjects → no-op + warn line "README update skipped — commit signal ambiguous (no conventional-commit subjects in <range>)" (per requirements error journey).
FR-UP-3Patch-fail guard: after staging the draft patch, re-run rubric.sh against the patched README. If ANY blocker check fails, ABORT the patch (drop, do not block release); log JSON line {event:"patch_dropped", reason:"rubric_blocker_fail", failed_checks:[...]} to .pmos/readme/update.log + emit /retro finding "patch failed self-rubric — manual review" (H6).
FR-UP-4Opt-in gate: hook fires only when BOTH (a) ~/.pmos/readme/config.yaml :: phase_7_6_hook_enabled: true AND (b) .pmos/complete-dev.lastrun.yaml :: readme_update_hook: true are set. Either absent or false → no-op (D14).
FR-UP-5Patched README is staged via git add <readme-path> only; no git commit by /readme. /complete-dev's commit flow picks it up in the release commit (D11).
FR-UP-6"Cleared for default-on" criterion (M1): ≥3 real-repo /complete-dev runs with zero patch_dropped events in update.log AND a /retro pass with neutral-or-positive disposition. Decision logged by skill author; flips the default in ~/.pmos/readme/config.yaml shipped template.

7.7 Simulated-reader

IDRequirement
FR-SR-1reference/simulated-reader.md defines 3 persona prompts: evaluator (60s scan — "keep reading?"), adopter (5min — "get to first success"), contributor (30min — "extend / contribute"). Each persona reads top-to-bottom and emits {persona, friction:[{quote:"≥40 chars verbatim", line:N, severity:"blocker|friction|nit"}]}.
FR-SR-2Dispatched in PARALLEL via 3 concurrent Task subagent calls (D13). Per-persona timeout 120s; on timeout, persona is skipped with a chat warn line; remaining personas continue.
FR-SR-3Parent-side validation (FR-50/51/52 pattern from /grill): substring-grep every quote against the un-stripped README source; any miss → hard-fail with simulated-reader returned quote not found in README: <prefix-30>…; pause with failure dialog.
FR-SR-41-iteration cap (D6); no convergence loop. Findings merged into the main rubric findings stream (severity-tagged); duplicates of rubric findings (same line ±2) deduped.
FR-SR-5Theater-check: if a persona returns zero friction AND the rubric returned ≥3 findings, re-prompt that persona ONCE with "you have alternatives and 90 seconds; what makes you bounce?" suffix injected (per requirements error-journey row). After re-prompt, accept the result (even if still empty).
FR-SR-6--skip-simulated-reader CLI flag bypasses the entire persona pass; logged as simulated-reader: skipped (--skip-simulated-reader). Useful for fast iteration; documented as not-recommended-for-final-runs.

7.8 Cross-file rules (monorepo)

IDRequirement
FR-CF-1R1 — Root links to every package: each detected workspace package's README.md is referenced (by relative link) from the root README's contents-table. Auto-fix: create-or-update (when table exists, append missing entries inline; when no table exists, structural rewrite — surface explicit AskUserQuestion: "root README has no contents-table — add one? [show preview]" before write) (M2).
FR-CF-2R2 — Packages link up to root: every package README has a relative-link reference back to root README. Auto-fix: add the link-up line at end-of-readme (configurable section).
FR-CF-3R3 — Install/Contributing/License sections at root only (warn-with-override): package READMEs with these sections either (a) match root byte-for-byte → friction, auto-fix replaces with link-up; (b) diverge → warn-with-override, not blocker. Auto-fix offers "replace with link-up to root" OR "mark this package as legitimate variance" (persisted in .pmos/readme.config.yaml :: package_variance: [<path>]). Real monorepos (kubernetes/kubernetes, Babel) legitimately diverge (H5).
FR-CF-4R4 — No duplicate hero-line text between root and any package: surface as friction; require manual edit (hero lines voice-sensitive — delegated to maintainer).
FR-CF-5A9 clarity test applied at design-time: each rule above produces a binary pass/fail with no human judgment required. Rules failing A9 demote to friction-tier advisory. Test executed during skill development; results logged in reference/cross-file-rules.md.

7.9 Voice delegation

IDRequirement
FR-V-1Structural-vs-prose boundary: structural = section order/presence/absence, hero-line shape, opening pattern, link-out direction, banned-phrase flagging only (token-level "line 14: 'blazing fast' — banned by rubric"). Prose = banned-phrase removal, sentence rewriting, paragraph-level voice, tone, register. /readme FLAGS, /polish REMOVES (H4).
FR-V-2When /readme structural rewrites would brush the prose line, output includes a suggested follow-up: Suggest: /polish <readme-path> --target=sections=<list> with the flagged ranges as targets.
FR-V-3Voice-drift gate (G4): scripts/voice-diff.sh <pre> <post> returns JSON {sentence_len_delta_pct, jaccard_new_tokens}. Gates: sentence_len_delta_pct < 15 AND jaccard_new_tokens ≥ 0.7. Either failure flips the gate; the diff-preview surfaces the failure as a blocker finding.

9. CLI & Subagent Contracts

No HTTP API. The "API surface" is the CLI argument grammar + the subagent dispatch contracts.

9.1 CLI grammar

/readme [--audit|--scaffold|--update <commit-range>]
        [--audit-all|--audit-one <pkg>|--scaffold-missing|--root-only]
        [--format md]                          # only md in v1
        [--skip-simulated-reader]
        [--non-interactive|--interactive]
        [<repo-path>]                          # defaults to cwd

9.2 Subagent dispatch contracts

9.2.1 Simulated-reader persona subagent

Dispatch: 3 concurrent Task calls, one per persona (evaluator / adopter / contributor). Prompt body inlines the persona definition from reference/simulated-reader.md + the un-stripped README markdown source. Per-call timeout: 120s. Return shape (validated parent-side):

{
  "persona": "evaluator" | "adopter" | "contributor",
  "friction": [
    { "quote": "<≥40-char verbatim substring of README>",
      "line": <1-indexed>,
      "severity": "blocker" | "friction" | "nit",
      "message": "<1-sentence concrete friction>" }
  ]
}

Parent validation: substring-grep every quote against README source; any miss → hard-fail (FR-SR-3).

9.2.2 Repo-miner subagent (scaffold mode)

Dispatch: single Task call. Prompt body inlines the supported-manifest list + the repo-root path. Return shape:

{
  "name": "<project name from manifest>",
  "entry_point": "<main|bin|cli-shebang path or null>",
  "example_usage": "<code snippet from tests/ or examples/ or null>",
  "license": "<SPDX id or filename or null>",
  "contributors": ["<name>", ...],   // top 10 by git-log frequency
  "repo_type_hint": "<type> | null"
}

14. Rollout Strategy

  1. v1 ship: all three modes (--audit, --scaffold, --update) land in one commit; update-mode opt-in/default-off via dual flag (FR-UP-4).
  2. Day-1 dogfood: /readme runs against this repo's root README + the 5 dogfood plugin READMEs (pmos-toolkit, etc.); G1 (≥90% rubric pass) and G3 (≥5 findings approved on first audit) gate the release.
  3. Cleared-for-default-on: after ≥3 real-repo /complete-dev runs with zero patch_dropped events + a /retro neutral-or-positive disposition, skill author flips the phase_7_6_hook_enabled default in the shipped ~/.pmos/readme/config.yaml template (FR-UP-6). This is a follow-up release, not v1.
  4. A4 trial: post-v1 dogfood pass invites 3 unrelated pmos-toolkit users to cold-read 5 READMEs; simulator runs in parallel; G2 gate. If A4 fails, simulated-reader demotes to opt-in via --skip-simulated-reader becoming the default + opt-in --with-simulated-reader.
  5. D12 taxonomy extension trigger: after v1, extension to an 8th repo-type is gated by ≥2 real repos that don't fit + a /retro pass with a documented rationale.
  6. Rollback: /readme is non-destructive (atomic write, snapshot-commit, never auto-commits). Rollback is git checkout HEAD <readme-path> or revert of the snapshot-commit. No deploy-time risk.

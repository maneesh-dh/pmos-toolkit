Rollback

/readme is non-destructive by design (atomic write, snapshot-commit, never auto-commits). The skill-bundle drop itself is reversible at the git layer; per-feature rollback paths:

  • Bundled scripts regress (selftest fails post-merge): revert the offending commit via git revert <sha> at plugins/pmos-toolkit/skills/readme/ scope. No external side effects.
  • Update-mode misbehaves on a /complete-dev run: flip ~/.pmos/readme/config.yaml :: phase_7_6_hook_enabled: false (per-user) or set .pmos/complete-dev.lastrun.yaml :: readme_update_hook: false (per-repo). Either disables the hook immediately; no /readme bundle changes needed.
  • Skill misbehaves on a user repo: git checkout HEAD <readme-path> reverts the README; the snapshot-commit (E11) provides the prior version if the dirty-tree path was hit.
  • Whole-skill rollback: the v1 ship is one feature-branch merge. git revert <merge-sha> at the parent merges back to pre-/readme state. Tag rollback per /complete-dev release-recipes.

Findings

Blockers (2)

B1 · overview — Mixed-mode behavior unspecified. The doc never specifies what happens when a repo legitimately needs two modes (e.g., a monorepo with some packages READMEd and some absent) — is it audit+scaffold in one run, or two invocations, and how is the diff preview unified?
"scaffold-missing (READMEs absent in subset)"
B2 · alternate-journey-update-mode — Section-affinity table is referenced but undefined. The conventional-commit classifier → section-affinity mapping is the load-bearing mechanism for update-mode but the table is not specified, sampled, or located. Where does it live, who authors it, what is its initial content?
"maps to impacted sections via section-schema.yaml's commit-affinity table"

High (8)

H1 · goals — ≥50% Jaccard threshold is arbitrary. What makes 50% the threshold rather than 30% or 70%, and what happens if the merged human list has only 4 items so Jaccard granularity is coarse?
"simulated-reader friction lists overlap ≥50% with a human cold-reader's friction on the same README"
H2 · goals — "Voice drift <15%" metric is underspecified. Is it mean sentence-length delta, type-token ratio delta, or something else, and why 15% rather than a perceptual threshold?
"<15% sentence-length / vocabulary drift — measured by a voice-diff script"
H3 · non-goals — 95% manifest-coverage claim has no citation. Which ecosystem survey supports this, and does the 95% hold weighted by repo-popularity or by raw count?
"the 8 chosen cover ≥95% of public repos"
H4 · overview — Voice-delegation boundary leaks prose work. "Banned-phrase removal at the token level" is unmistakably a prose/voice judgement. What's the principled line that prevents this from being a backdoor for /readme to do prose work it claims to delegate?
"banned-phrase removal at the token level"
H5 · monorepo-cross-file-rules — R3 treats legitimate divergence as blocker. Many monorepos (kubernetes/kubernetes, Babel) intentionally have per-package install instructions that differ from root. Why is divergence a blocker rather than a signal of legitimate per-package variance?
"(b) diverge from the root (drift — surface as blocker)"
H6 · alternate-journey-update-mode — Phase 7.6 staging interaction with /verify is unspecified. The patch is staged but never auto-committed; /complete-dev runs after /verify. What happens if the staged patch fails the rubric or breaks an in-flight commit — does the release abort, proceed without the README change, or block?
"patch is staged but never auto-committed — /complete-dev's existing commit flow picks it up"
H7 · design-decisions — D10 taxonomy closure has no extension trigger. What's the explicit trigger criterion for adding an 8th repo type for v2, given Q1 is the same question unresolved?
"The repo-type taxonomy is closed at 7 types for v1."
H8 · open-questions — Q5 is a decision disguised as a question. Three of the riskiest assumptions (A2/A4/A8) gate the skill's value but validation is punted to /spec. If validation comes back negative at /spec time, the entire approach changes.
"How are A2 (rubric distinguishes slop), A4 (simulated-reader is real signal), A8 (workspace discovery ≥19/20) experimentally validated"

Medium (4)

M1 · non-goals — Phase 7.6 "validation criterion" is circular. Who decides a run "clears" it, what's the pass criterion, and what stops this from being permanently default-off because nobody opts in?
"opt-in / default-off until ≥3 real-repo runs and a /retro pass clear it"
M2 · monorepo-cross-file-rules — R1 auto-fix assumes a contents-table exists. What happens when R1 fires on a root that doesn't yet have one, and is creating the table itself a structural rewrite requiring approval?
"R1 — Root links to every package."
M3 · design-decisions — D6 1-iteration cap reuses a prose-domain argument. /polish operates on prose and /readme operates on structure. Why does the same convergence-drift argument apply?
"Convergence runs drift the LLM toward 'everything looks fine'"
M4 · open-questions — Q3 contradicts R3. R3 already forces package READMEs to lack Install/Contributing/License, so Q3 is partially answered yet still listed as open.
"Probably: package has no Install / Contributing / License sections"

Validation-dropped — advisory only (1)

F15 · empty-states-and-edge-cases — Multi-manifest disambiguation. When multiple workspace-manifest types coexist (e.g., package.json#workspaces AND pnpm-workspace.yaml, or pyproject.toml + Cargo workspace), discovery behavior is unspecified — does it pick one, merge, or ask? Quote failed FR-52 verbatim substring check (subagent capitalized "Workspace" vs source's lowercase). Substantive concern is real and counted in dispositions.

Problem

Users of this toolkit (PMs, researchers, founders) field surveys via Typeform / SurveyMonkey / Google Forms / Qualtrics, get back a CSV / Excel / PDF of raw responses, and then face the slowest step of the loop: turning hundreds or thousands of mixed-type responses (single-select, multi-select, Likert, NPS, ranking, matrix, numeric, open-text) into a credible report that answers "so what should we do?". Doing it by hand in a spreadsheet takes hours and the typical pitfalls (treating ordinal Likert as interval; computing multi-select on the wrong base; reporting NPS as an average; over-fitting themes onto a tiny open-end sample; spurious "significant" subgroup differences from multiple-comparisons) silently degrade the conclusions. There is no skill in pmos-toolkit that closes that loop today — /survey-design builds the questionnaire but stops at fielding; everything after collection is manual.

Who experiences this?

A PM, researcher, founder, or analyst with a fielded survey's raw responses in hand who needs a defensible report (executive summary + per-question deep-dive) to share with a stakeholder or to drive a decision. Typically n is 50–5,000; one survey; mixed question types; English-language verbatims.

Why now?

The toolkit just shipped /survey-design (v2.36.0). The natural pair-with — analysing the responses the survey produced — is the obvious next step and the user explicitly asked for it. The two skills compose: /survey-design's survey.json is a high-fidelity questionnaire-context input that /survey-analyse can consume verbatim.

Goals & Non-Goals

Goals

  • Turn raw responses into a credible report in one skill invocation. The user points the skill at a response file (CSV/XLSX/TSV/PDF) and optionally a context doc/survey.json; the skill returns an HTML report with executive summary, whole-survey synthesis, and per-question analysis — measured by: a single run on a ≤2,000-row survey produces a report the user can hand to a stakeholder without manual rework.
  • Use methodologically defensible numbers, not LLM eyeball estimates. Per-question stats (frequencies, T2B/B2B, NPS = %P − %D, median for ordinal/numeric, average rank, cross-tab cells, Holm-adjusted p-values, theme counts) come from bundled deterministic Python helper modules — not LLM inspection of the raw CSV. Per run, the LLM authors an analysis.py that imports those helpers and runs via Bash; helper-emitted JSON drives the report's numbers. Measured by: numeric tables are byte-identical across runs on the same cleaned input (narrative + theme names are LLM-generated and may drift — disclosed in the Methodology section).
  • Bake in the survey-methodology pitfalls catalogue. Bundled reference/ guidelines cover per-question-type methods, open-end thematic coding, whole-survey statistics (significance, multiple comparisons, MoE, weighting caveats), data cleaning, and synthesis/reporting standards — measured by: the produced report never (a) reports a Likert mean without also showing the distribution and T2B, (b) reports a multi-select option's % on the response base instead of the respondent base, (c) reports NPS as an average, (d) headlines a subgroup difference without showing base sizes, or (e) themes open-ends without representative verbatim quotes.
  • Surface limitations honestly. Every report carries a Methodology & Limitations section that records the sample size, completion rate (if derivable), cleaning rules applied (with counts removed), the probability-vs-non-probability classification (and whether MoE is reported or suppressed accordingly), and any flagged risks (small N for a headlined subgroup; high "Don't know" rate on a key item; speeder/straightliner share) — measured by: the section is non-empty on every run; the report does not headline a subgroup with n < 30 without an explicit caveat.
  • Compose cleanly with the rest of the toolkit. Skill follows the /survey-design shape (standalone utility; writes to its own dated run folder under {docs_path}/survey-analyses/; HTML-substrate artifacts with companion sections.json and feature-folder index.html; the runbook's output_format gate; the canonical non-interactive block; platform-adaptation section) — measured by: the skill passes the /feature-sdlc Phase 6a skill-eval.md rubric (§A–§F) with no failed checks at iteration 1.

Non-Goals (explicit scope cuts)

  • NOT a data-collection or survey-fielding tool — because /survey-design covers design and the user fields the survey on their platform of choice. /survey-analyse starts at the response export.
  • NOT a longitudinal / wave-tracking analyzer in v1 — because single-wave coverage is the dominant use case and wave-over-wave comparisons need a separate cross-run identifier story. Re-runnable on each wave separately; tracked as a future enhancement.
  • NOT a real-time dashboard — because the artifact is a one-shot HTML report. Re-run for refreshed numbers.
  • NOT a statistical-modelling / regression / clustering tool — because the user's stated job is "explain what the responses say and what to do" not "build a predictive model". Limited to descriptive stats + the planned bivariate significance tests.
  • NOT a weighting designer — the skill will apply a user-supplied per-respondent weight column if present and document it; computing weights against population benchmarks is out of scope.
  • NOT a PII redactor — anonymizing free-text quotes is the user's responsibility; the skill will flag obvious PII patterns (emails, phone numbers) in surfaced quotes as a quality check but will not silently rewrite them.
  • NOT translation — multilingual verbatim analysis is out of scope in v1; user is expected to feed English (or pre-translated) responses.
  • NOT auto-redaction of PII in verbatims — because false positives on names hurt signal more than they help (a brand name like "Apple" looks like a proper noun); the skill flags regex-detected PII (emails, phones, name-with-honorific patterns) and shows a count in chat ("3 quotes contain potential PII — review before sharing"), but emits the verbatims unmodified. User responsibility to scrub before sharing externally; --redact opt-in is a future enhancement.

User Experience Analysis

Motivation

  • Job to be done: "I have survey responses; I need a defensible report I can share or act on by the end of today."
  • Importance / urgency: high — the survey was fielded to inform a decision; the analysis is the blocker between data and decision.
  • Alternatives: hand-rolled spreadsheet pivot tables (slow, error-prone on Likert/multi-select/NPS conventions); the survey platform's built-in reporter (rigid, no cross-tabs or theming); paying a research vendor (expensive, slow); asking an LLM to "summarize this CSV" (no rigour, no per-question methodology).

Satisfaction Signals

  • The exec summary headline ("62% of respondents picked X; among Enterprise (n=140) it was 78%") reads like something the user would have written if they had time.
  • The per-question sections show charts/tables matching the question's type (diverging stacked bars for Likert; horizontal bars with multi-answer label for multi-select; the headline NPS number large + the distribution).
  • The methodology section is short enough to read and specific enough to defend ("attention-check fail = 1 of 3 missed; 7 respondents excluded; speeders n=3 (<30% median time) excluded").

Solution Direction

A standalone utility skill at plugins/pmos-toolkit/skills/survey-analyse/SKILL.md, modelled on /survey-design's shape (sister skill — produces analysis where /survey-design produces questionnaire). The skill runs eight numbered phases and writes a single dated run folder under {docs_path}/survey-analyses/{YYYY-MM-DD}_<slug>/:

  1. Phase 0 — Setup. Read .pmos/settings.yaml (default docs_path if absent — same fallback as /survey-design, NOT first-run pipeline-setup). Parse flags: --responses <path>, --context <path|url>, --survey-json <path>, --sheet <name|N>, --weight-col <name>, --format html|md|both, --non-interactive|--interactive, --engine python|node|auto, --skip-cleaning, --no-significance. Resolve output_format. Pick a unique dated run-folder slug. Inline the canonical non-interactive block. Load ~/.pmos/learnings.md under ## /survey-analyse.
  2. Phase 1 — Ingest responses. Read the input file. Dispatch by extension: .csv/.tsv → native; .xlsx/.xls → pandas read_excel in the Python helper (or sheetjs in the Node helper); .pdf → Read tool tabulates with a one-table-per-question heuristic + LLM fallback for layout-broken PDFs, with limitations explicitly noted. Multi-sheet XLSX → prompt for which sheet unless --sheet set. Save a normalised responses.csv in the run folder as the canonical analysis input. Stop and ask the user if the file fails to parse or yields zero rows.
  3. Phase 2 — Survey-context intake (always user-confirmed schema). If --survey-json was passed (a /survey-design output): read it; treat sections[].questions[] as the proposed schema. Else: ingest any --context doc/URL and infer types from the data + column-header text. Either way, surface the full schema column-by-column for confirmation via AskUserQuestion (batched per column-group; ≤4 per call) — every column has an explicit user-confirmed type before Phase 4 runs. The same batched flow asks: survey purpose; key decision(s) the analysis should inform; segment columns of interest. No silent auto-proceed on inferred types; the schema confirmation is the safety net for the LLM-narrates-deterministic-stats contract.
  4. Phase 3 — Clean. Run the cleaning checklist from reference/data-quality.md: straightliners (per matrix, zero or near-zero per-respondent variance), speeders (if a duration column is present, <½ × median default; configurable), incompletes (configurable threshold), exact-duplicates, attention-check fails (only if the schema flags such a question), numeric outliers (1.5×IQR review). Each rule documents: rule, threshold, records flagged, records removed. Skipped under --skip-cleaning with a warning in the methodology section. Output: cleaned_responses.csv + cleaning.json log.
  5. Phase 4 — Per-question analysis (LLM authors analysis.py; helpers compute). Python-only. The skill ships per-question-type helper modules under scripts/helpers/ — categorical.py, multi_select.py, likert.py, nps.py, ranking.py, matrix.py, numeric.py — pure stdlib + openpyxl for xlsx ingest; chi-square / column-proportion z-test / Holm correction hand-rolled (~30 LOC each; peer-reviewed formulas; self-tests via --selftest). For this run the LLM authors a per-run analysis.py in the run folder that imports these helpers, iterates the confirmed schema, and writes per_question.json. The LLM then runs it via Bash (python3 analysis.py) — a single consolidated permission ask covers the whole run. Helper-emitted JSON is the source of truth; the LLM never recomputes numbers. analysis.py is saved as the audit artifact. Per question-type, helpers emit per reference/question-type-analysis.md:
    • single-select / categorical: freq, %, mode.
    • multi-select: per-option % of respondents (not responses), mean options selected.
    • Likert / rating (ordinal): full distribution, median, mode, T2B, B2B, T2B−B2B; mean reported with an explicit ordinal caveat.
    • NPS (auto-detected: 0–10 scale, "recommend" wording): %P, %Passive, %D, NPS = %P − %D as an integer.
    • ranking: average rank + weighted points; convention stated.
    • matrix / grid: per-row Likert stats; whole-matrix straightlining flag.
    • numeric / open numeric: n, mean, median, SD, min, max, percentiles, histogram bins, outliers flagged.
    • open-text: deferred to Phase 5.
    Output: per_question.json — keyed by column name; values are typed stat blocks. The LLM never recomputes these numbers; it narrates on top of them.
  6. Phase 5 — Open-end thematic coding (subagent-per-question). For each open-text column with n ≥ 5: dispatch a fresh subagent per column whose prompt embeds (a) the question wording, (b) all verbatims (chunked to ≤200 per call; multi-call concatenation if needed), and (c) the Braun & Clarke 6-phase contract from reference/text-analysis.md. Subagent returns structured {themes: [{name, definition, response_ids: [...], representative_quote_ids: [...]}]}. A second consolidation pass groups near-duplicate themes across columns where appropriate. PII detection regex runs over selected quotes; matches surface a chat-side count (no auto-redaction — per Non-Goals). Final themes.json records: themes (name, definition, count, % of open-end responders, 1–3 representative quotes, sentiment lean). Themes < 5 mentions on n < 50 open-ends labelled "preliminary".
  7. Phase 6 — Whole-survey synthesis. Run cross-tabs of every closed question against user-confirmed segment variables (Phase 2). For each cross-tab: column %, n per cell, chi-square overall, pairwise column-proportion z-tests, and Holm-adjusted p-values applied by default across the full family of tests in the cross-tab batch. Significance markers (*, **, ***) use the adjusted value. Report frames the correction in plain language ("we adjusted significance because many comparisons were run at once"); the term "Holm" appears only in the Methodology section. --raw-p-only opts out of correction for analysts who want raw. Auto-flag n < 30 cells. Synthesise key takeaways: top 3–5 findings + the open-end themes that explain them. Output: synthesis.json + a narrative draft.
  8. Phase 7 — Report rendering. Render report.html via the _shared/html-authoring/ substrate (template.html, kebab-case heading ids, sections.json companion, ?v=<plugin-version> cache-bust, assets/ copy, atomic temp-then-rename). Sections (in order): Executive Summary; Methodology & Limitations; Key Findings; Per-question Analysis; Open-end Themes; Cross-tab Appendix; Data-quality Log. Inline SVG bar/diverging-stacked-bar charts (no external libs; SVG round-trips to MD via turndown). Companion report.sections.json. Regenerate the run-folder index.html with the inlined manifest. output_format: both → emit report.md sidecar via the substrate's html-to-md.js. Output: report.html (+ report.md sidecar when applicable).
  9. Phase 8 — Workstream / Learnings / Handoff. No workstream enrichment (the skill does not load workstream context — see D1). Read ~/.pmos/learnings.md under ## /survey-analyse. Emit the one-line learnings reflection (Learning: … or No new learnings this session because …). Handoff line: print the path to report.html + the run folder.

The control flow is linear — no review loops at the skill level (the user re-runs the skill to iterate; phase outputs are saved as JSON for cheap re-rendering without re-coding). The skill has a single ## Track Progress contract listing the 8 phases.

User Journeys

Primary journey (happy path)

  1. User runs /survey-analyse --responses ~/Downloads/survey.xlsx --survey-json docs/pmos/survey-design/2026-05-08_pricing-survey/survey.json in a Claude Code session.
  2. Phase 0 sets up; Phase 1 reads the XLSX → asks "Sheet 1 (Responses) or Sheet 2 (Metadata)?"; user picks Sheet 1.
  3. Phase 2 loads survey.json; schema confirmed without re-prompting; one batched AskUserQuestion: "Purpose of this survey? Key decision? Segments to compare? Anything else to flag?". User answers.
  4. Phase 3 cleans: 11 records flagged (3 speeders, 2 straightliners, 6 incompletes); user sees the log and confirms exclusion. cleaned_responses.csv = 489 rows.
  5. Phase 4 runs scripts/analyse.py; per_question.json written; agent narrates each question's stat block.
  6. Phase 5 themes the 3 open-text columns; themes.json written.
  7. Phase 6 produces cross-tabs by user-confirmed segments (plan tier, region); flags 2 cells with n < 30; synthesises 4 key takeaways.
  8. Phase 7 renders report.html + report.sections.json; index.html regenerated.
  9. Handoff: user opens report.html in a browser, reviews, shares with the stakeholder.

Alternate — no survey.json, only a context doc

  1. User runs /survey-analyse --responses survey.csv --context "this was a pricing-willingness survey fielded to 600 founders in April".
  2. Phase 2 infers the schema from the response columns; confidence per question; one prompt asks the user to confirm/correct, and to fill in survey purpose / decisions / segments.
  3. Rest of the flow continues as Primary.

Alternate — PDF input

  1. User runs /survey-analyse --responses survey-export.pdf (Google Forms / Typeform PDF export).
  2. Phase 1 attempts Read-tool tabulation; if structured tables are recovered, proceeds with a warning ("PDF ingestion is best-effort; verify the normalised responses.csv before committing"); if not, falls back to an LLM-tabulation pass over the PDF text and prompts the user to confirm; if still ambiguous, the skill pauses and asks the user to export CSV/XLSX instead.

Alternate — non-interactive run

  1. User runs /survey-analyse --responses survey.csv --non-interactive (e.g. inside a CI / automation).
  2. The canonical non-interactive block kicks in: every AskUserQuestion is auto-picked on its (Recommended) option or deferred per the defer-only tag; deferred questions buffer to _open_questions.md in the run folder; the report emits with a Run Outcome: deferred marker and an explicit "this run did not get user clarifications X, Y, Z" section.

Error — Python missing, Node missing too

  1. Phase 4 detects no python3; falls back to node; if neither is on PATH: stops with a clear "install python3 or node" message, surfaces a one-line copy-pasteable install command, writes the partial run-folder state, and exits.

Error — empty / unparseable file

  1. Phase 1 → zero rows or parse error → stops, surfaces the parser's error verbatim, asks the user to re-export or pick another file. No subsequent phases run.

Empty states & edge cases

ScenarioConditionExpected behaviour
Single-column fileresponses.csv has 1 columnTreat as 1 free-text or 1 categorical question; warn that cross-tabs are not possible; produce a degenerate but valid report.
No open-text columnsschema has zero free-text questionsSkip Phase 5 silently; report has no Themes section; methodology notes the absence.
No segment variables identifiedPhase 2 surfaces zero confirmed segmentsSkip Phase 6 cross-tabs; produce univariate-only synthesis; report flags this explicitly.
Very small N (n < 30)cleaned_responses.csv has 1–29 rowsRun the analysis; suppress all significance tests; the report's Methodology section headlines the small-N caveat; no subgroup analysis attempted.
PDF with no recoverable tablestabulation fails twice (Read + LLM)Stop at Phase 1 with the export-as-CSV recommendation; do not produce a partial report.

Control flow (8 phases)

Linear, no orchestrator-level review loops. The phase contract maps 1:1 to the Phase numbers in 01_requirements.html#solution-direction:

  1. Phase 0 — Setup (settings, flags, non-interactive block, learnings)
  2. Phase 1 — Ingest (CSV/TSV/XLSX/XLS/PDF → responses.csv)
  3. Phase 2 — Schema + context (always user-confirmed, column-by-column)
  4. Phase 3 — Clean (rules + counts; cleaned_responses.csv + cleaning.json)
  5. Phase 4 — Per-question analysis (LLM authors + Bash-runs analysis.py; per_question.json)
  6. Phase 5 — Open-end thematic coding (subagent-per-question; themes.json)
  7. Phase 6 — Whole-survey synthesis (cross-tabs + Holm; synthesis.json)
  8. Phase 7 — Report rendering (substrate-compliant report.html)
  9. Phase 8 — Learnings + handoff

API contracts

Helper module signatures

Every helper module exposes pure functions taking a list-of-rows + a column name (or two for cross-tabs) and returning a JSON-serializable dict. No global state. Each helper carries a --selftest CLI entry point asserting known-answer fixtures.

The per-run analysis.py shape

"""Generated by /survey-analyse on YYYY-MM-DD for run <slug>.
   Edit by hand only if you know what you're doing — re-runs of /survey-analyse
   overwrite this file."""
import csv, json, sys
sys.path.insert(0, "/abs/path/to/plugins/pmos-toolkit/skills/survey-analyse/scripts")
from helpers import categorical, multi_select, likert, nps, ranking, matrix, numeric, stats

ROWS = list(csv.DictReader(open("cleaned_responses.csv")))
SCHEMA = json.load(open("schema.json"))

per_q = {}
for col, meta in SCHEMA["columns"].items():
    kind = meta["type"]
    if kind == "single_select":     per_q[col] = categorical.freq_table(ROWS, col)
    elif kind == "multi_select":    per_q[col] = multi_select.multi_select_table(ROWS, col, meta.get("delimiter", "|"))
    elif kind == "likert":          per_q[col] = likert.likert_stats(ROWS, col, scale_size=meta["scale_size"])
    elif kind == "nps":             per_q[col] = nps.nps(ROWS, col)
    # ... ranking, matrix, numeric, open_text (skipped — handled in Phase 5)

# Cross-tabs (Phase 6) — Holm-corrected across the family.
cross_tabs = {}
for segment in SCHEMA["segments"]:
    fam_pvals = []
    cross_tabs[segment] = {}
    for col, meta in SCHEMA["columns"].items():
        if meta["type"] in {"single_select", "likert", "nps"}:
            cells, raw_p = stats.cross_tab(ROWS, row_col=col, segment_col=segment)
            cross_tabs[segment][col] = {"cells": cells, "p_raw": raw_p}
            fam_pvals.append(raw_p)
    adjusted = stats.holm_correct(fam_pvals)
    # assign adjusted back to each entry...

json.dump({"per_question": per_q, "cross_tabs": cross_tabs}, open("per_question.json", "w"), indent=2)

Phase 5 subagent contract

For each open-text column: dispatch a fresh Agent with prompt template:

[mode: <current-mode>]
Question: <question text from schema>
Verbatims (n=<N>): <chunked list, each prefixed with response_id>

Apply Braun & Clarke 6-phase inductive thematic analysis:
1. Familiarise. 2. Open code. 3. Search for themes. 4. Review themes.
5. Name & define. 6. Write up.

Return JSON: {
  "themes": [
    {"name": "<3–5 word name>",
     "definition": "<one-sentence what's-in / what's-out>",
     "response_ids": [<ids that touch this theme>],
     "representative_quote_ids": [<1–3 ids>],
     "sentiment_lean": "positive|neutral|negative|mixed"}
  ],
  "uncoded_response_ids": [<ids that didn't fit any theme>]
}
A response MAY appear in multiple themes; report double-counting in
<themes_meta> (sum of theme counts will be > N when this happens).

Non-Functional Requirements

  • NFR-1 (size). SKILL.md body ≤500 lines (Tier 3 cap; matches §C). Reference files individually ≤600 lines.
  • NFR-2 (performance). Phase 4 analysis.py processes 5,000 rows × 30 columns in ≤10s on a 2020-era laptop.
  • NFR-3 (portability). Helpers stdlib-only except openpyxl; openpyxl pinned to ≥3.0 in helper docstrings. PDFs use stdlib only (best-effort tabulation); falls back to the Read tool for layout-broken PDFs.
  • NFR-4 (atomicity). All run-folder file writes are temp-then-rename; no half-written artifacts.
  • NFR-5 (observability). Skill logs phase entry/exit to chat; cleaning log surfaces every excluded record.
  • NFR-6 (mode passthrough). Inherits [mode: …] first-line marker from /feature-sdlc when invoked downstream; honours the canonical non-interactive block.

Testing strategy

  1. Helper self-tests. Every module under scripts/helpers/ ships --selftest with known-answer fixtures (small-N Likert, NPS, multi-select, etc.). Run via tests/selftest.sh in CI.
  2. Fixture-driven integration. tests/fixtures/ ships ≥3 small response files (CSV + XLSX + PDF). The /verify smoke runs the skill against each and asserts: report.html written; methodology section non-empty; per_question.json matches expected (golden file).
  3. Skill-eval rubric (Phase 6a). feature-sdlc/tools/skill-eval-check.sh --target claude-code plugins/pmos-toolkit/skills/survey-analyse runs the deterministic half; reviewer subagent runs the LLM-judge half against skill-eval.md.
  4. Anti-pattern checks. The 5 zero-tolerance defects (T2B+distribution mandatory on Likert; multi-select on respondent base; NPS as integer; subgroup base shown; theme quotes) are CI-asserted on the integration fixture's produced report.html.

Open items deferred to /plan

  • OQ3: exact n < ? cutoff for "skip Phase 6 entirely". Spec sets n < 30 as the default; /plan task tags this for fixture review.
  • PDF tabulation reliability: spec sets the contract (Read-then-LLM-tabulate; pause and ask for CSV if both fail). /plan adds a dedicated task to harden this on at least 2 platform-exported PDFs (Typeform, Google Forms).

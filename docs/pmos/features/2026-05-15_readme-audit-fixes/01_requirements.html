/readme audit fixes — Requirements

Date: 2026-05-15 · Last updated: 2026-05-15 · Status: Draft · Tier: 2 — Enhancement · Mode: skill-feedback · Target skill: /readme at plugins/pmos-toolkit/skills/readme/

Problem

A retro on /readme (1 audit + 1 rewrite session) surfaced that the 15-check rubric and 3-persona simulated-reader pass deliver structural wins (license, contributing, install shape, line-length, cross-file checks) but are blind to value-prop accuracy and information-architecture fit. The audit reported "clean against rubric" on the most consequential structural question — that the README was organized maintainer-backward (by skill category, not by job-to-be-done) and that the hero line falsely scoped the toolkit to "software delivery". The user surfaced these issues in 5 minutes of reading; the rubric never could.

Two additional defects compound this: (a) BSD-awk false-fails on macOS due to \b usage in awk patterns (every macOS user sees install-or-quickstart-presence FAIL despite valid content); (b) the Suggest: /polish follow-up line documented in §11 (FR-V-2) does not fire on the audit edge; (c) the §1 step 4 findings contract conflates audit (read-only) with scaffold/update (writes), leading to UX divergence in pure-audit runs; (d) FR-SR-3 sub-40 quote enforcement was silently relaxed in practice — the exact "silent recovery" the spec forbids — with no regression test to catch it.

Who experiences this?

Returning-user developers running /readme --audit against a real plugin/monorepo README — the persona who needs to map intent → skill, not the first-contact reader the existing 3-persona triad optimises for. Also every macOS user of /readme (BSD-awk default).

Why now?

The retro represents real session friction. The audit's verdict "clean against rubric" was true and useless — without these fixes, the next user running /readme --audit on the pmos-toolkit README (or any README organized by maintainer category) will get the same false-pass and the same BSD-awk false-fail.

Goals & Non-Goals

Goals

  • The rubric + persona pass surfaces IA-fit and value-prop issues on a README organized maintainer-backward — measured by the audit catching the retro's specific IA finding when re-run against the pmos-toolkit README pre-rewrite.
  • rubric.sh passes install-or-quickstart-presence on a valid README under BSD awk — measured by macOS CI / selftest on a fixture README that previously false-failed.
  • Every successful /readme --audit run emits the Suggest: /polish line — measured by tracer test on the audit-only edge.
  • Pure-audit runs do not fire AskUserQuestion — measured by §1 step 4 mode-branched contract + integration test.
  • A simulated-reader return with a sub-40 quote hard-fails and pauses — measured by new regression fixture.

Non-Goals

  • NOT rewriting the existing 3 personas (evaluator/adopter/contributor) — because the retro proposes adding a 4th lens; existing personas already optimise first-contact bounces correctly for their role.
  • NOT changing BSD grep -qE \b usages on lines 213/223 of rubric.sh — because BSD grep supports \b; only awk is broken.
  • NOT introducing voice/prose rewriting into /readme — because §11 explicitly delegates that to /polish (Anti-pattern).
  • NOT re-architecting the rubric.yaml schema — because the 3 new IA-fit checks fit the existing check-row shape.
  • NOT preserving 15/15 pass on existing READMEs — because the 3 new checks are advisory; user disposes per-finding via the §1 step 4 contract (audit-mode table or scaffold/update-mode batched ask). Net rubric size 15 → 18 is a deliberate widening.

Solution Direction

Five surgical fixes against the existing skill, each preserving the established invariants (3-mode substrate, monorepo-aware discovery, voice→/polish delegation, never auto-commit):

  1. 4th persona: returning-user-navigator — extend reference/simulated-reader.md §1 with the new persona prompt ("I know this plugin exists — can I find the skill for my current job?"). Return-shape contract from §2 is reused verbatim — no new JSON schema. Persona dispatch loop in SKILL.md §2 step 1 changes from 3 concurrent Task calls to 4 concurrent Task calls. Per-persona 120s budget unchanged; theater-check (§3 FR-SR-5) extends to the new persona symmetrically.
  2. 3 new deterministic IA-fit rubric checks in reference/rubric.yaml — (a) hero-scope-matches-surface: hero line claim ≥ surface-area implied by Skills table (regex-extract Skills table rows + hero claim noun phrase; FAIL on superset mismatch); (b) primary-index-by-jtbd: top-level ## headings index by job-to-be-done, not by maintainer category (heuristic: ≥3 of N headings match imperative verb / "How to X" / problem-domain pattern); (c) cross-cutting-capabilities-surfaced: at least one section mentions cross-cutting platform capability (HTML/MD, worktrees, auto-tiering, self-eval). Total: 15 → 18 checks. All three are [D] deterministic (regex/awk-based, no LLM-judge).
  3. BSD-awk fix in scripts/rubric.sh — replace \b with ($|[^A-Za-z]) at lines 40 and 119; mirror the workaround already at line 60 (check_what_it_does_in_60s with its existing comment "BSD awk lacks \< / \> word boundaries; use a portable boundary class"). Audit every remaining awk \b usage in the script as part of the change.
  4. §1 step 6 close-out template fix — emit Suggest: /polish <path-to-README.md> — tighten prose without changing meaning. on the audit close-out path unconditionally (currently only fires after §6 scaffold and §7 update). Cite FR-V-2. Existing §11 wording stays authoritative.
  5. §1 step 4 mode-branch + §2 step 2 wording tighten + regression fixture — split §1 step 4 into "audit mode: emit findings table only, no AskUserQuestion" vs "scaffold/update mode: batched 4-option AskUserQuestion (Apply/Modify/Skip/Defer)"; FR-OUT-2 citation branches on mode. Tighten §2 step 2 so a sub-40 quote MUST pause (no silent drop). Add tests/integration/simulated_reader_sub40_quote.sh that mocks a persona return with a 24-char quote via READMER_PERSONA_STUB and expects the parent to hard-fail with the documented FR-SR-3 message.

User Journeys

Primary Journey — returning-user audit on a maintainer-backward README

  1. User on macOS runs /readme --audit README.md on the pmos-toolkit README pre-rewrite.
  2. Rubric script runs 18 checks; the 3 new IA-fit checks FAIL (hero-scope mismatch; non-JTBD index; cross-cutting capabilities surfaced is borderline). install-or-quickstart-presence PASSES under BSD awk (previously false-failed).
  3. 4 personas dispatched in parallel; the new returning-user-navigator persona returns a friction entry citing the maintainer-backward IA.
  4. Findings rendered as a table — no AskUserQuestion fires (audit mode read-only contract).
  5. Close-out emits two lines: rubric: 14 pass / 4 fail and Suggest: /polish README.md — tighten prose without changing meaning.

Error / Edge Cases

  • Persona returns sub-40 quote — parent hard-fails with simulated-reader returned quote shorter than 40 chars: <quote>; pauses with failure dialog. Regression fixture exercises this exact case.
  • BSD vs GNU awk environment — selftest in rubric.sh --selftest exercises both boundary semantics; portable class works under both.
  • README has no Skills table (rare, non-monorepo single-tool) — hero-scope-matches-surface check vacuously passes (no surface to compare against).
  • Scaffold/update mode — §1 step 4 still fires AskUserQuestion batched 4-option ask; no change to existing flow.

Acceptance against skill-patterns.md (standing FRs)

The revised /readme must conform to plugins/pmos-toolkit/skills/feature-sdlc/reference/skill-patterns.md. Mapped FRs:

  • §A frontmatter: existing frontmatter preserved; description still carries trigger phrases; no changes warranted by this edit. FR-AC-A.
  • §B description+triggering: no new trigger phrases needed; existing 5+ phrases ("audit my README", "scaffold a README", "fix my README", "generate a README", "review my README structure") cover the bug-fix delta. FR-AC-B.
  • §C structure / progressive disclosure: changes confined to §1 step 4/6, §2 step 2, §11 wiring, and reference/* files — no new top-level ## sections; SKILL.md stays under skill-eval line cap. FR-AC-C.
  • §D body content: wording tightening + mode-branch split is structural body work; no marketing prose; no implementation details bleeding in. FR-AC-D.
  • §E scripts: scripts/rubric.sh change uses existing patterns (printf TSV row, named function, --selftest via the established _lib.sh); new fixture follows tests/integration/ conventions. FR-AC-E.
  • §F platform-conditional: no platform-conditional frontmatter changes warranted. FR-AC-F.

Architecture Principles Skill

Created using pmos‑toolkit

TL;DR

Ship a single /arch skill with four invocation modes (scaffold / spec-check / audit / code-check) backed by a versioned principles.yaml (tiered: universal → stack → project), each rule citing its source book/article and producing draft ADRs for structural violations. Start TS-only; delegate declarative checks (deps, layers, cycles) to existing tools (Dependency Cruiser, ts-arch) rather than reimplementing. Bake in the OpenAI “harness engineering” opinion that opinionated structure is what makes coding agents 10× — that’s the differentiating thesis vs improve-codebase-architecture and the lint-only competition.

Recommendation: GO — but ship the surface of Finalist #2 (four-subcommand) wrapped around the substance of Finalist #1 (principles-as-data + ADR-driven + harness-eng scaffold opinions). Defer Finalist #3’s declarative engine to v2; delegate to existing AST tools for v1. Run /requirements architecture-principles-skill next.

How Might We

How might we encode opinionated software-architecture principles as a single skill that scaffolds greenfield repos, audits proposed designs in /spec, audits existing repos, and verifies code generated during /execute — without becoming a one-size-fits-all lint that everyone disables?

Target user & job-to-be-done

Who: the same engineer / PM-engineer who already lives in /feature-sdlc — i.e., someone driving multi-stage agentic dev cycles and reviewing agent-generated code.

When / where: (a) starting a new repo, (b) reviewing a /spec before approving the design, (c) inheriting a legacy repo and wanting a punch-list of architectural improvements, (d) merging code that an agent just produced during /execute.

JTBD: “When I’m at any of those four moments, I want a consistent, opinionated, book-grounded architecture check that flags violations with rationale, so my codebase stays maintainable and agent-friendly (per the harness-engineering thesis) without me having to remember every principle from Clean Architecture, Pragmatic Programmer, DDD, and Tidy First on demand.”

Hypothesis

We believe shipping one /arch skill with four mode-specific framings over a shared, citation-rich, tiered principles.yaml will produce consistent, agreed-with architectural feedback across scaffolding, spec review, audit, and code review because (a) the LLM is already strong at applying named principles when the principle is in-context and cited, (b) the harness-engineering thesis predicts that opinionated, uniform structure compounds agent output quality, and (c) keeping rules as data (not prompt) means we can version, diff, and override per project without forking the skill.

Idea variants considered

Generated via First Principles + Analogous Inspiration. Finalist(s) flagged with ★.

  1. ★ Single skill, four sub-commands (/arch scaffold | spec-check | audit | code-check) over one principles file
  2. ★ Principles-as-data, skill-as-runner — versioned principles.yaml with rule + rationale + book citation
  3. Tiered rulebook: L1 universal (SOLID, boundaries), L2 stack (TS / Python / Go), L3 project overrides
  4. Pure prompt skill, no rule engine — SKILL.md encodes everything; cheapest, least deterministic
  5. Hook-based — PostToolUse on Edit/Write + SessionStart scaffold; most agent-native, least controllable
  6. Embedded checkpoint in /feature-sdlc only, between /execute and /verify — smallest surface, biggest miss
  7. improve-codebase-architecture clone, multi-mode — adopt Pocock’s ruleset as v1; pmos wrapper adds 4 modes
  8. ★ “Harness Engineering” opinionated scaffold — bake OpenAI’s blog opinion (single test runner, single fmt, monorepo conventions) into /arch scaffold
  9. ★ ESLint-for-architecture — declarative ruleset (forbidden-dependency, layer-violation, circular-import) compiled into prompts + optional CI checks
  10. ★ ADR-driven — every flagged structural violation drafts an ADR the user accepts / rejects / edits; principles file becomes the cumulative ADR log
  11. Citation-bot — every rule carries source: book/chapter; violations cite why. Educational; risk of noise
  12. Tidyfirst (Beck) — classify each proposed change as structural-only vs behavioral; gate them differently
  13. Two-skill split — /arch-principles (audit + gates) and /arch-scaffold (greenfield), separate release cycles
  14. Per-stack flavor packs — like ESLint configs (@pmos/arch-ts-node, @pmos/arch-fastapi, @pmos/arch-react); skill is the runner

How it works (chosen finalist: #2 + #3 + #8 + #10 composite, with #1 as the surface)

  • Surface: one skill, /arch <mode> [args]. Modes: scaffold, spec-check <spec.html>, audit [path], code-check [diff|HEAD~1]. Bare /arch shows a mode picker.
  • Substance: a versioned principles.yaml with three tiers (L1 universal ≤15 rules, L2 stack-specific, L3 project overrides). Each rule: id, statement, rationale, source (book / chapter / URL), severity (info / warn / block), applies_to (which mode(s) the rule fires in).
  • Sources baked in v1: Clean Architecture (Martin), Pragmatic Programmer (Hunt & Thomas), Domain-Driven Design (Evans), Tidy First? (Beck), OpenAI “Harness Engineering” (2025), Matt Pocock’s improve-codebase-architecture.
  • Scaffold mode: opinionated greenfield generator — single test runner, single formatter, single deps file, folder-by-feature, no barrel files, explicit boundary layers. Writes a docs/architecture/principles.md + a starter principles.yaml the project can amend.
  • Spec-check mode: reads the spec artifact, surfaces which L1/L2 rules the proposed design violates (semantic check via LLM), suggests rewrites inline. Gates /feature-sdlc Phase 2.
  • Audit mode: walks the repo, runs declarative checks via delegated tools (Dependency Cruiser for cycles, ts-arch for layer violations, ruff/pylint for Python equivalents), plus semantic checks via LLM against L1/L2. Output: prioritized punch list with ADR drafts.
  • Code-check mode: runs on a diff (post-/execute, pre-/verify). Same checks as audit but scoped to changed files + their immediate import graph. Severity block rules fail the gate; warn attaches notes.
  • ADR-driven persistence: any block-severity violation in audit or code-check spawns a draft ADR (template under docs/architecture/decisions/) the user accepts / rejects / edits. Accepted ADRs become L3 project overrides — the principles file grows by accretion, not edict.
  • Citations on every violation: output format includes “Violates L1-04: Dependency Inversion (Clean Architecture, ch. 11) — rationale.” so each flag is its own micro-lesson.

Success signals

Leading (first 30 days): ≥3 of 4 modes (scaffold / spec-check / audit / code-check) invoked at least once in real work; ≥1 substantive violation per /feature-sdlc run that the human agrees with; the principles file gets edited (i.e., user accretes L3 overrides — proof of engagement); A/B harness-eng test shows measurable structural difference in agent output.

Lagging (90 days): A new repo scaffolded with /arch scaffold ships its first feature without an architecture-debt entry in /verify; a legacy repo audited at day 0 shows ≥30% punch-list closure by day 90; ADR count grows monotonically (proof of compounding); skill is cited in /feature-sdlc retro as “saved time” at least once per month.

Next steps

  1. Validate the harness-eng A/B first (riskiest H/M assumption, cheapest test) — before investing in the skill, prove that an opinionated principles.yaml + scaffold actually changes agent output on a real task. ½-day spike.
  2. Gap-map existing tools (Dependency Cruiser, ts-arch, ruff, ArchUnit) against 20 candidate L1 rules — decide what to delegate vs author. 2 hours.
  3. Author principles.yaml v0 — 12 L1 rules with citations to Clean Architecture, Pragmatic Programmer, DDD, Tidy First?, harness-engineering, and Pocock’s skill. No code yet; just the data file + 5 example violation renders.
  4. Show 5 sample violation outputs to a non-author (you, in a week) — confirm the citation format reads as educational, not preachy. If >2 of 5 feel noisy, redesign the format before writing the skill.
  5. Run /requirements architecture-principles-skill with the v0 principles.yaml + this artifact attached. The four-mode surface, ADR loop, tier resolution, and tool-delegation strategy become the explicit FRs.

Open questions

  • Should principles.yaml live per repo (committed) or per skill (one canonical set the skill ships) — or both (skill ships L1+L2, repo owns L3)? Recommended: both. Confirm.
  • How opinionated should scaffold be? Single TS flavor (matching harness-eng article literally) v1, or pluggable flavor packs from day one?
  • Should code-check wire as a /feature-sdlc Phase 6.5 hook automatically, or stay manual until v2?
  • ADR drafts under docs/architecture/decisions/ — Nygard ADR format (lightweight) or MADR (structured)? Default Nygard, allow override?
  • Is “architecture” the right name, or does it overlap with /spec in users’ minds? Candidates: /arch, /principles, /house-style, /harness.
  • What’s the right relationship to /verify’s existing code-review subagents? Embed code-check into /verify vs run as a sibling gate?
  • For non-TS stacks, do we ship anything in v1, or explicitly scope to TS + Node only and add Python/Go in v2?

README skill — Ideation

Created using pmos‑toolkit

TL;DR

Ship /readme as a single pmos-toolkit skill with three modes — audit (default when README exists), scaffold (default when absent), and diff-update (v2, invoked from /complete-dev) — all sharing one section-schema.yaml, one binary rubric (~15 checks), and one simulated-reader verifier (3 personas at 60s / 5min / 30min time-budgets). Prose voice delegated to /polish. The opinionated default opening is the 5-block hero pattern repeated across every strong exemplar README (FastAPI / ripgrep / htmx / Stripe / Tailwind / Astro / shadcn): user-verb tagline → proof row → show-don't-tell demo → install → why-this-exists. Monorepos are first-class: the skill detects workspace manifests (pnpm-workspace / lerna / nx / turbo / cargo / go / pyproject / plugin marketplaces) and treats root vs. package READMEs as different repo-types with different opening shapes — root README's job is map + identity (contents-table replaces show-don't-tell demo), package README's job is the standard 5-block opening with link-up to the root for shared context.

Recommendation: Proceed to /requirements readme-skill, but gate the spec on validating three riskiest assumptions first — A2 (rubric distinguishes AI-slop), A4 (simulated-reader is real signal), and the new A8 (workspace-discovery heuristic detects ≥95% of real monorepos). All three are <1 day each.

How Might We

How might we generate README.md files that are user-facing — answering what / who / why / how-to-start in the first 200 words — rather than internal-structure dumps, for both greenfield repos and incrementally during /complete-dev?

Target user & job-to-be-done

Who: Maintainers of pmos-toolkit-installed repos (initially: this repo's author; later: any user of the plugin). Three reader archetypes the README itself must serve: evaluator (60s — keep reading?), adopter (5min — get to first success), contributor (30min — extend / contribute).

When / where: Two trigger contexts — (a) the maintainer pauses mid-session and runs /readme against an existing repo whose README has rotted into a file-inventory; (b) /complete-dev Phase 7.6 reaches the README hook after a feature merges and asks whether to patch impacted sections.

JTBD: When I ship or update a repo, I want a README that lets a stranger decide in <60s whether to adopt it and start using it in <5min, so I don't lose adoption to a wall of "structure" prose.

Hypothesis

We believe shipping /readme with a binary structural rubric (~15 checks) plus a simulated-reader verifier and the 5-block opening discipline will produce generated READMEs that pass the binary rubric ≥90% of checks on first emit, and a human-validated "would adopt" judgment from 3+ unrelated readers because every strong exemplar README converges on the same opening shape, and the failure mode of every existing tool (readme-md-generator, README-AI, Copilot's generator) is template-filling with no opinionated rubric — leaving a clear niche for a rubric-gated alternative that delegates prose to /polish.

Prose polish delegated to /polish.

How it works (chosen finalist)

  • Three modes, one substrate. --audit (default when README.md exists), --scaffold (default when absent), --update (v2; invoked from /complete-dev Phase 7.6 as opt-in hook). All three read the same section-schema.yaml and the same rubric.yaml.
  • Section schema (the spine). Standard-readme ordered backbone (Title → Short Description → TOC → Install → Usage → Contributing → License) augmented with: (a) Status + Maintainers fields per Google's open-source style; (b) a Diátaxis-style Documentation linkout section instead of inlining tutorials/reference. Per-repo-type variants for the seven supported types: library / CLI tool / plugin / app / monorepo-root / monorepo-package / plugin-marketplace-root.
  • Opening shapes (per repo-type). library / CLI / plugin / app / monorepo-package → the standard 5-block opening: (1) hero line — user-verb + outcome, ≤15 words; (2) proof row — badges + one-line social proof; (3) show-don't-tell — 5–10 lines of working code/CLI demonstrating the most common use-case; (4) install + first run — 2–3 copy-pasteable commands; (5) why this exists — 2–3 sentences. monorepo-root / plugin-marketplace-root → a map+identity shape: (1) hero line — what the project family does; (2) proof row; (3) contents table (auto-generated from workspace manifests, replaces show-don't-tell) listing each package/plugin with its one-line purpose + link; (4) most common entry point quickstart — one canonical "start here" path; (5) why these pieces live in one repo. Anti-pattern enforced: root never duplicates package content; packages always link up to the root for shared install/contributing/license.
  • Binary rubric (~15 checks). Examples: hero-line-≤15-words, opens-with-user-verb-not-noun, has-working-quickstart, quickstart-is-copy-pasteable, no-directory-tree-as-prose, status-field-present, links-out-not-inlines-reference, no-banned-slop-phrases ("🚀 Features", "blazing fast", "Welcome to"), tldr-or-tagline-in-first-200-words, install-before-architecture. Severity-tiered (blocker / friction / nit).
  • Simulated-reader verifier. Three persona subagents — evaluator (60s skeptical reader with alternatives), adopter (5min trying to reach first success), contributor (30min trying to extend). Each reads top-to-bottom, must quote concrete passages where they got lost, and emits a friction list. Failures feed back as additional rubric findings.
  • Audit mode flow. Score existing README → list findings ordered worst-first → auto-apply mechanical fixes → per-finding approval for structural rewrites → final simulated-reader pass → write README.md (atomic temp+rename) with diff preview before commit.
  • Scaffold mode flow. Detect repo-type from file structure (fallback: ask). Mine repo for: project name, package manifest, primary entry point, example usage from tests/examples folder, license, contributors. Q&A only for what can't be detected: hero line, why-this-exists, target user. Emit 5-block opening + standard-readme spine + Diátaxis linkouts. Run rubric + simulated-reader. Diff preview.
  • Update mode (v2). /complete-dev Phase 7.6 hook (opt-in, default-off until v2 stabilizes). Classify merged commits via conventional-commit heuristics → map to impacted sections via section-schema.yaml's commit-affinity table → patch only those sections → require human confirm on every section touched → never auto-commit. Cache "no README changes needed" verdict.
  • Voice preservation. Audit/scaffold delegate prose-level polish to /polish (already solves voice with lock zones + per-finding approval + PRESERVE_VOICE_CONFLICT). /readme owns structure + content accuracy only; never sentence-level voice.
  • Monorepo orchestration. When invoked at a repo root, the skill runs workspace discovery: parses pnpm-workspace.yaml, package.json#workspaces, lerna.json, nx.json, turbo.json, Cargo [workspace], Go go.work, Python pyproject.toml [tool.uv.workspace], and the pmos-toolkit plugins/*/ convention. If any workspaces are detected → repo-type is monorepo-root (or plugin-marketplace-root when .claude-plugin/marketplace.json exists). The skill then surfaces a single AskUserQuestion: (a) audit-all READMEs in one batch (root + every package, per-file findings, single diff preview); (b) audit-one (pick from a list of discovered READMEs); (c) scaffold-missing (find workspaces lacking README.md and emit stubs); (d) root-only (default for a casual audit). Each file is processed against the rubric variant matching its detected sub-type. Cross-file checks: root must link to every package; packages must link up to root; no duplicate Install / Contributing / License sections between root and packages (these live at root, packages link up).
  • Position vs. existing tools. No existing tool does structural README audit (gap). No existing tool does monorepo-aware README orchestration (second gap — readme-md-generator / README-AI all treat one repo as one README). Diff-mode is also a genuine gap (release-please does CHANGELOGs, not READMEs). Scaffold is the crowded space — differentiator is the rubric + simulated-reader + /polish delegation + monorepo awareness.

All three fit in Phase 0 of the spec, <1 day of work each. A9 (cross-file rules are binary) is a design-time check, not a pre-build experiment — fold into /spec.

Success signals

Leading: (1) Generated READMEs pass ≥90% of binary rubric checks on first emit. (2) Simulated-reader friction list overlaps with human cold-reader ≥50%. (3) Audit on pmos-toolkit's current README surfaces ≥5 actionable findings the maintainer agrees with. (4) Diff between author's hand-edited README pre/post-skill shows <15% voice drift. (5) Workspace-discovery enumerates ≥19/20 of the test-set monorepos correctly (A8 gate).

Lagging: (1) Skill is invoked >3× per maintainer per quarter. (2) ≥3 unrelated users say they'd adopt a project after reading a skill-generated README cold. (3) /complete-dev v2 hook stays opted-in by >50% of installs after first 4 releases. (4) Zero "skill regenerated my README and destroyed my voice" complaints in /retro. (5) On agent-skills' own monorepo, the audit-all run produces a maintainer-approved diff for root + ≥80% of plugins on first pass.

Open questions

  • Should /readme be a pmos-toolkit skill or its own plugin? (Leaning pmos-toolkit — it composes with /polish, /verify, /complete-dev.)
  • Should the repo-type taxonomy (currently 7 types) be closed or extensible via user config at ~/.pmos/readme/types/? (Closed for v1; revisit if A8 dogfood surfaces a common type the 7 don't cover.)
  • Does the v2 diff-mode patch the working tree README, or emit a suggested patch the user applies via /complete-dev? (Patch-suggestion is safer; touching working-tree during release is risky.)
  • Does the simulated-reader pass cap at 1 iteration or run-to-convergence? (Cap at 1 like other pmos skills — more iterations = LLM drift, not better readers.)
  • For monorepo audit-all, should findings be presented per-package (12 separate review streams) or rolled up by severity across all packages (one blocker list)? (Lean: roll-up by severity for the first pass, drill-down per-package on demand.)
  • What's the rubric-variant delta between monorepo-package and library? (Probably: monorepo-package has no Install / Contributing / License sections — those link up to root. Validate during /spec.)
